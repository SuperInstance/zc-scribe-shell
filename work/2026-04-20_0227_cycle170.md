## Step 2: Documentation Quality Report

**Overall Fleet Documentation Health: Mixed.**
Scores ranged from 4 to 9, with an average of **6.4/10**.

**Patterns Observed:**
1.  **Strong on Vision, Weak on Onboarding:** Repos like `plato-torch` and `constraint-theory-core` successfully explain *what* the project is and *why* it matters, but only `constraint-theory-core` provides a complete, copy-paste friendly path to a first run.
2.  **Placeholder Anti-Pattern:** The presence of `TODO` or example command placeholders (`--help`) without actual examples (`--config ./example.yaml`) is a common failure mode, as seen in `fleet-simulator` and `cudaclaw`.
3.  **Internal Knowledge Assumed:** Several READMEs reference internal fleet concepts ("PLATO rooms," "tiles," "P0/P1/P2") without linking to foundational documents like `PROJECT-VISION.md` or `DEADBAND-PROTOCOL.md`. This creates a high barrier for external contributors.

**Recommendations (P0/P1/P2):**
-   **P0 (Don't break):** Do not publish READMEs with non-functional example commands or empty `TODO` sections. This creates a negative first impression and blocks users.
-   **P1 (Find safe channel):** Establish a fleet-wide README template. Mandatory sections: "What & Why," "Quick Start (Install + Run)," "Key Concepts," "API/Usage Examples," "Project Structure."
-   **P2 (Optimize):** Add a "Documentation" badge to each repo's README, linking to a central doc quality dashboard. Implement a pre-commit hook that checks for placeholder text in README.md.
