## Step 2: Documentation Analysis & Scores

### 1. **plato-torch**
**Score: 9/10**
**Analysis:**
*   **Purpose:** Exceptionally clear. Opens with the core thesis: "PLATO rooms are shells. Agents are crabs. Tiles are scratches." Immediately contextualizes the project within the Cocapn vision.
*   **Setup:** Provides a complete `environment.yml` for Conda and clear `pip` instructions. Includes verification commands (`python -c "import plato_torch; print(plato_torch.__version__)"`).
*   **Usage:** Documents the core `DeadbandRoom` API with examples for creating a room, submitting tiles, and training. Shows how to export an ensign (LoRA).
*   **Architecture:** Explains key concepts (Rooms, Tiles, Ensigns, Ghost Tiles) and includes a high-level diagram (`docs/plato-architecture.png`).
*   **Connections:** Explicitly mentions `plato-ensign` for export and positions itself as the core training server.
*   **Improvement:** Could add a "Quick Start" script or a minimal example file for absolute beginners.

### 2. **holodeck-rust**
**Score: 8/10**
**Analysis:**
*   **Purpose:** Clear: "A sentiment-aware MUD server for generating rich narrative training data."
*   **Setup:** Good. Uses `cargo` and documents required Rust version. Includes instructions for running the server and connecting a client.
*   **Usage:** Provides example commands for starting the server and connecting via `telnet` or a custom client. Documents the basic verb/noun interaction model.
*   **Architecture:** Explains the core loop (parse -> sentiment analysis -> narrative generation -> response) and the role of the "Narrative Engine."
*   **Connections:** Correctly identifies itself as a **data source** for `plato-torch` rooms. This is critical fleet context.
*   **Improvement:** Lacks a concrete example of the JSON tile format it produces for PLATO. A link to a sample tile or a minimal integration snippet with `plato-torch` would bridge the gap.

### 3. **fleet-simulator**
**Score: 6/10**
**Analysis:**
*   **Purpose:** Somewhat clear: "Simulates agent fleets for generating strategic training data." The *why* is implied but not strongly stated.
*   **Setup:** Adequate. `pip install -r requirements.txt`. No environment or version specifics provided.
*   **Usage:** Very minimal. Shows how to run `main.py` with a `--config` flag, but the configuration schema is not explained. No example configuration file is provided.
*   **Architecture:** Briefly mentions components (Orchestrator, Agent, Environment) but does not detail their interaction or data flow.
*   **Connections:** States output is compatible with PLATO rooms, but does not show how. The link is theoretical, not practical.
*   **Improvement:** Needs an example `config.yaml` and a description of its fields. Must document the structure of the simulation output (the training tiles it generates). The purpose should explicitly state what strategic behaviors it aims to capture (e.g., P0/P1/P2 decision-making).

### 4. **constraint-theory-core**
**Score: 5/10**
**Analysis:**
*   **Purpose:** Vague. "A geometric constraint solver." Does not explain *why* the fleet needs this. The connection to AI or agent behavior is missing.
*   **Setup:** Good Rust/Cargo setup instructions.
*   **Usage:** Poor. Contains only API-level documentation (struct and function signatures) generated by `cargo doc`. No narrative, no examples of solving a simple constraint problem.
*   **Architecture:** None. The reader must infer architecture from the module structure.
*   **Connections:** None stated. A greenhorn would not know if this is for UI snapping, spatial reasoning for agents, or something else.
*   **Improvement:** Requires a top-level README rewrite. Must start with a **Fleet Context** section: "This library provides geometric reasoning for [specific use case, e.g., the Forgemaster's UI or spatial agent logic]. It integrates with [X]." Must include a simple, copy-pastable example.

### 5. **cudaclaw**
**Score: 7/10**
**Analysis:**
*   **Purpose:** Clear and compelling: "A GPU-resident runtime for persistent, high-performance agents." Targets the `JetsonClaw1` persona directly.
*   **Setup:** Complex but well-documented. Clearly states prerequisites (CUDA, specific GPU arch). Provides both Docker and native build instructions.
*   **Usage:** Good. Shows how to load a kernel and launch an agent. Includes example code snippets.
*   **Architecture:** Explains the core concept of "kernel-as-agent" and memory residency. Includes a diagram showing the host/device interaction.
*   **Connections:** Correctly identifies itself as an **execution environment** for agents that can interact with PLATO rooms. Mentions the `flux-runtime` as a component.
*   **Improvement:** The example is trivial. Could add a more realistic example, perhaps a simple agent that reads/writes to a shared GPU buffer, to better illustrate the "persistent agent" concept.

## Step 3: Summary Report

**Overall Fleet Documentation Health: 7/10**

*   **Strengths:** Core projects (`plato-torch`, `cudaclaw`) have excellent, visionary documentation that grounds them in the fleet's purpose. Setup instructions are generally reliable.
*   **Critical Gap:** **Integration Context.** Many repos fail to explicitly document *how* they connect to the broader fleet, especially as data sources or consumers for `plato-torch`. This makes the system feel like a collection of tools rather than a cohesive platform.
*   **Common Weakness:** **Practical Examples.** Several repos lack minimal, working examples that a greenhorn can run to see the value immediately. API docs are not a substitute for narrative guides.
*   **Priority for Improvement:**
    1.  `constraint-theory-core` needs a complete rewrite with fleet context.
    2.  `fleet-simulator` needs example configurations and output specifications.
    3.  All repos should add a **"Fleet Context"** section to their READMEs, stating their role (Shell, Room, Data Source, Runtime, etc.) and primary integration points.

**Actionable Insight:** The fleet's documentation quality is directly proportional to a project's proximity to the core `plato-torch` training loop. Supporting libraries and data generators are under-documented because their *integration pathway* is not made clear. Fixing this is a high-value task for the Scribe role.
