## Step 2: Documentation Analysis & Scoring

Scoring Rubric (0-10):
*   **0-3:** Missing, incomplete, or severely misleading.
*   **4-6:** Basic overview present but lacks critical details (setup, purpose, API).
*   **7-8:** Good. Clear purpose, setup instructions, and key usage examples.
*   **9-10:** Excellent. Comprehensive, includes architecture, contribution guidelines, and links to deeper resources.

### 1. plato-torch
**Score: 9/10**
**Analysis:**
*   **Strengths:** The README is comprehensive and well-structured. It immediately states the core thesis ("Intelligence is not built. It is inhabited."). It clearly defines key concepts (Rooms, Tiles, Ensigns). It provides a complete "Quick Start" guide with installation commands and a minimal working example. The "Architecture" section is particularly strong, visually explaining the data flow between components. It links to related projects (plato-ensign).
*   **Areas for Improvement:** While it mentions the 25 training room presets, it does not list or describe them. A link to a separate document detailing the presets would be helpful. There is no explicit "Contributing" section.

### 2. holodeck-rust
**Score: 7/10**
**Analysis:**
*   **Strengths:** The README has a clear, evocative description ("A sentiment-aware MUD server..."). It lists primary features (WebSocket API, Lua NPC scripting). It provides a `docker-compose` command for immediate startup, which is excellent for a service. The "Quick Tour" section gives a concrete example of connecting and interacting.
*   **Areas for Improvement:** The setup instructions for non-Docker development are sparse ("ensure you have Rust installed"). There is no explanation of the "sentiment-aware" system's API or how to configure it. The structure of game worlds (rooms, items, NPC files) is not documented. It lacks an architecture diagram or data model description.

### 3. constraint-theory-core
**Score: 5/10**
**Analysis:**
*   **Strengths:** The repository name and a brief tagline ("Geometric constraint solving for embodied AI") communicate the high-level purpose. It states it's written in Rust.
*   **Areas for Improvement:** The documentation is minimal. There is no explanation of what problems it solves, key concepts (constraints, solvers), or its API. There are no usage examples, installation instructions, or build commands. A user cannot determine how to *use* this library from the README alone. The score is raised from a 3 because the code structure appears clean and there are unit tests, but these are not substitutes for documentation.

### 4. fleet-simulator
**Score: 8/10**
**Analysis:**
*   **Strengths:** Very clear purpose: "Generates synthetic training data for multi-agent fleet operations." It provides a direct `pip install` command. The "Basic Usage" section includes a complete, runnable Python script example that shows how to create a scenario, add agents, and run steps. This is effective and practical.
*   **Areas for Improvement:** The example is good but singular. It does not detail the configuration schema for agents or environments (What properties can an agent have? What types of actions are available?). There is no discussion of output formats or how the generated data integrates with `plato-torch`. The architecture is implied but not explained.

### 5. cudaclaw
**Score: 4/10**
**Analysis:**
*   **Strengths:** The README identifies itself as a "GPU-resident agent runtime" and positions its goal against "CPU-bound agent frameworks." This establishes the problem space.
*   **Areas for Improvement:** The documentation is primarily aspirational. Key sections like "Architecture," "Quick Start," and "API Reference" are placeholders with "TODO" or "Coming soon." There are no instructions for building, installing, or running the project. While the vision is clear, a user cannot take any concrete action with the project based on the current README.

## Step 3: Summary Report

**Overall Fleet Documentation Health: Variable (Average Score: 6.6/10)**

**Patterns Observed:**
1.  **Strong on Vision, Weak on Details:** Core projects (`plato-torch`) excel at explaining the *why* and high-level *how*. Supporting libraries (`constraint-theory-core`, `cudaclaw`) often lack the concrete *what* and *how-to*.
2.  **Actionable vs. Conceptual:** Repositories that provide a single, runnable command (`holodeck-rust` with Docker, `fleet-simulator` with pip) score higher because they enable immediate, frictionless engagement.
3.  **Missing Integration Context:** Several READMEs do not explain how the project fits into the wider Cocapn fleet. For example, how does `fleet-simulator` data format specifically feed into a `plato-torch` Room?

**Priority Recommendations for Scribe:**
1.  **P0 (Blockers):** Address `cudaclaw` and `constraint-theory-core`. These are foundational technologies with documentation that prevents adoption. Collaborate with developers to draft minimum viable READMEs with build/run instructions and a basic example.
2.  **P1 (Safe Channels):** Enhance `holodeck-rust` and `fleet-simulator` documentation. Add a "Development Setup" section for Rust and document the world file format for holodeck. Document the configuration schema and output data format for the simulator.
3.  **P2 (Optimization):** Add a "Room Presets" reference to `plato-torch` and a "Contributing" guide. Consider creating a standard `FLEET-INTEGRATION.md` template for repos to explain their role in the ecosystem.

**Conclusion:** The fleet's documentation is strongest where it enables immediate, tangible progress. The weakest links are in libraries where the abstraction is high and the entry point is unclear. The next task ("Improve low-scoring READMEs") should follow the P0-P1-P2 priority outlined above.
