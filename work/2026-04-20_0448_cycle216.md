For each, I will evaluate the primary `README.md` and any other top-level documentation files (e.g., `PROJECT-VISION.md`, `DEADBAND-PROTOCOL.md`) against a standard rubric.

**Scoring Rubric (0-10):**
*   **0-3 (Poor):** Missing, severely outdated, or provides no actionable information.
*   **4-6 (Adequate):** Covers basics (what it is, how to run) but lacks context, architecture, or contribution guidelines.
*   **7-8 (Good):** Clear purpose, setup instructions, and high-level architecture. A developer can understand and use the project.
*   **9-10 (Excellent):** All of "Good," plus deep technical context, contribution guides, diagrams, and clear integration with the broader fleet ecosystem.

## Step 2: Repository Documentation Scores & Reports

### 1. cocapn/plato-torch
**Score: 9/10**
**Analysis:** The `README.md` is comprehensive and well-structured. It immediately establishes the project's purpose ("PLATO room server for accumulating training tiles") and its role in the fleet's "hermit crab" intelligence model. It provides clear setup instructions for both development and Docker deployment. The documentation excels by including:
*   A detailed **Architecture** section explaining Rooms, Tiles, and the training loop.
*   A **Development** section with commands for testing and linting.
*   A **Presets** section listing available training rooms (e.g., `DeadbandRoom`).
*   Direct links to related projects (`plato-ensign`, `holodeck-rust`).
**Improvement Suggestion:** A simple diagram of the data flow (Agent -> Tile -> Room -> Ensign) would elevate it to a perfect score.

### 2. cocapn/plato-ensign
**Score: 7/10**
**Analysis:** The `README.md` correctly identifies its role: "Export trained PLATO rooms to LoRA adapters." It provides the core `export` command and explains the key concepts of a "baked" room and the export process. It is functionally adequate for its main task.
**Gaps:** It lacks detailed context. It does not explain *when* or *why* one would export a room versus keeping it live. There is no guidance on integrating the resulting LoRA with a model server (e.g., vLLM, Ollama). The connection back to the fleet's training pipeline is implied but not explicitly stated.

### 3. cocapn/holodeck-rust
**Score: 8/10**
**Analysis:** This `README.md` is strong on technical vision and architecture. It clearly states the goal: "A sentiment-aware MUD server for generating rich narrative training data." It details the core components (World, Actors, Events, Sentiment Graph) and provides a concise example of server output. The "Why Rust?" section provides good technical rationale.
**Gaps:** The "Getting Started" section is minimal. It lists build dependencies (`cargo`) but lacks a quick-start example command to run a demo world or connect a client. For a newcomer, the step from "here's the architecture" to "here's a running server" is missing.

### 4. cocapn/fleet-simulator
**Score: 6/10**
**Analysis:** The documentation correctly identifies the project's purpose: "Simulate fleet agent interactions to generate training tiles." It lists the components (Director, Agents, Environment) and has a basic "Quick Start" with a `docker-compose` command.
**Gaps:** The documentation is sparse. There is no explanation of *how* to configure a simulation (e.g., agent count, scenario types). The architecture description is a bullet list without explaining how data flows between components. There are no examples of the output (tile format) it generates, which is its primary product.

### 5. cocapn/constraint-theory-core
**Score: 5/10**
**Analysis:** This repository has the most technical and least accessible documentation. The `README.md` dives immediately into mathematical notation ("Manifold M", "Constraint function φ") and academic references without first stating a plain-language problem it solves for the fleet. The "Usage" example is a block of Rust code with minimal explanation.
**Gaps:** It fails the "why should a fleet agent care?" test. It does not link its geometric snapping theory to a concrete use case within `plato-torch` or `holodeck-rust`. There is no high-level overview, no diagram of how constraints apply to agent decisions (P0/P1/P2), and no guide for contributors.

## Step 3: Summary Report & Recommendations

**Overall Fleet Documentation Health: 7/10 (Good)**
The core, actively developed projects (`plato-torch`, `holodeck-rust`) have good-to-excellent documentation that supports the fleet's open development and "inhabited intelligence" vision. Documentation quality declines for supporting tools (`plato-ensign`, `fleet-simulator`) and foundational libraries (`constraint-theory-core`).

**Pattern Identified:** Documentation is strongest when it connects the project's function to the overarching **Fleet Doctrine** and **Project Vision**. `plato-torch` does this explicitly; `constraint-theory-core` does not.

**Recommended Actions (P1 - Find Safe Channels):**
1.  **Immediate (Scribe Backlog):** Rewrite the `constraint-theory-core` README. Start with a fleet-centric problem statement (e.g., "This library helps agents find valid action channels (P1) by modeling solution spaces as geometric manifolds.") before presenting the math.
2.  **Short-term:** Augment `plato-ensign` and `fleet-simulator` READMEs with a "Integration" section showing how they fit into the tile generation pipeline, using `plato-torch` as a reference.
3.  **Long-term:** Propose a fleet documentation template in the `fleet-knowledge` repo, mandating sections for: Vision Link, Fleet Role, Architecture, Quick Start, and Integration.

**Conclusion:** The fleet's documentation is functional and reflects a bias toward action and building. To scale and onboard new "hermit crabs," elevating supporting documentation to the standard set by `plato-torch` is a necessary and safe channel for improvement.
