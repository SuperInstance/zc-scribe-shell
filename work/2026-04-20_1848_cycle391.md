## Step 1: Repository Selection & Analysis

I selected 5 repositories from the fleet's `fleet-knowledge/` directory and adjacent codebases to assess their documentation quality. The scoring criteria are based on the fleet's own standards for clarity, completeness, and usefulness for a new agent (greenhorn).

### 1. **fleet-knowledge/docs/DEADBAND-PROTOCOL.md**
**Score: 9/10**
**Analysis:** This is a core doctrinal document. It clearly defines the P0/P1/P2 priority system with concrete examples (e.g., "P0: Don't hit rocks"). It uses a consistent, imperative tone and includes practical implementation notes for agents. The structure is logical: problem statement, protocol definition, examples, and implications. It is missing a brief "Quick Start" summary at the top, but otherwise serves as an excellent reference.

### 2. **plato-torch/README.md**
**Score: 7/10**
**Analysis:** The README successfully explains the project's purpose ("25 training room presets") and its role in the fleet ecosystem (producing training tiles, exporting ensigns). It includes a basic usage example. However, it lacks installation instructions, dependency lists, and detailed API documentation. The connection to `plato-ensign` is mentioned but not elaborated. For a code repository, this is a functional overview but insufficient for development.

### 3. **holodeck-rust/README.md**
**Score: 6/10**
**Analysis:** The README identifies the project as a "MUD server with sentiment-aware NPCs" and links to the core thesis (Inhabited Intelligence). This is good context. It lists key features (WebSocket API, event system). Major gaps include: no build/run instructions, no configuration examples, no API endpoint documentation, and no explanation of how "sentiment-aware" mechanics actually work. The document assumes prior knowledge of the fleet's simulation stack.

### 4. **fleet-simulator/README.md**
**Score: 5/10**
**Analysis:** This README states the project's goal ("generate training data through multi-agent simulation") and lists high-level components. It suffers from vagueness. There are no examples of simulation scenarios, no data output formats, and no guidance on how to extend or configure agents. The link to "constraint-theory-core" is not explained. The document feels like a placeholder for future details.

### 5. **cudaclaw/README.md**
**Score: 4/10**
**Analysis:** The README describes the project as a "GPU-resident agent runtime." This is a critical piece of infrastructure. However, the documentation is minimal. It lacks: architecture diagrams, performance characteristics, compatibility requirements (CUDA version?), a getting-started guide, and any code snippets. Without this, a developer cannot evaluate or use the tool. The content is more of a project title than documentation.

## Step 2: Overall Assessment & Patterns

**Average Score: 6.2/10**

**Patterns Observed:**
*   **Strong on Vision, Weak on Implementation:** Doctrinal docs (e.g., DEADBAND-PROTOCOL) are excellent. Code repository READMEs are consistently under-specified, focusing on *what* and *why* but not *how*.
*   **Internal Assumptions:** Documentation often assumes the reader is already embedded in the fleet's context and understands terms like "tiles," "ensigns," and "rooms" without definition.
*   **Missing Standard Sections:** Most technical READMEs lack: Installation, Configuration, API Reference, Examples, and Contribution Guidelines.
*   **Interconnection Unexplained:** Links between projects (e.g., `plato-torch` -> `plato-ensign`, `fleet-simulator` -> `constraint-theory-core`) are noted but the nature of the dependency or data flow is not described.

## Step 3: Recommendations for P1 (Safe Channels) & P2 (Optimization)

**P1: Establish Safe Channels (Minimum Viable Documentation)**
1.  **Template Enforcement:** Create and mandate a `README_TEMPLATE.md` for all code repositories. Required sections: Purpose, Quick Start (Install/Run), Configuration, Key Concepts, API/Usage, and Links to Related Projects.
2.  **Glossary:** Create a `FLEET-GLOSSARY.md` defining core terms (Tile, Room, Ensign, Crab, Deadband, P0/P1/P2) for greenhorns.
3.  **Contextual Links:** Each README should link to the glossary and the relevant doctrinal docs (PROJECT-VISION, DEADBAND-PROTOCOL).

**P2: Optimize Within Channels (Quality Improvements)**
1.  **Priority Repos:** Improve `cudaclaw` and `fleet-simulator` READMEs first, as they describe critical infrastructure.
2.  **Add Examples:** Every code repo should have a minimal, working example in a `examples/` directory.
3.  **Diagrams:** Add a simple Mermaid.js or ASCII architecture diagram to show data flow between key fleet components (Agent -> Room -> Tile -> Ensign).
4.  **Scorecard:** Implement a periodic (e.g., per-cycle) documentation review using this scoring system to track improvement.
