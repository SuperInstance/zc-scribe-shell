## Step 2: Documentation Analysis & Scoring

**Scoring Rubric (0-10):**
*   **0-3:** Missing, minimal, or severely outdated. No practical guidance.
*   **4-6:** Basic structure present (Purpose, Setup, Usage). May be incomplete, unclear, or lack examples.
*   **7-8:** Clear, comprehensive, and useful. Includes purpose, setup, API/usage examples, and contribution guidelines.
*   **9-10:** Exemplary. Includes all of the above plus architecture diagrams, deep dives into concepts, and clear links to related projects.

### 1. plato-torch
**Score: 9/10**
**Analysis:** The README is excellent. It immediately states the project's purpose ("PLATO room server for accumulating training tiles") and core thesis. It provides a clear, concise Quick Start with a working example. The documentation includes:
*   A well-defined `Room` API with method signatures and descriptions.
*   A detailed explanation of key concepts (Tiles, Rooms, Ensigns, Ghost Tiles).
*   A "Presets" section listing available training rooms.
*   A "Development" section for contributors.
*   Links to related projects (`plato-ensign`).
**Strengths:** Conceptual clarity, practical examples, and good structure. It serves both users and contributors.
**Weaknesses:** Could include a simple architecture diagram showing the tile flow from agent to room to ensign.

### 2. holodeck-rust
**Score: 7/10**
**Analysis:** The README is functional and clear. It defines the project ("A MUD server for generating sentiment-aware NPC dialogue") and provides a straightforward "Getting Started" guide with `cargo run`. Key sections include:
*   "Core Concepts" explaining NPCs, Affordances, and Sentiment.
*   A basic example of interacting with an NPC.
*   A "Development" section.
**Strengths:** Good conceptual overview and a simple, working setup.
**Weaknesses:** Lacks detailed API documentation for the `Holodeck` struct or NPC traits. The example is minimal; more complex usage scenarios would be helpful. No clear link to how its output integrates with the broader fleet (e.g., as a tile source for `plato-torch`).

### 3. fleet-simulator
**Score: 6/10**
**Analysis:** The README is adequate but sparse. It correctly identifies the purpose: "A discrete-event simulation framework for generating training data from multi-agent fleets." It provides setup instructions and a basic example of defining a `Fleet` and `Agent`.
**Strengths:** The example code is the strongest part, giving a concrete starting point.
**Weaknesses:** The documentation is very brief. It lacks:
*   Explanation of key simulation concepts (events, clocks, metrics).
*   Details on how to define complex agent behaviors or interactions.
*   Guidance on extracting or formatting the generated "training data."
*   Any mention of integration points with other fleet components.

### 4. constraint-theory-core
**Score: 5/10**
**Analysis:** This is a research-focused repository, and its documentation reflects that. The README states the goal ("Geometric constraint solving for CAD-like snapping in Rust") and dives directly into mathematical motivation and references.
**Strengths:** Excellent for providing academic context and linking to foundational papers (Hoffmann, Fudos).
**Weaknesses:** Severely lacking in practical guidance. There is no "Quick Start," no example of how to use the library, and no API documentation. A new contributor or user would not know how to begin. The `src/lib.rs` file has minimal doc comments.

### 5. cudaclaw
**Score: 4/10**
**Analysis:** The documentation is minimal. The README has a one-line description ("GPU-resident agent runtime") and a "Build" section with CMake instructions.
**Strengths:** The build instructions are clear.
**Weaknesses:** Critical information is missing:
*   What is a "GPU-resident agent"?
*   What problem does this solve?
*   How does it work at a high level?
*   Is there an API? How do you launch an agent?
*   No examples, no usage guide, no architecture overview.

## Step 3: Summary Report

**Overall Fleet Documentation Health: Mixed.**
The fleet has strong foundational documentation for its core thesis (`PROJECT-VISION.md`) and its central component (`plato-torch`). However, documentation quality drops significantly for supporting tools and research projects.

**Patterns Observed:**
1.  **Core vs. Satellite:** Core projects (`plato-torch`) are well-documented. Satellite tools/research projects (`cudaclaw`, `constraint-theory-core`) are under-documented.
2.  **Conceptual vs. Practical:** Some docs (`holodeck-rust`) explain concepts well but lack technical depth. Others (`constraint-theory-core`) are deep on theory but lack practical hooks.
3.  **Integration Blindness:** Most READMEs do not explicitly describe how the project fits into the larger Cocapn fleet architecture.

**Recommendations (P0/P1/P2):**
*   **P0 (Don't break):** Ensure `plato-torch` documentation remains the gold standard. Do not let it degrade.
*   **P1 (Find safe channels):** Prioritize adding basic usage examples and clear purpose statements to the lowest-scoring repos (`cudaclaw`, `constraint-theory-core`). A working example is the minimum viable documentation.
*   **P2 (Optimize):** Enhance mid-tier docs (`fleet-simulator`, `holodeck-rust`) with API details and integration guides. Add a standard "## Fleet Integration" section to all READMEs explaining the project's role in the tile/room/ensign pipeline.

**Top Candidate for Immediate Improvement:** `cudaclaw`. Its critical role ("GPU-resident agent runtime") is obscured by the complete lack of explanatory documentation.
