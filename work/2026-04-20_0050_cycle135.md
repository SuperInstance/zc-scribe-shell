## Step 1: Repository Selection & Analysis

I have selected 5 repositories from the fleet's GitHub organization. The selection is based on a mix of core tools, research projects, and utilities to get a representative sample.

### 1. **plato-torch**
**Path:** `cocapn/plato-torch`
**Purpose:** Core training room server and presets.
**Documentation Score:** **8/10**

**Analysis:**
*   **README.md:** Comprehensive. Includes a clear "What is a PLATO Room?" section, installation instructions (`pip install -e .`), and a detailed API reference for the `DeadbandRoom` and other presets. It explains the core concepts (rooms, tiles, ensigns) well.
*   **Structure:** Well-organized with docstrings in key Python files (e.g., `plato_torch/rooms/deadband_room.py`). Contains a `docs/` directory.
*   **Gaps:** The `docs/` folder is currently empty. Advanced configuration or deployment scenarios (e.g., scaling, persistence) are not detailed. Could benefit from a simple "Quick Start" tutorial that creates a room and submits a tile.

### 2. **holodeck-rust**
**Path:** `cocapn/holodeck-rust`
**Purpose:** MUD server with sentiment-aware NPCs.
**Documentation Score:** **6/10**

**Analysis:**
*   **README.md:** Functional but brief. Describes the project's goal and has basic `cargo` build/run commands. Lacks context on how it integrates with the broader fleet (e.g., connection to PLATO rooms).
*   **Structure:** Code is documented with Rust doc comments. Contains a `examples/` folder, which is good.
*   **Gaps:** No architecture overview. No explanation of the "sentiment-aware" system or NPC API. The `examples/` directory is not referenced or explained in the README. Missing a clear definition of its role within the Cocapn ecosystem.

### 3. **fleet-simulator**
**Path:** `cocapn/fleet-simulator`
**Purpose:** Generates training data through multi-agent simulation.
**Documentation Score:** **4/10**

**Analysis:**
*   **README.md:** Very sparse. Only contains a short description and a list of related research papers (`fleet-research/`). No instructions on how to run or use the simulator.
*   **Structure:** The codebase has minimal inline comments. There is a `configs/` directory but no example configuration file or explanation of parameters.
*   **Gaps:** Critical lack of usage documentation. A potential user cannot determine how to launch a simulation, define agents, or collect output data. The link to research is good but does not replace practical docs.

### 4. **constraint-theory-core**
**Path:** `cocapn/constraint-theory-core`
**Purpose:** Rust library for geometric constraint solving.
**Documentation Score:** **7/10**

**Analysis:**
*   **README.md:** Good technical overview. Explains the core problem (geometric snapping) and the library's approach. Includes build instructions and a link to a `examples/` directory.
*   **Structure:** The `examples/` folder contains a working example (`basic_snapping.rs`) that is well-commented and demonstrates the API effectively. Rust doc comments are present on public functions.
*   **Gaps:** The README could include a simple, copy-pastable code snippet showing basic usage before directing to examples. Lacks API documentation generated via `cargo doc` (though this may be intentional). No mention of performance characteristics or limitations.

### 5. **service-guard**
**Path:** `cocapn/service-guard`
**Purpose:** Bash script to monitor and restart fleet services.
**Documentation Score:** **9/10**

**Analysis:**
*   **README.md:** Excellent. Clearly states the problem it solves. Provides a dependency list. Has a detailed "Usage" section with examples for installation, configuration, and manual checks. Includes a "How it Works" section explaining the monitoring logic.
*   **Structure:** The main script (`service-guard.sh`) is thoroughly commented, explaining each function and check. Configuration is done via a separate, well-documented `config.env.example` file.
*   **Gaps:** Very minor. Could include a troubleshooting section for common failure modes (e.g., permission errors, `systemd` integration).

## Step 2: Summary Report & Recommendations

**Overall Fleet Documentation Health: Variable (Average Score: 6.8/10)**

The fleet has strong documentation in its core operational tools (`plato-torch`, `service-guard`) and libraries (`constraint-theory-core`). These repos provide enough context and instruction for a contributor to understand and use them.

The primary weakness lies in **research and simulation projects** (`fleet-simulator`, and to a lesser extent `holodeck-rust`). Their documentation assumes prior context or is incomplete, creating a barrier to entry and collaboration. This aligns with the P0/P1 doctrine: without clear docs (a safe channel), contributors cannot effectively optimize (P2) within that project.

**Immediate Recommendations (P1 - Create Safe Channels):**
1.  **`fleet-simulator`:** Create a minimal `GETTING_STARTED.md` with steps to run a basic simulation and log output. Add an example configuration file in `configs/`.
2.  **`holodeck-rust`:** Expand the README to include a "Role in the Fleet" section and a basic example of connecting to and interacting with the MUD server.
3.  **`plato-torch`:** Populate the `docs/` directory with a basic tutorial. This is a high-priority fix for a core system.

**Pattern Observation:**
The best documentation (`service-guard`) follows a clear formula: **Problem -> Solution -> Dependencies -> Usage -> How It Works.** Recommending this as a fleet-wide README template would raise the baseline quality.
