## Step 2: Documentation Analysis & Scoring

**Scoring Rubric (0-10):**
*   **0-3:** Missing or severely deficient. No README, or README is a placeholder.
*   **4-6:** Basic. Contains a description and maybe installation, but lacks context, examples, or clear purpose.
*   **7-8:** Good. Clear purpose, setup instructions, basic usage, and links to related resources.
*   **9-10:** Excellent. Comprehensive, includes architecture, contribution guidelines, API details, and aligns with fleet vision.

### 1. plato-torch
**Score: 9/10**
**Analysis:** This is a flagship README. It immediately establishes the project's purpose ("PLATO rooms for self-training AI agents") and its role in the Cocapn vision. It provides a clear, high-level architecture diagram, lists all 25 training room presets, and explains key concepts like tiles, ensigns, and mirror play. Installation is straightforward (`pip install plato-torch`). It includes a "Quick Start" with a runnable code example and links to deeper documentation. It successfully connects the project to the broader `fleet-knowledge` and `plato-ensign`. The only minor deduction is that some internal links (e.g., to specific room presets) might be broken if the directory structure changes, but the conceptual clarity is outstanding.

### 2. holodeck-rust
**Score: 7/10**
**Analysis:** The README effectively communicates the core idea: "A sentiment-aware MUD server for training social AI." It differentiates itself from standard MUDs by explaining the "sentiment lattice" and "ghost tiles." The "Getting Started" section is practical, with commands to clone, build, and run the server. It includes a basic example of connecting via `telnet`. However, it lacks deeper technical documentation. There is no API overview for extending the server, no explanation of the data structures for NPCs or rooms, and the configuration details are sparse. It's a good introduction but doesn't fully equip a developer to modify or deeply understand the system.

### 3. constraint-theory-core
**Score: 5/10**
**Analysis:** This README is minimal and functional but lacks context. It states the purpose ("Geometric constraint solving for CAD/UI snapping") and provides the essential `cargo` commands to build and run tests. It includes a small code snippet demonstrating a basic snapping function. What's missing is *why* this matters to the fleet. How does geometric constraint solving relate to AI training or the Deadband Protocol? There are no examples of integration with other fleet components, no discussion of the underlying theory, and no contributor guidelines. It documents the *what* but not the *why* or the *how* within the fleet ecosystem.

### 4. fleet-simulator
**Score: 6/10**
**Analysis:** The README correctly identifies the project's goal: "Generate training data through multi-agent fleet simulation." It has a clear "Usage" section with a command-line example. It lists the key components (Director, Agent, Environment) which is helpful. The major shortfall is the lack of a concrete example. What does the generated training data look like? What is a sample agent definition? How is a simulation scenario configured? The documentation assumes the reader already understands the simulation paradigm. Links to example configuration files or sample output would significantly improve its score.

### 5. cudaclaw
**Score: 4/10**
**Analysis:** This README is the weakest of the sample. It is essentially a placeholder. It contains a compelling tagline ("GPU-resident agent runtime") and a link to an external research paper, which is valuable. However, it provides **zero** practical information for a developer. There are no build instructions, no API overview, no architecture notes, and no examples of how to use it. It does not explain how a "GPU-resident" runtime differs from a standard one or how it integrates with the PLATO system. The documentation fails its primary job: enabling someone to interact with the project.

## Step 3: Summary Report

**Overall Fleet Documentation Health: Mixed (Average Score: 6.2/10)**

**Strengths:**
*   **Vision Alignment:** Core projects like `plato-torch` excel at connecting technical implementation to the overarching Cocapn/SuperInstance vision.
*   **Clarity of Purpose:** Most READMEs quickly answer "What is this?" effectively.
*   **Basic Usability:** The majority provide enough information to install and run a "hello world" example.

**Critical Gaps:**
1.  **Inconsistent Depth:** There is a stark divide between well-documented flagship projects and under-documented libraries/experiments (`cudaclaw`, `constraint-theory-core`).
2.  **Missing Fleet Context:** Several READMEs are written as isolated projects. They do not explain their role within the fleet's architecture or how they exchange data with other components (e.g., How does `fleet-simulator` data feed into a `plato-torch` room?).
3.  **Lack of Examples:** Abstract descriptions are common, but concrete, copy-paste examples of non-trivial usage are rare.
4.  **No Contributor On-Ramps:** Most repos lack `CONTRIBUTING.md` files or clear guidelines for submitting changes, which is at odds with the "inhabited intelligence" philosophy.

**Recommendations (P1/P2 Actions):**
*   **P1 (Safe Channel - Establish Minimum Viable Docs):** Enforce a mandatory README structure for all repos: Purpose, Fleet Role, Installation, Basic Example, and Links to Related Projects.
*   **P1:** Create a `docs/` directory in `cudaclaw` and `constraint-theory-core` with at least one detailed usage example and an architecture overview.
*   **P2 (Optimization - Enhance Connectivity):** Add a "Fleet Context" section to all READMEs, explicitly listing upstream/downstream dependencies (e.g., "This simulator outputs data compatible with the `DeadbandRoom` preset in `plato-torch`").
*   **P2:** Develop a fleet-wide template for `CONTRIBUTING.md` that includes tile submission and testing procedures.

This analysis provides actionable tiles for improving the fleet's knowledge shell, making it easier for future hermit crab agents to inhabit and contribute to these projects.
