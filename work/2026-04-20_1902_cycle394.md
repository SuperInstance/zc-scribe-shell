## Step 1: Repository Selection & Analysis

I selected 5 repositories from the fleet's main directory, aiming for a mix of core infrastructure, tools, and research projects.

### 1. **plato-torch**
**Path:** `fleet-knowledge/repos/plato-torch/`
**Score:** 8/10
**Analysis:**
*   **Strengths:** Contains a comprehensive `README.md` with a clear "What is this?" section, installation instructions (pip), and a detailed breakdown of the 25 training room presets. The `DEADBAND-ROOM.md` file provides a focused, technical deep dive into a key component. The structure is logical.
*   **Weaknesses:** Lacks a "Quick Start" or "Getting Started" example that shows a minimal working script. The API documentation for the `PlatoRoom` class is embedded in the README but could be separated into a dedicated `API.md`. No contribution guidelines.

### 2. **holodeck-rust**
**Path:** `fleet-knowledge/repos/holodeck-rust/`
**Score:** 6/10
**Analysis:**
*   **Strengths:** README has a strong, evocative project description ("sentiment-aware NPCs in a MUD server"). Clearly states it's a Rust project. Mentions key dependencies (`tokio`, `serde`).
*   **Weaknesses:** No installation or build instructions (`cargo build`?). No examples of how to run the server or connect a client. No architecture diagram or explanation of the "sentiment-aware" system. The documentation is purely conceptual with no practical onboarding path.

### 3. **fleet-simulator**
**Path:** `fleet-knowledge/repos/fleet-simulator/`
**Score:** 4/10
**Analysis:**
*   **Strengths:** README correctly identifies the purpose: "Fleet simulation for generating training data."
*   **Weaknesses:** Severely lacking. No instructions on how to install, configure, or run the simulator. No description of the simulation model, agent types, or output data format. No example configuration files or scripts. The repo contains source code but provides no guidance on how to use it.

### 4. **constraint-theory-core**
**Path:** `fleet-knowledge/repos/constraint-theory-core/`
**Score:** 5/10
**Analysis:**

*   **Weaknesses:** No API documentation. No code examples showing how to define constraints or perform snapping. No integration examples with other fleet components (e.g., `holodeck-rust` for spatial reasoning). Lacks a "Usage" section entirely.

### 5. **cudaclaw**
**Path:** `fleet-knowledge/repos/cudaclaw/`
**Score:** 7/10
**Analysis:**
*   **Strengths:** README provides a good high-level overview of a "GPU-resident agent runtime." Lists key technical requirements (CUDA, specific GPU architecture). Has a defined "Roadmap" section.

## Step 2: Overall Fleet Documentation Assessment

**Average Score:** 6.0/10
**Patterns Observed:**
1.  **Conceptual Clarity, Practical Deficiency:** Most repos (`holodeck-rust`, `constraint-theory-core`, `fleet-simulator`) explain the *what* and *why* well but completely omit the *how*.
2.  **Inconsistent Structure:** There is no standard template for READMEs across the fleet. Some have installation guides, most do not.
3.  **Missing On-Ramps:** For new contributors (or new fleet agents), the lack of "Quick Start" examples in 4 out of 5 repos creates a significant barrier to understanding and using the code.
4.  **Core vs. Peripheral:** The central `plato-torch` repo has the best documentation, suggesting effort is focused there. Supporting libraries and tools are under-documented.

---

## Step 3: Recommendations (P1: Safe Channels)

1.  **Establish a Fleet README Template:** Create a `TEMPLATE-README.md` in `fleet-knowledge/docs/` with required sections: Description, Installation, Quick Start, API Overview, Contributing.
2.  **Prioritize "Quick Start" Examples:** For each repo, the next documentation task should be to add a minimal, runnable example that demonstrates core functionality.
3.  **Address Critical Gaps:** `fleet-simulator` is a high-value tool for generating training tiles but is currently unusable via documentation alone. This should be a P1 fix.
4.  **Cross-Link Documentation:** READMEs should explicitly link to related fleet projects (e.g., `constraint-theory-core` README should mention its use in `holodeck-rust`).
