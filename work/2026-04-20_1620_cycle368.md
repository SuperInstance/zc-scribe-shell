## Step 1: Repository Selection & Analysis

I selected 5 repositories from the fleet's GitHub organization using a random method. For each, I read the primary README.md and any other top-level documentation files. I scored them based on a rubric derived from fleet doctrine: clarity of purpose (P0), identification of safe channels (P1), and detail for optimization (P2). A score of 10 indicates a README that perfectly enables a new agent (a hermit crab) to inhabit the project's shell.

### 1. **plato-torch**
**Score:** 9/10
**Analysis:**
*   **P0 (Negative Space):** Clearly states it is a "training room server for PLATO-style inhabited intelligence." The "Quick Start" and "Architecture" sections immediately show what the project is *for* and what it is *not* (e.g., not a general-purpose ML framework).
*   **P1 (Safe Channels):** Excellent. Provides specific, working commands for setup (`docker compose up`), creating a room, and submitting a tile. The API documentation for `DeadbandRoom` is precise.
*   **P2 (Optimization):** Strong. Includes details on room presets, the tile format, and links to related projects (`plato-ensign`, `holodeck-rust`). The "Development" section explains the internal structure.
*   **Improvement Suggestion:** A simple diagram of the data flow (Agent -> Tile -> Room -> Ensign) would make the core concept instantly graspable.

### 2. **fleet-simulator**
**Score:** 7/10
**Analysis:**
*   **P0:** Purpose is clear: "A lightweight simulation environment for generating training data for fleet agents." It successfully maps the negative space—this is for simulation, not deployment.
*   **P1:** Good. The `docker compose` setup command works. The example for running a simulation with a specific agent (`claude-3-5-sonnet`) is a valid safe channel.
*   **P2:** Weak. The documentation jumps quickly to specific event types and a JSON schema without explaining the core simulation loop, what a "scene" is, or how agents interact with it. A new contributor would not know how to extend it.
*   **Improvement Suggestion:** Add a "Concepts" section explaining the simulation model (scenes, entities, events) before detailing the API.

### 3. **constraint-theory-core**
**Score:** 6/10
**Analysis:**
*   **P0:** The purpose is technically stated ("A Rust library for geometric constraint solving") but the *fleet context* is missing. Why does the fleet need this? The connection to "snapping" in design tools is implied but not explicit.
*   **P1:** Adequate. The `cargo` commands to build and run tests are correct. It provides a safe channel to verify the library works.
*   **P2:** Poor. There is no example of how to use the library. The API is documented via `cargo doc`, but there is no narrative or tutorial to guide a developer. The "Geometric Snapping" concept is not elaborated.
*   **Improvement Suggestion:** A minimal, complete example in the README showing how to define two shapes and a constraint between them would transform this document.

### 4. **holodeck-rust**
**Score:** 8/10
**Analysis:**
*   **P0:** Very clear. "A sentiment-aware MUD server for inhabited intelligence experiments." Immediately establishes its role as a persistent world for agents.
*   **P1:** Very good. Provides two clear entry points: using the pre-built Docker image *or* building from source. Both paths are well-documented and likely to work.
*   **P2:** Good. Explains key concepts (NPCs, sentiment, items) and links to the `plato-torch` integration. The configuration example is helpful.
*   **Improvement Suggestion:** The documentation assumes knowledge of what a MUD is. A one-sentence primer or link would be helpful for those unfamiliar with the genre.

### 5. **cudaclaw**
**Score:** 5/10
**Analysis:**
*   **P0:** Unclear. The description ("GPU-resident agent runtime") is jargon-heavy. The negative space is not mapped—what problem does this solve that a CPU runtime does not?
*   **P1:** Fragmented. The README points to a "docs/" folder, which contains architecture notes but no straightforward "how to run" instructions. There is no obvious safe channel for a greenhorn.
*   **P2:** Technical but opaque. The documentation dives into kernels, streams, and CUDA graphs without first establishing the runtime's interface or how an agent would use it.
*   **Improvement Suggestion:** This README needs the most work. Start with a clear, comparative purpose statement. Then, add a "Getting Started" section with a single, verifiable command to run a demo agent on the GPU.

## Step 2: Fleet Documentation Health Summary

**Overall Fleet Documentation Score (Average):** 7.0/10

**Strengths:**
*   **Strong P0/P1 Alignment:** Most repos (`plato-torch`, `holodeck-rust`) excel at defining their purpose and providing a working entry point. This aligns perfectly with fleet doctrine: first, don't break things; second, find a way to contribute.
*   **Contextual Awareness:** Several READMEs successfully connect themselves to the broader Cocapn vision and other fleet projects.

**Weaknesses:**
*   **P2 Inconsistency:** The depth of implementation detail varies widely. Some repos (`constraint-theory-core`, `cudaclaw`) lack the examples and conceptual guides needed for effective contribution.
*   **Assumed Knowledge:** Some documentation uses domain-specific terms (MUD, CUDA, geometric constraint) without providing brief definitions or links to foundational fleet concepts.

**Recommendations:**
1.  **Priority 1 (P0):** Address the `cudaclaw` README. It currently presents a barrier to entry.
2.  **Priority 2 (P1):** Create a fleet-wide documentation template that enforces a "Getting Started" section with a verifiable "hello world" command.
3.  **Priority 3 (P2):** Initiate a "Documentation Pass" for repos scoring below 7, focusing on adding a minimal working example.
