## Step 1: Repository Selection & Analysis

I selected 5 repositories from the fleet's `fleet-knowledge/` directory and adjacent projects to assess their current documentation state. The scoring criteria (0-10) are based on: clarity of purpose, setup instructions, API/usage examples, and adherence to fleet conventions.

### 1. **plato-torch**
**Score: 8/10**
**Location:** `fleet-knowledge/docs/plato-torch/`
**Analysis:**
*   **Strengths:** Excellent high-level vision in `PROJECT-VISION.md`. The `README.md` clearly states the project's purpose ("25 training room presets, self-training rooms"). It includes a concise architecture diagram and defines key terms (Rooms, Tiles, Ensigns). It successfully maps the conceptual model from the vision to implementation.
*   **Weaknesses:** Lacks specific, runnable code examples for creating a Room or submitting a Tile. The setup instructions are implied but not explicit (e.g., "pip install -r requirements.txt"). No contribution guidelines or testing instructions are present.
*   **Improvement Path:** Add a "Quick Start" section with a minimal working example. Document the Room class interface.

### 2. **holodeck-rust**
**Score: 6/10**
**Location:** `fleet-knowledge/docs/holodeck-rust/`
**Analysis:**
*   **Strengths:** The `README.md` has a strong, evocative description ("MUD server with sentiment-aware NPCs"). It successfully connects its purpose to the larger fleet doctrine ("generates training tiles for PLATO rooms"). It lists key technologies (Rust, Tokio, SQLite).
*   **Weaknesses:** No setup or build instructions. No API documentation or example of how to run the server, connect a client, or interact with an NPC. The documentation is purely descriptive, not operational.
*   **Improvement Path:** Add `## Building` and `## Running` sections. Provide a minimal client connection example. Document the NPC sentiment API.

### 3. **fleet-simulator**
**Score: 4/10**
**Location:** `fleet-knowledge/docs/fleet-simulator/`
**Analysis:**
*   **Strengths:** The `README.md` correctly identifies its role ("Fleet simulation for training data"). It mentions integration points (`plato-torch`, `holodeck-rust`).
*   **Weaknesses:** Extremely vague. No explanation of *what* is being simulated (agent interactions? network traffic?). No architecture, no setup, no usage examples. It reads as a placeholder.
*   **Improvement Path:** This README requires a complete rewrite. Define the simulation's scope, inputs, outputs, and data format. Add a diagram of the simulation loop.

### 4. **constraint-theory-core**
**Score: 7/10**
**Location:** `fleet-knowledge/docs/constraint-theory-core/`
**Analysis:**
*   **Strengths:** Clear, technical focus ("Rust geometric snapping"). The `README.md` effectively explains the core theory (P0/P1/P2) and its application to geometry. It includes a useful code snippet demonstrating the `ConstraintSolver` struct.
*   **Weaknesses:** The example snippet is incomplete (missing imports, context). No `Cargo.toml` dependencies are listed. No guidance on integrating this library into other fleet projects.
*   **Improvement Path:** Provide a complete, copy-pastable example in `examples/`. Add a "Usage in Fleet" section.

### 5. **cudaclaw**
**Score: 5/10**
**Location:** `fleet-knowledge/docs/cudaclaw/`
**Analysis:**
*   **Strengths:** Concise definition ("GPU-resident agent runtime"). Acknowledges its experimental status.
*   **Weaknesses:** Lacks all implementation details. No mention of required CUDA version, build prerequisites, or how an agent interfaces with the runtime. The connection to `flux-runtime` is noted but not explained.
*   **Improvement Path:** Document the hardware/software prerequisites. Provide a minimal agent skeleton that can be loaded into the runtime. Describe the lifecycle of a GPU-resident agent.

## Step 2: Overall Fleet Documentation Health Summary

**Average Score: 6.0/10**

**Patterns Observed:**
1.  **Strong Vision, Weak Execution:** Most repos (`plato-torch`, `holodeck-rust`) articulate their *purpose* well within the fleet's philosophical context but fail to provide the concrete instructions needed for a new contributor to *run* them.
2.  **Inconsistent Depth:** Documentation quality varies significantly. `plato-torch` and `constraint-theory-core` are functional starting points, while `fleet-simulator` is essentially a stub.
3.  **Missing Conventions:** None of the repos followed a standardized README structure (e.g., Purpose, Build, Run, Test, Contribute). There are no links to contributing guidelines or code of conduct.

**Critical Gaps:**
*   **No "Getting Started" Guide:** The fleet lacks a top-level onboarding document that explains how to set up a development environment and interact with the core systems.
*   **Absence of Examples:** The `examples/` directory is missing or underutilized across all projects.
*   **Undocumented Dependencies:** Inter-project relationships (e.g., how `holodeck-rust` generates tiles for `plato-torch`) are described conceptually but not technically.

## Step 3: Recommended Actions (P1: Safe Channels)

1.  **Immediate (P1):** Establish a `README_TEMPLATE.md` in `fleet-knowledge/docs/` mandating sections: `## Purpose`, `## Architecture`, `## Getting Started`, `## Examples`, `## API Reference`, `## Contributing`.
2.  **Targeted Rewrites (P1):** Prioritize rewriting the `fleet-simulator` README and enhancing `holodeck-rust` with build/run instructions, as these block understanding of key data flows.
3.  **Create Examples (P2):** For `plato-torch` and `constraint-theory-core`, develop canonical examples that demonstrate their core use case. This is an optimization within the safe channel of improved docs.
