## Step 2: Documentation Analysis & Scoring

Scoring Rubric (0-10):
*   **0-3:** Missing, incomplete, or severely misleading.
*   **4-6:** Basic overview exists but lacks critical details (installation, API, purpose).
*   **7-8:** Good. Clear purpose, setup, and usage. Could improve with examples or architecture.
*   **9-10:** Excellent. Comprehensive, includes examples, architecture, contribution guidelines, and clear vision.

### 1. plato-torch
**Score: 8/10**
**Analysis:**
*   **Strengths:** README.md provides a strong, clear "North Star" vision statement upfront ("Intelligence is not built. It is inhabited."). It effectively explains core concepts (rooms, tiles, ensigns). Includes a basic "Quick Start" with installation commands (`pip install -e .`). Documents the 25 preset training rooms. Has a dedicated `USAGE.md` with detailed API examples for room creation and interaction.
*   **Areas for Improvement:** The main README could benefit from a more prominent "Getting Started" section. The architecture of how `plato-torch` interacts with a PLATO server is implied but not explicitly diagrammed. Contribution guidelines are absent.

### 2. holodeck-rust
**Score: 6/10**
**Analysis:**
*   **Strengths:** README.md has a compelling tagline ("A sentiment-aware MUD server for training social AI"). It lists key features clearly (WebSocket API, NPC memory, emotion engine). Provides a simple `docker-compose up` command to run.
*   **Areas for Improvement:** Lacks any API documentation or examples of how to connect an agent or interact with NPCs. No explanation of the "sentiment" model or how NPC memory works. The "Quick Start" does not show what a successful run looks like or how to verify it's working. No testing or development instructions.

### 3. constraint-theory-core
**Score: 5/10**
**Analysis:**
*   **Strengths:** README.md states the purpose clearly ("A Rust library for geometric constraint solving and snapping"). It lists core types (`Point`, `Line`, `Constraint`) and has a link to generated Rust docs (docs.rs).
*   **Areas for Improvement:** There is no example code snippet in the README showing basic usage. No guidance on how to add it as a dependency (`Cargo.toml`). The link to docs.rs is helpful but requires the user to leave the repo to understand *how* to use the library. No context on how this integrates with the larger fleet (e.g., used by `flux-runtime` for UI snapping?).

### 4. fleet-simulator
**Score: 7/10**
**Analysis:**
*   **Strengths:** README.md has a clear, concise purpose statement ("Generates synthetic training data by simulating fleet agent interactions"). It provides a full, copy-pasteable example in a code block, showing how to create a simulator, add agents, run steps, and collect events. This is excellent.
*   **Areas for Improvement:** The installation instructions are vague ("Clone and run"). Does it need `npm install` or `pip install`? The example uses `Agent` and `Event` types without showing their imports or linking to their source. No information on how the generated data is formatted or where it's meant to be used.

### 5. cudaclaw
**Score: 4/10**
**Analysis:**
*   **Strengths:** The repository name and a brief description ("GPU-resident agent runtime") indicate its high-level purpose.
*   **Areas for Improvement:** The README.md is essentially a placeholder. It contains only a broken link to a "Design Doc" and a "Coming Soon" message. There is no information on prerequisites (CUDA version?), build instructions, or a conceptual overview of how an agent resides in GPU memory. This is the weakest documentation in the sample.

## Step 3: Summary Report

**Overall Fleet Documentation Health: Variable (Average Score: 6.0/10)**

The fleet's documentation quality varies significantly between projects that are central to the vision (`plato-torch`) and those that are more specialized or nascent (`cudaclaw`).

*   **Best Practice Observed (`plato-torch`, `fleet-simulator`):** A clear vision/purpose statement followed by a working code example is the most effective pattern. `fleet-simulator`'s complete, runnable example is a standout.
*   **Common Gaps:**
    1.  **Lack of examples:** Most repos would benefit from a minimal, copy-pasteable code snippet that demonstrates core functionality.
    2.  **Ambiguous setup:** Instructions like "clone and run" are insufficient. Specify the build system (Cargo, npm, pip) and any required environment variables.
    3.  **Missing context:** Several repos (`constraint-theory-core`, `holodeck-rust`) do not explain their role within the larger Cocapn architecture. A "## Integration" section would help.
    4.  **Incomplete API docs:** READMEs should at least summarize the main API entry points, even if detailed docs are elsewhere.

3.  **Cross-cutting:** Propose a **Fleet Documentation Standard** template for READMEs to ensure all new repos include: Purpose, Quick Start (with a verifiable example), Integration Context, and API Overview.
