## Step 2: Documentation Assessment & Scoring

I will score each repository's documentation on a 0-10 scale based on the following criteria derived from fleet context and standard best practices:
*   **Clarity & Purpose (P0):** Is the core purpose and thesis of the project immediately clear? Does it state what problem it solves and for whom?
*   **Getting Started (P1):** Does it provide clear, actionable steps for a developer to install, configure, and run a basic example?
*   **Architecture & Concepts (P2):** Does it explain key components, data flow, and important concepts (e.g., "rooms," "tiles," "ensigns")?
*   **Context & Integration:** Does it explain how this project fits into the larger Cocapn fleet ecosystem?
*   **Completeness:** Are there obvious gaps (missing API docs, no contribution guidelines, broken links)?

### **1. plato-torch**
**Score: 8/10**
*   **Clarity & Purpose (Strong):** README opens with a powerful, clear thesis: "PLATO rooms are shells. Agents are crabs. Tiles are scratches." It immediately defines its role as the "training room server" for the fleet.
*   **Getting Started (Good):** Provides a `docker-compose.yml` example and clear commands (`docker-compose up -d`). The "Quick Start" section is effective.
*   **Architecture & Concepts (Very Good):** Explains the core loop (Room -> accumulates Tiles -> can export Ensign). Defines key terms (Room Preset, Ghost Tile, Mirror Play). The included `ROOM-PRESETS.md` is a major strength.
*   **Context & Integration (Good):** Explicitly mentions its relationship to `plato-ensign` and the broader "inhabited intelligence" vision.
*   **Areas for Improvement:** The "API Reference" section is a placeholder link. Adding even a basic overview of key endpoints (e.g., `POST /tile`, `GET /room/:id`) would be helpful. A simple diagram of the room/tile/agent data model could further enhance clarity.

### **2. fleet-simulator**
**Score: 6/10**
*   **Clarity & Purpose (Adequate):** States it is a "training data generator" for multi-agent fleets. The purpose is clear but could be more forcefully tied to the fleet's P0/P1/P2 doctrine.
*   **Getting Started (Weak):** The README lists prerequisites (Rust, Python) but jumps directly to `cargo run --release`. There is no example of what a simulation configuration looks like or what output to expect. A user wouldn't know what to do after it runs.
*   **Architecture & Concepts (Good):** Describes the core components well: `Simulation`, `Agent`, `Environment`, `Event`. The concept of generating "interaction tiles" is correctly highlighted.
*   **Context & Integration (Excellent):** Clearly explains its role in the ecosystem: "Generates the raw interaction data that feeds into `plato-torch` rooms." This is crucial.
*   **Areas for Improvement:** The lack of a basic example configuration file (`simulation.toml` or similar) is the biggest gap. A "Quick Simulation" section showing a minimal 2-agent scenario and the resulting log/tile output would raise this score significantly.

### **3. holodeck-rust**
**Score: 7/10**
*   **Clarity & Purpose (Good):** Clearly defines itself as a "sentiment-aware MUD (Multi-User Dungeon) server" for training social reasoning agents. The purpose is specific and well-articulated.
*   **Getting Started (Good):** Provides `docker-compose` instructions and a link to a `demo.docker-compose.yml`. The commands are clear.
*   **Architecture & Concepts (Adequate):** Explains the high-level idea (NPCs with emotional states, spatial graph) but lacks detail on how sentiment is modeled or how agents interact with the world state. The `docs/` folder exists but wasn't assessed in this scan.
*   **Context & Integration (Good):** Mentions its data can be used to train "ensigns," correctly placing it within the PLATO room pipeline.
*   **Areas for Improvement:** The README could benefit from a simple example of a player/NPC interaction showcasing the sentiment system. A brief overview of the protocol (e.g., Telnet, custom commands) would be helpful.

### **4. constraint-theory-core**
**Score: 4/10**
*   **Clarity & Purpose (Weak):** The README is a single, dense paragraph. It states it's a "geometric constraint solver" but uses highly technical language ("manifold," "Lie groups," "snapping algorithms") without connecting it to a *fleet use-case*. A greenhorn would struggle to understand *why* this exists.
*   **Getting Started (Very Weak):** Only provides `cargo build`. No examples, no tests to run, no indication of what a successful build looks like or how to call the library.
*   **Architecture & Concepts (Poor):** No explanation of the library's structure, main functions, or key data structures. The user is left to decipher the source code immediately.
*   **Context & Integration (Missing):** Does not explain how this solver is used by other fleet components (e.g., for spatial reasoning in `fleet-simulator` or `holodeck-rust`).
*   **Areas for Improvement:** This repo needs the most work. It requires a rewritten README that starts with a *fleet problem statement* (e.g., "Agents need to reason about physical spaces and object relationships"). It needs a "Basic Usage" section with a code snippet solving a simple 2D constraint. It must explain its role in the fleet.

### **5. cudaclaw**
**Score: 5/10**
*   **Clarity & Purpose (Adequate):** Describes itself as a "GPU-resident agent runtime." The core idea (keeping agent state in VRAM) is clear.
*   **Getting Started (Poor):** Instructions are minimal: `make` and `./cudaclaw`. There is no guidance on required CUDA versions, hardware prerequisites, or how to configure/load an agent.
*   **Architecture & Concepts (Weak):** Mentions "kernel fusion" and "zero-copy messaging" but does not elaborate on the runtime architecture. How does an agent interact with it? What is the API?
*   **Context & Integration (Good):** Correctly positions itself as a performance-critical backend for "high-frequency agent swarms."
*   **Areas for Improvement:** Like `constraint-theory-core`, it lacks a concrete example. A minimal example showing how a simple agent loop is defined and launched on the GPU would be transformative. Prerequisites and build troubleshooting tips are also needed.
