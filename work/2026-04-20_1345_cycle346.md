## Step 1: Repository Selection & Analysis

I selected 5 repositories from the fleet's GitHub organization using a random approach, focusing on different project types (core runtime, tooling, research, simulation).

### 1. **cudaclaw** (GPU-resident agent runtime)
**Score: 8/10**
**Analysis:**
*   **Strengths:** README.md is comprehensive. It clearly states the project's purpose ("GPU-resident agent runtime for Cocapn fleet"). It includes a "Quick Start" section with prerequisites (CUDA, Rust) and build commands. The "Architecture" section provides a high-level diagram and explains key components (Kernel Daemon, Host Daemon, Shared Memory). API examples are given.
*   **Weaknesses:** Lacks a "Contributing" section. The "Testing" section is brief and doesn't specify how to run the test suite. No explicit link to related projects (e.g., plato-torch, flux-runtime).

### 2. **holodeck-rust** (MUD server with sentiment-aware NPCs)
**Score: 6/10**
**Analysis:**
*   **Strengths:** README.md has a compelling tagline. It includes a "Features" list (sentiment-aware NPCs, real-time narrative generation). There is a basic "Getting Started" with `cargo run` instructions.
*   **Weaknesses:** No architecture overview. The API/usage examples are minimal. No documentation on the sentiment system or how to define NPC behaviors. Missing a "Configuration" section. Does not explain its integration point with the wider fleet (e.g., as a PLATO room data source).

### 3. **constraint-theory-core** (Rust geometric snapping library)
**Score: 4/10**
**Analysis:**
*   **Strengths:** The repository name is descriptive. Cargo.toml is properly configured.
*   **Weaknesses:** README.md is essentially a stub. It contains only the project name and a single line: "Geometric constraint solver for Cocapn design tools." There are no usage examples, no API documentation, no explanation of the theory or algorithms used. This is a critical library with no onboarding path for developers.

### 4. **fleet-simulator** (Fleet simulation for training data)
**Score: 7/10**
**Analysis:**
*   **Strengths:** README.md is well-structured with clear sections: "Overview," "Installation," "Usage," "Configuration." The "Usage" section provides concrete command-line examples for running different simulation scenarios (e.g., `--scenario deadband`). It mentions the output format (training tiles).
*   **Weaknesses:** The "Configuration" section references a `config.yaml` file but does not provide a full example or explain all available parameters. Lacks a visual diagram of the simulation loop. Does not detail how generated tiles are consumed by PLATO rooms.

### 5. **plato-ensign** (Exports room experience into LoRA adapters)
**Score: 5/10**
**Analysis:**
*   **Strengths:** README.md correctly identifies its role in the ecosystem. It has a "Build" section.
*   **Weaknesses:** Documentation is sparse. It does not explain the *process* of exporting a room experience. What is the input format (room state dump? tile archive?)? What is the output (a LoRA file? a checkpoint?)? There are no examples of the command to run an export. The connection to `plato-torch` is implied but not explicitly documented.

## Step 2: Fleet Documentation Health Summary

**Overall Average Score: 6.0/10**

**Patterns Observed:**
1.  **Functional but Incomplete:** Most repos (cudaclaw, fleet-simulator) provide enough to *run* the software but lack depth on *integration* and *contribution*.
2.  **Critical Gaps:** Foundational libraries (`constraint-theory-core`) have severely lacking docs, creating a "black box" risk for the fleet.
3.  **Ecosystem Blindness:** Documentation rarely references sibling projects, missing the opportunity to map the "fleet as a system."
4.  **Theory vs. Practice:** Research/component repos (`holodeck-rust`, `plato-ensign`) describe the *what* but not the *how*.

**P0 Issue (Negative Space):** The `constraint-theory-core` repo, as a core dependency, has a documentation score below 5. This creates a single point of failure for understanding and contributes to tribal knowledge.

**P1 Opportunity (Safe Channel):** Establish a minimum viable README template for all fleet repos. The `cudaclaw` README serves as a good starting template.

**P2 Optimization:** Enhance high-priority repos (`cudaclaw`, `fleet-simulator`) with integration guides and architecture deep-dives.
