## Step 1: Repository Selection & Scoring

I selected 5 repositories from the fleet by examining the directory structure and choosing a mix of core and auxiliary projects.

### 1. `plato-torch`
**Score: 8/10**
**Analysis:**
*   **Strengths:** The `README.md` is comprehensive. It clearly states the project's purpose ("A PyTorch-based training room server for the Cocapn fleet"), lists key features (25 presets, self-training rooms), and provides a quick start guide with installation commands (`pip install -e .`). It documents the API for creating rooms and has a well-defined "Architecture" section explaining rooms, tiles, and ensigns. It links to related projects (`plato-ensign`).
*   **Weaknesses:** The "Quick Start" assumes prior knowledge of the fleet's ecosystem (e.g., what a "room" is for). There is no explicit "Contributing" section or guidelines for adding new presets. Some internal terminology ("Deadband Protocol," "P0/P1/P2") is used without in-context definition.

### 2. `holodeck-rust`
**Score: 6/10**
**Analysis:**
*   **Strengths:** The `README.md` has a strong, clear tagline: "A sentiment-aware MUD server for training embodied agents." It lists primary features (WebSocket API, Lua scripting, sentiment engine) and dependencies. The "Getting Started" section provides build commands (`cargo build --release`) and a basic server startup command.
*   **Weaknesses:** Documentation is minimal beyond setup. There is no example of a Lua script, no API endpoint documentation, and no explanation of how the "sentiment-aware" system works or how to configure it. The link to "Fleet Research" is broken (points to a non-existent `FLEET-RESEARCH.md`). It lacks a clear connection to the broader PLATO room system.

### 3. `fleet-simulator`
**Score: 4/10**
**Analysis:**
*   **Strengths:** The repository contains a `README.md` file.
*   **Weaknesses:** The documentation is severely lacking. The `README.md` is essentially a placeholder, containing only the project name and a single sentence: "Fleet simulation environment for generating training data." There are no instructions for installation, configuration, or execution. There is no description of what the simulation models, its inputs/outputs, or how it integrates with other fleet components. The repository contains source code but no guidance on how to use it.

### 4. `constraint-theory-core`
**Score: 7/10**
**Analysis:**
*   **Strengths:** The `README.md` is technically focused and clear for its intended audience. It defines the project's purpose ("Geometric constraint solving and snapping library in Rust") and its core abstraction ("Relation Graph"). It provides a concise code example demonstrating the creation of points, a distance constraint, and solving. It lists key features (2D/3D support, solver backends).
*   **Weaknesses:** It lacks a "Why?" section connecting it to the fleet's larger goals (e.g., how this enables "Mirror Play" or specific training rooms). The installation section is vague ("add to your `Cargo.toml`"). There is no advanced usage guide or integration examples with other fleet projects like `holodeck-rust`.

### 5. `cudaclaw`
**Score: 5/10**
**Analysis:**
*   **Strengths:** The `README.md` states a bold vision ("GPU-resident agent runtime") and outlines a high-level architecture with components (Kernel Scheduler, CUDA Graph Pool, Agent Contexts). It lists required hardware (CUDA 12.x, NVIDIA GPU).
*   **Weaknesses:** The documentation is primarily aspirational. There are no build instructions, no API examples, and no working code snippets. Key terms like "resident micro-agents" are not explained. The "Status" section is empty. It functions more as a design document than as usable documentation for developers.

## Step 2: Report & Recommendations

**Overall Fleet Documentation Health: Variable.** Scores range from 8 (`plato-torch`) to 4 (`fleet-simulator`). Core infrastructure projects have functional documentation, but auxiliary and research projects are under-documented.

**Common Patterns & Issues:**
1.  **Assumed Context:** Most READMEs assume the reader is already familiar with Cocapn fleet concepts (Deadband Protocol, tiles, rooms).
2.  **Missing "Getting Started":** Projects often lack step-by-step instructions to go from `git clone` to a running example.
3.  **Broken or Missing Links:** References to internal documents (e.g., `FLEET-RESEARCH.md`) are frequently broken.
4.  **Isolation:** Documentation rarely explains how a project fits into the wider fleet architecture.

**P1 (Safe Channel) Recommendations for Scribe:**
1.  **Immediate Action (Current Task Backlog):** Improve the `fleet-simulator` README. Add a basic purpose, setup instructions, and an example of how to run a minimal simulation.
2.  **Template Creation:** Draft a standard `README.md` template for fleet repositories to ensure consistency. Include required sections: Purpose, Fleet Context, Quick Start, Architecture/Integration, and Development.
3.  **Contextual Bridges:** For projects like `holodeck-rust` and `constraint-theory-core`, add a "Fleet Integration" subsection explaining their role in the training tile pipeline.

**P2 (Optimization) Future Consideration:**
*   Create a central `docs/` directory with a `project-index.md` that maps all repositories, their purpose, state, and inter-dependencies, reducing the load on individual READMEs.
