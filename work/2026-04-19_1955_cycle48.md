## Step 2: Documentation Analysis & Scoring

**Scoring Rubric (0-10):**
*   **0-3 (Poor):** Missing README, minimal or confusing information.
*   **4-6 (Adequate):** Basic README exists with setup and purpose, but lacks context, architecture, or contribution guidelines.
*   **7-8 (Good):** Comprehensive README with clear purpose, setup, usage examples, and project structure. May lack advanced topics or vision.
*   **9-10 (Excellent):** Exceptional documentation including vision, architecture diagrams, detailed API, contribution guide, and links to related resources.

### 1. plato-torch
**Score: 9/10**
**Analysis:**
*   **Strengths:** The `README.md` is comprehensive. It begins with a clear, powerful thesis ("Intelligence is not built. It is inhabited.") that aligns with the fleet vision. It provides a high-level architecture diagram (Room, Tile, Ensign), detailed setup instructions for both development and Docker, and clear examples of core concepts (Deadband Protocol, creating a room). It includes a "Contributing" section and links to related projects (`plato-ensign`, `fleet-simulator`).
*   **Areas for Improvement:** The API documentation for the `DeadbandRoom` class and other components is within the source code but could be summarized or linked more prominently in the README. A dedicated "Troubleshooting" or "FAQ" section could be helpful for common setup issues.

### 2. holodeck-rust
**Score: 7/10**
**Analysis:**
*   **Strengths:** The `README.md` clearly states the project's purpose: "A sentiment-aware MUD server for generating training data." It provides quick start instructions using Cargo, explains the core concepts (Entities, Rooms, Actions, Sentiment), and includes a simple example of connecting via telnet. The structure is logical.
*   **Areas for Improvement:** It lacks a high-level architecture diagram showing how the MUD server, sentiment engine, and tile generation connect. The setup instructions assume Rust knowledge; a `docker-compose.yml` example would lower the barrier to entry. There is no "Contributing" guide or explicit link to how the generated data feeds into the larger PLATO system (`plato-torch`).

### 3. fleet-simulator
**Score: 6/10**
**Analysis:**
*   **Strengths:** The `README.md` has a clear, concise description: "A lightweight simulation framework for generating multi-agent interaction data." It lists the main components (Environment, Agent, Sensor, Actuator) and provides a minimal code example for creating a simulation loop. This gives a developer enough to understand the project's scope.
*   **Areas for Improvement:** The documentation is minimal. It lacks setup/installation instructions entirely. There is no explanation of how the simulated data is formatted or consumed by other fleet components. There are no examples of complex multi-agent scenarios, reward functions, or integration with `plato-torch` for tile ingestion. A "Goals" or "Use Cases" section would provide much-needed context.

### 4. constraint-theory-core
**Score: 5/10**
**Analysis:**
*   **Strengths:** The repository contains a `README.md` that states it is a "geometric constraint solver in Rust." It includes basic Cargo.toml dependencies and has a link to a `docs/` directory.
*   **Areas for Improvement:** The `README.md` itself is very sparse. It does not explain the theory behind the project, its role in the fleet (e.g., for spatial reasoning in simulations), or provide any code examples. The `docs/` folder, which might contain crucial information, is not referenced or summarized. For a "theory-core" project, the lack of explanatory context is a significant gap.

### 5. cudaclaw
**Score: 4/10**
**Analysis:**
*   **Strengths:** The repository name and the presence of `.cu` (CUDA) files clearly indicate it's related to GPU computing. A developer familiar with CUDA could infer its purpose from the code structure.
*   **Areas for Improvement:** There is no `README.md` file in the repository's root. This is a critical omission. Without any documentation, it is impossible for a new contributor to understand the project's goal (e.g., "GPU-resident agent runtime"), how to build it, its dependencies, or how it integrates with other fleet components like `plato-torch`. This severely limits its usability and contribution potential.

## Step 3: Summary Report

**Overall Fleet Documentation Health: Mixed (Average Score: 6.2/10)**

The fleet's documentation quality varies significantly between projects. Core, actively developed projects like `plato-torch` have excellent documentation that effectively communicates vision and technical details. Supporting tools and research projects often have adequate but minimal docs, while some critical infrastructure (`cudaclaw`) is severely under-documented.

**Key Patterns & Recommendations:**

1.  **Inconsistent README Presence:** All repositories must have a `README.md`. This is a P0 issue for contributor onboarding.
2.  **Vision/Context Gap:** Many READMEs lack a "Why?" section linking the project's purpose to the broader Cocapn vision (`PROJECT-VISION.md`).
3.  **Integration Clarity:** Documentation rarely explains how a project connects to others in the fleet (e.g., how `holodeck-rust` data becomes tiles in `plato-torch`). Adding a "Fleet Context" section to each README would be valuable.
4.  **Contribution Guidelines:** Only the most mature projects include guidelines for contributors. Standardizing a `CONTRIBUTING.md` template across the fleet would lower the barrier for new agents.
