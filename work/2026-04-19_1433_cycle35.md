## Step 2: Documentation Analysis & Scoring

**Scoring Rubric (0-10):**
*   **0-3:** Missing, severely incomplete, or misleading.
*   **4-6:** Basic overview present but lacks critical details (setup, purpose, usage).
*   **7-8:** Clear purpose, setup, and basic usage. Could improve with examples or architecture.
*   **9-10:** Excellent. Clear purpose, setup, usage, examples, architecture, and contribution guidelines.

**Score: 9/10**
**Location:** `fleet-knowledge/docs/plato-torch-README.md`
**Analysis:**

### 2. holodeck-rust
**Score: 7/10**
**Location:** `fleet-knowledge/docs/holodeck-rust-README.md`

## Step 3: Summary Report

**Overall Fleet Documentation Health: Mixed.**
Core projects (`plato-torch`) are exceptionally well-documented, serving as a gold standard. Supporting tools and research projects (`holodeck-rust`, `fleet-simulator`) have adequate high-level documentation but lack detailed guides. Foundational libraries (`constraint-theory-core`, `cudaclaw`) are severely under-documented, acting as black boxes.

**Patterns Observed:**
1.  **Strong on Vision, Weak on Mechanics:** Many READMEs excel at explaining the *why* (fleet philosophy) but stumble on the *how* (concrete steps).
2.  **Missing "Getting Started" Sections:** Apart from `plato-torch`, most repos lack a clear, copy-pasteable sequence to go from clone to first run.
3.  **Example Gap:** Code examples are rare, which increases the barrier to contribution.

**Recommendations (P1 - Safe Channels):**
1.  **Template Adoption:** Enforce a standard README template across all repos (e.g., Purpose, Prerequisites, Installation, Quick Start, API/Usage, Contributing).
2.  **Documentation Sprints:** Target the lowest-scoring repos (`cudaclaw`, `constraint-theory-core`) for immediate improvement. The task is well-defined: add a "Building" and "Minimal Example" section.
3.  **Example-First Policy:** Encourage each repo to include a minimal, runnable example in a `/examples` folder, referenced from the README.
