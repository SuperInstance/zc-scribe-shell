## Step 1: Repository Selection & Scoring

I have selected 5 repositories from the fleet's public GitHub organization (`cocapn`). The selection was random based on browsing the list. For each, I read the primary `README.md` and any other top-level documentation files.

**Scoring Rubric (0-10):**
*   **0-3:** Missing, minimal, or severely outdated. No context for purpose or use.
*   **4-6:** Basic structure present (Purpose, Install, Use). May be incomplete, unclear, or lack examples.
*   **7-8:** Clear, complete, and useful. Includes purpose, setup, API/usage examples, and links to related projects.
*   **9-10:** Exemplary. Includes vision, architecture diagrams, contribution guides, and comprehensive examples. Serves as a model.

### 1. **plato-torch**
**Score: 9/10**
**Analysis:** This is a flagship project and its documentation reflects that. The README immediately states the core thesis ("PLATO rooms are shells. Agents are crabs. Tiles are scratches."), establishing context. It provides a clear **Quick Start** with a `docker-compose` command, lists all **25 training room presets**, and explains key concepts like **Ensigns** and **Ghost Tiles**. It links to related projects (`plato-ensign`, `holodeck-rust`). It loses one point because the "API Reference" section is a placeholder link; the actual API details require digging into the code or other docs.

### 2. **fleet-simulator**
**Score: 7/10**
**Analysis:** Documentation is functional and gets the job done. The README explains the purpose (generating training data via simulated fleet interactions), provides a clear **Usage** example with CLI commands, and defines the structure of the generated **Experience Tiles**. It links to its dependency (`plato-torch`). It could be improved with a visual diagram of the simulation pipeline and more example configurations for different agent swarm behaviors.

### 3. **constraint-theory-core**
**Score: 5/10**
**Analysis:** This is a technical, low-level library (Rust geometric constraint solver). The README has a good title and tagline ("A fast, deterministic 2D geometric constraint solver"). It states the goal and provides a **very basic** code snippet. However, it lacks critical information: a proper **Installation** section (Cargo.toml dependencies?), a detailed **API** overview explaining key structs and functions, and meaningful **Examples** beyond a single line. Users must read the source code to understand how to use it.

### 4. **holodeck-rust**
**Score: 8/10**
**Analysis:** Strong documentation for a complex project (MUD server). The README starts with a compelling **Vision** statement. It clearly differentiates the project ("sentiment-aware NPCs"). The **Getting Started** guide is comprehensive, covering build steps, database setup, and server startup. It includes an **Architecture** section and a **Roadmap**. It's marked down because the "API for Agents" section is brief and would benefit from a concrete example of an agent connecting and interacting with the world.

### 5. **cudaclaw**
**Score: 4/10**
**Analysis:** This appears to be a critical, low-level component (GPU-resident agent runtime), but the documentation is minimal. The README is a single sentence: "A runtime for keeping AI agents resident in GPU memory." There is no information on how to build it, its interface, how it integrates with other fleet components (like `plato-torch`), or what problems it specifically solves. This creates a high barrier to understanding or contribution.

## Step 2: Summary Report & Recommendations

**Overall Fleet Documentation Health: Variable.** The quality correlates strongly with the project's perceived position in the stack. High-level, user-facing tools (`plato-torch`, `holodeck-rust`) are well-documented. Lower-level, foundational libraries (`constraint-theory-core`, `cudaclaw`) are under-documented, creating knowledge silos and friction for new contributors.

**P0 Recommendation (Avoid Rocks):** Do not assume contributors can infer the purpose or use of a repo from its name or code alone. This is a rock.

**P1 Recommendation (Find Safe Channels):** Establish a **minimum viable README standard** for all fleet repositories. A safe channel includes:
1.  **One-Sentence Purpose:** What this is and why it exists.
2.  **Fleet Context:** How it relates to the core thesis (shells/crabs/tiles) or other key repos.
3.  **Install/Build:** How to get it running.
4.  **Basic Usage:** A simple, working code example or command.
5.  **Links:** To related fleet projects and detailed API docs (if they exist elsewhere).

**P2 Recommendation (Optimize Path):**
1.  **Priority Fix:** Author a `FLEET-DOC-STANDARD.md` guide and update the 5 lowest-scoring repos.
2.  **Create `docs/` directories** for `constraint-theory-core` and `cudaclaw` with architectural notes and examples.
3.  **Add a "Fleet Map"** section to key READMEs, visually linking components (e.g., `plato-torch` <- generates tiles for -> `fleet-simulator`).
