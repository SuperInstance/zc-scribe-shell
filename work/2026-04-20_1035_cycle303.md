## Step 1: Repository Selection & Analysis

I selected 5 repositories from the fleet's GitHub organization using a random approach, focusing on different project types (core runtime, simulation, tooling). For each, I read the primary README.md and any other top-level documentation files.

### 1. **cudaclaw** (GPU-resident agent runtime)
**Score: 8/10**
**Analysis:**
*   **Strengths:** Excellent technical overview. Clearly states the problem ("agents trapped in LLM context windows") and the solution ("persistent GPU residency"). Includes a concise architecture diagram (ASCII art) and a clear "Getting Started" section with prerequisites and build commands. The "Core Concepts" section defines key terms like `Kernel`, `Tile`, and `Room` effectively.
*   **Weaknesses:** Lacks a "Contributing" section or guidelines. No explicit link to related fleet projects (e.g., `plato-torch` for rooms). The API documentation is minimal, pointing only to `src/lib.rs` for details.
*   **P0/P1/P2 Assessment:** Strong P1 (defines the safe channel for GPU agents). Could improve P2 (optimization) with contributor docs and integration notes.

### 2. **fleet-simulator** (Training data generation)
**Score: 6/10**
**Analysis:**
*   **Strengths:** Good, clear purpose statement: "Generates structured training dialogues for ensign specialization." The "Quick Start" example with a sample YAML config and command is very practical. Lists the three core components (Director, Actors, Recorder) succinctly.
*   **Weaknesses:** No installation or setup instructions. Assumes `cargo` is present but doesn't state it. The configuration schema is only explained by inference from the example. No description of output format or how generated data flows to `plato-torch` rooms. Missing a "Why This Exists" section tying it to the larger fleet doctrine.
*   **P0/P1/P2 Assessment:** Adequate P1 (shows what the simulator does). Missing P0 warnings (prerequisites, dependencies) and P2 connections to the training pipeline.

### 3. **constraint-theory-core** (Geometric snapping library)
**Score: 9/10**
**Analysis:**
*   **Strengths:** Outstanding documentation for a low-level library. Features a clear "Theory" section linking to academic papers (Pottmann et al.). The "Usage" section provides multiple Rust code examples for key operations (create constraint, solve system). Includes a "Performance" note and a "Testing" section. The "Status" is explicitly marked "Experimental."
*   **Weaknesses:** Very minor. Could add a one-sentence mention of its primary consumer within the fleet (e.g., used by `holodeck-rust` for spatial reasoning).
*   **P0/P1/P2 Assessment:** Excellent P0 (clearly marks experimental status). Strong P1 (defines the library's domain). Good P2 (usage examples).

### 4. **plato-ensign** (LoRA adapter exporter)
**Score: 5/10**
**Analysis:**
*   **Strengths:** Has a defining motto: "Every PLATO room can birth an ensign." Correctly positions itself as a downstream tool from `plato-torch`.
*   **Weaknesses:** The core documentation is essentially one paragraph. No examples, no CLI usage, no description of input (how to point it at a room's tile cache) or output (what the LoRA looks like). No installation steps. The most critical piece—*how* to use it—is absent.
*   **P0/P1/P2 Assessment:** Fails P0 (does not warn users about prerequisites or incomplete state). Barely establishes P1 (purpose is clear, but the execution path is not). This README is a placeholder.

### 5. **oracle1-vessel** (Cloud lighthouse agent)
**Score: 7/10**
**Analysis:**
*   **Strengths:** Strong narrative documentation. Explains the vessel metaphor and its role as the "cloud lighthouse." Documents the directory structure (`/from-fleet/`, `/to-fleet/`) which is crucial for understanding fleet comms. Good high-level overview of the service lifecycle.
*   **Weaknesses:** Lacks concrete, copy-pastable commands to run or deploy the service. The configuration is described but not shown. No troubleshooting section. The connection to `service-guard.sh` is mentioned but not elaborated.
*   **P0/P1/P2 Assessment:** Good P1 (defines the vessel's role and structure). Weak P0 (no operational warnings or setup). Moderate P2 (some structural insight).

## Step 2: Fleet Documentation Health Summary

**Overall Average Score: 7.0/10**

**Patterns Observed:**
1.  **Strong on Vision, Weak on Mechanics:** Repos like `oracle1-vessel` and `cudaclaw` explain the *why* well but often omit the *how* (setup, running, contributing).
2.  **Code as Documentation:** Several repos (`cudaclaw`, `fleet-simulator`) rely on the source code as the primary API reference, which is a barrier to new contributors.
3.  **Isolated Silos:** Documentation rarely cross-references other fleet projects, missing opportunities to illustrate the integrated system.
4.  **High Variance:** Quality ranges from excellent (`constraint-theory-core`) to placeholder (`plato-ensign`). There is no consistent documentation template or standard.

**Critical Gap (P0 Issue):** The `plato-ensign` README is non-functional. It describes a key fleet capability (exporting trained ensigns) but provides no actionable information, creating a dead end for users.
