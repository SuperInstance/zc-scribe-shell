## Step 2: Individual Repository Analysis & Scoring

**Scoring Rubric (0-10):**
*   **0-3:** Missing or severely deficient. No context, no usage instructions.
*   **4-6:** Basic. Has a description and some setup steps, but lacks examples, architecture, or clear purpose.
*   **7-8:** Good. Clear purpose, setup, and basic API/usage. Could improve with examples or deeper context.
*   **9-10:** Excellent. Compelling vision, clear getting started, examples, architecture diagrams, contribution guidelines, and links to related projects.

### 1. `constraint-theory-core`
*   **File:** `README.md`
*   **Content:** The README is concise. It states the project is a "geometric constraint solver" in Rust, part of the Cocapn fleet. It lists key features (n-dimensional, incremental solving) and dependencies. It includes a **basic example** of defining points and a distance constraint, which is valuable.
*   **Strengths:** Clear, technical description. Working code example.
*   **Weaknesses:** No "Getting Started" or installation instructions (`cargo add` is mentioned but not a full guide). No architecture overview. No explanation of how it integrates with the broader fleet (e.g., connection to `plato-torch` rooms).
*   **Score:** **6/10**. It fulfills its basic purpose but is not welcoming to a new contributor or someone unfamiliar with the fleet's context.

### 2. `holodeck-rust`
*   **File:** `README.md`
*   **Content:** This README is more narrative. It opens with the project's role ("MUD server for generating training data") and core thesis ("NPCs with persistent sentiment"). It lists components (World, Agent, Commands) and has a **detailed "Getting Started" section** with `cargo run` instructions and example server output.
*   **Strengths:** Strong narrative hook. Practical setup guide. Clear component breakdown.
*   **Weaknesses:** Lacks a full API example or a tutorial on creating a new NPC/sentiment rule. The connection to PLATO rooms and tile generation is implied but not explicitly detailed.
*   **Score:** **8/10**. A solid, useful README that explains the *why* and *how* for basic operation.

### 3. `cudaclaw`
*   **File:** `README.md`
*   **Content:** Very brief. Declares itself as a "GPU-resident runtime for Cocapn agents." Lists a goal ("sub-millisecond inference") and a technical note about CUDA graphs.
*   **Strengths:** Succinct.

### 4. `flux-runtime`

## Step 3: Summary Report & Fleet Documentation Health

**Overall Fleet Documentation Assessment (Sample):**

*   **Average Score:** **5.4/10**
*   **Trend:** Documentation quality is **bimodal**. Projects closer to the core data pipeline (`plato-ensign`, `holodeck-rust`) have good, practical docs. More experimental or infrastructure-level projects (`cudaclaw`, `flux-runtime`) have minimal, visionary docs that lack implementation details.
*   **Common Strength:** Projects consistently state their high-level purpose within the Cocapn fleet.
*   **Common Weakness:** Lack of beginner-friendly "Getting Started" tutorials and visual architecture diagrams. Integration points between projects are often implied, not documented.

**Recommendations (P1: Safe Channels for Improvement):**

1.  **Template & Standard:** Establish a fleet README template requiring: Vision, Getting Started (install/run), Basic Example, Architecture/Data Flow, and Links to Related Projects.
2.  **Priority Targets:** `cudaclaw` and `flux-runtime` are the highest priority for documentation expansion. They represent critical infrastructure but are currently opaque.
3.  **Integration Maps:** Encourage (or require) a `FLEET-CONTEXT.md` file in each repo that explicitly lists upstream/downstream dependencies (e.g., `holodeck-rust` -> `plato-torch` -> `plato-ensign`).

**Conclusion:** The fleet's documentation is sufficient for internal developers familiar with the overall vision but presents a significant barrier to entry for new contributors or those trying to understand the system as a whole. The foundational ideas are strong, but the "scratches" of practical knowledge are inconsistently applied across the project shells.
