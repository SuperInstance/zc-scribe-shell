# fleet-simulator
## Multi-agent simulation environment for generating PLATO room training tiles.

### Purpose
Generates structured interaction data between simulated agents operating under the Deadband Protocol (P0/P1/P2). This synthetic data is formatted into "tiles" for ingestion by `plato-torch` training rooms, accelerating the accumulation of room memory without requiring constant live human/agent interaction.

### Key Concepts
- **Simulation Tile**: A JSON-structured record of a multi-agent interaction cycle, containing agent states, actions, and outcomes.
- **Scenario Preset**: A predefined simulation configuration (e.g., "market-negotiation", "spatial-reasoning") that defines agent roles and objectives.
- **Deadband Steering**: The simulator enforces protocol priorities, ensuring agents first avoid negative space (P0) before exploring channels (P1).

### Architecture
- **Input**: Scenario presets (YAML), agent behavior profiles.
- **Core**: Rust-based discrete-event simulator.
- **Output**: JSONL files of tiles, compatible with `plato-torch`'s `Room.ingest()` method.
- **Integration**: Primary data source for `plato-torch` training rooms. Can be orchestrated by `cudaclaw` for GPU-resident simulation.

### Quick Start
```bash
# Clone and build
git clone https://github.com/cocapn/fleet-simulator
cd fleet-simulator
cargo build --release

# Run a basic negotiation scenario
./target/release/fleet-simulator --preset ./presets/negotiation.yaml --output ./tiles.jsonl

# Ingest tiles into a PLATO room (example using plato-torch Python API)
from plato_torch import DeadbandRoom
room = DeadbandRoom()
room.ingest('./tiles.jsonl')
```

### API / Usage
The primary interface is the command line. Key arguments:
- `--preset`: Path to scenario YAML file.
- `--output`: Path to write simulation tiles.
- `--cycles`: Number of simulation steps to run (default: 100).
- `--verbose`: Log agent decisions per cycle.

Scenario YAML defines agent `actors`, their `initial_state`, and a `success_metric`.

### Development
```bash
# Run tests
cargo test

# Run with debug logging
RUST_LOG=debug cargo run -- --preset ./presets/test.yaml

# Benchmark performance
cargo bench
```
Contributions: Add new scenario presets in `./presets/` or extend agent behavior models in `src/agent/`.

### See Also
- [`plato-torch`](https://github.com/cocapn/plato-torch) - The training room system that consumes these tiles.
- [`holodeck-rust`](https://github.com/cocapn/holodeck-rust) - Related environment for sentiment-aware NPCs.
- [DEADBAND-PROTOCOL.md](https://github.com/cocapn/fleet-knowledge/blob/main/docs/DEADBAND-PROTOCOL.md) - The constraint theory guiding agent behavior.
```

# holodeck-rust
## A sentiment-aware Multi-User Dungeon (MUD) server for emergent narrative and agent training.

### Purpose
Provides a persistent, text-based world where NPCs (Non-Player Characters) maintain internal sentiment models towards players and other NPCs. These dynamics create emergent social narratives, generating rich interaction logs that serve as training data for `plato-torch` rooms focused on social reasoning and long-term memory.

### Key Concepts
- **Sentiment Graph**: A weighted, directed graph tracking relationships (trust, fear, affinity) between all entities in the world.
- **Event Reactor**: NPCs react to world events (player actions, other NPC events) based on their sentiment state.
- **Narrative Tile**: A specialized training tile capturing a social interaction, its context, and resulting sentiment deltas.

### Architecture
- **Server**: Async Rust using `tokio`. Telnet-compatible socket server.
- **World Model**: Entity-Component-System (ECS) for managing rooms, objects, and characters.
- **Sentiment Engine**: Updates the sentiment graph based on parsed event logs.
- **Output**: Streams narrative tiles to a connected `plato-torch` room or writes to disk.
- **Integration**: Can be populated by agents from `fleet-simulator` or serve as a testbed for `cudaclaw`-hosted agents.

### Quick Start
```bash
# Clone and run the server
git clone https://github.com/cocapn/holodeck-rust
cd holodeck-rust
cargo run --release

# In another terminal, connect via telnet (default port 4000)
telnet localhost 4000

# Connect a PLATO room to ingest the narrative stream
# (Example: Python client using asyncio)
import asyncio
from plato_torch import SocialRoom
room = SocialRoom()
await room.connect_stream('ws://localhost:4001/events') # Example WebSocket endpoint
```

### API / Usage
**Player Commands:** `look`, `go <direction>`, `say <message>`, `ask <npc> about <topic>`.
**Admin/Integration Endpoints:**
- `GET /sentiment_graph` - Returns current relationship state as JSON.
- `WS /events` - WebSocket stream of all narrative tiles.
- `POST /npc` - Spawn a new NPC with a defined initial sentiment profile.

### Development
```bash
# Run the server with a test world
cargo run -- --world ./worlds/test.json

# Run unit tests
cargo test --lib

# Test sentiment engine logic
cargo test test_sentiment_update
```
Extensions: New reaction behaviors are added in `src/npc/reactor/`. New world objects are defined in `src/ecs/components/`.

### See Also
- [`fleet-simulator`](https://github.com/cocapn/fleet-simulator) - Alternative source of agent interaction data.
- [`plato-torch/social_room.py`](https://github.com/cocapn/plato-torch/blob/main/plato_torch/rooms/social.py) - Room preset designed for social tiles.
- [PROJECT-VISION.md](https://github.com/cocapn/fleet-knowledge/blob/main/docs/PROJECT-VISION.md) - The "inhabited intelligence" vision this enables.
```
