## Step 2: Documentation Analysis & Scoring

**Scoring Rubric (0-10):**
*   **0-3:** Missing, incomplete, or severely misleading.
*   **4-6:** Basic structure present but lacks critical details (setup, purpose, API).
*   **7-8:** Good. Clear purpose, setup instructions, and basic usage.
*   **9-10:** Excellent. Comprehensive, includes architecture, examples, contribution guidelines, and connects to fleet context.

### 1. plato-torch

**Analysis:**
*   **Strengths:** The README is comprehensive. It clearly states the project's purpose ("A library for creating self-training rooms"). It provides a high-level architectural diagram (Rooms, Tiles, Ensigns, Ghost Tiles). Installation instructions (`pip install plato-torch`) are present. ## Step 3: Summary Report

**Overall Fleet Documentation Health: Variable (Average Score: 6.0/10)**

The fleet's documentation quality is inconsistent, ranging from excellent (`plato-torch`) to critically deficient (`cudaclaw`). This creates a high barrier to entry for new contributors (greenhorns) and increases the cognitive load for fleet agents trying to integrate systems.

**Patterns Observed:**
1.  **Core vs. Peripheral:** Core libraries (`plato-torch`, `constraint-theory-core`) tend to have better documentation, likely because they are direct dependencies.
2.  **Rust Projects:** Show a bias towards good technical/API documentation but sometimes lack higher-level "why fleet?" context.
3.  **Missing Basics:** Several repos (`holodeck-rust`, `fleet-simulator`, `cudaclaw`) lack the absolute basics: **how to build/run the software.** This is a P0 issue for usability.
4.  **Fleet Context:** Only `plato-torch` successfully integrates its documentation with the overarching Cocapn vision and doctrine. Others operate as isolated technical projects.

**Recommendations (P1/P2):**
*   **P1 (Safe Channel - Minimum Viable Docs):** Establish and enforce a fleet-wide README template requiring: **Purpose, Build/Run Instructions, Basic Usage Example, and Fleet Context**.
*   **P2 (Optimize):**
    *   Prioritize remediation of `cudaclaw` and `fleet-simulator` READMEs.
    *   Enhance `holodeck-rust` with setup and API examples.
    *   Add a "Fleet Integration" subsection to all library READMEs (e.g., "How this feeds tiles to PLATO").
    *   Consider a centralized `fleet-knowledge/docs/` directory for architectural overviews that individual repos can link to.
