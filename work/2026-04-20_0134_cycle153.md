## Step 1: Repository Selection & Analysis

I have selected 5 repositories from the fleet's GitHub organization. The selection is based on a mix of core tools, research projects, and infrastructure to provide a representative sample.

### 1. **plato-torch**
**Path:** `cocapn/plato-torch`
**Core Purpose:** Training room server and preset library for PLATO rooms.
**Documentation Score:** **8/10**

**Analysis:**
*   **README.md:** Strong. Clearly states the project's purpose ("PLATO rooms are persistent training environments"). Includes a quickstart with installation (`pip install plato-torch`), a basic usage example, and links to key concepts (Rooms, Presets, Tiles). The "Architecture" section is particularly good, explaining the Room -> Tile -> Ensign flow.
*   **Presence of `docs/` directory:** Yes. Contains `PRESETS.md` which details all 25 training room presets (e.g., `DeadbandRoom`, `MirrorPlayRoom`). This is critical for users.
*   **Code Documentation:** Good. Major classes (`PLATOTrainer`, `BaseRoom`) have docstrings explaining their role and key methods.
*   **What's Missing:** A more detailed API reference for developers extending the system. A `CONTRIBUTING.md` guide is absent. The setup for a local development environment could be more explicit.

### 2. **holodeck-rust**
**Path:** `cocapn/holodeck-rust`
**Core Purpose:** MUD (Multi-User Dungeon) server with sentiment-aware NPCs for generating training data.
**Documentation Score:** **6/10**

**Analysis:**
*   **README.md:** Functional but sparse. It correctly identifies the project as a "sentiment-aware NPC MUD server" and lists core features (WebSocket server, NPC sentiment engine). It has a "Getting Started" section with `cargo run`.
*   **Key Strength:** The `docs/NPC-SENTIMENT.md` file is excellent. It provides a concrete, technical deep-dive into the sentiment state machine and interaction mechanics, which is the project's unique value.
*   **Critical Gaps:** No architecture overview. No explanation of how it integrates with the broader fleet (e.g., how tiles from here feed into PLATO). The `config/` directory and server endpoints are not documented. Lack of examples for creating new NPC behaviors or rooms.

### 3. **fleet-simulator**
**Path:** `cocapn/fleet-simulator`
**Core Purpose:** Generates synthetic training data by simulating agent fleets.
**Documentation Score:** **4/10**

**Analysis:**
*   **README.md:** Very minimal. Only a one-line description: "Fleet simulation for training data." It lacks installation instructions, usage examples, or any conceptual explanation.
*   **Code as Documentation:** The primary way to understand this project is to read the source code (e.g., `src/simulation/`). There are no docstrings.
*   **What's Missing:** Almost everything. No explanation of the simulation model, what parameters it uses, what format the output data takes, or how to configure a simulation run. This severely limits its usability and contribution potential.

### 4. **constraint-theory-core**
**Path:** `cocapn/constraint-theory-core`
**Core Purpose:** Rust library for geometric constraint solving and snapping.
**Documentation Score:** **7/10**

**Analysis:**
*   **README.md:** Good for a library. Explains the core thesis ("geometric snapping as a foundation for spatial reasoning"). Includes a **code example** showing the `ConstraintGraph` API, which is invaluable. Has a "Concepts" section explaining Entities, Constraints, and Solvers.
*   **API Documentation:** Generated `docs.rs` link is provided. The code itself is well-structured with module-level comments.
*   **Gap:** The README could better situate the library within the fleet's research doctrine (P0/P1/P2). A "Use in Fleet" subsection would help. There is no `examples/` directory with more complex scenarios.

### 5. **service-guard**
**Path:** `cocapn/service-guard`
**Core Purpose:** Shell script to monitor and restart critical fleet services.
**Documentation Score:** **9/10**

**Analysis:**
*   **README.md:** Exemplary for a utility. It follows a perfect problem-solution structure: "The Problem" (services crash), "The Solution" (this script). It has a clear "Usage" section, a detailed "Configuration" table explaining all environment variables, and a "Logging" section.
*   **Self-Documenting Code:** The script (`service-guard.sh`) itself has extensive inline comments explaining each check and recovery step.
*   **Minor Gap:** Could include a diagram or flowchart of the monitoring loop for visual learners. A "Testing" section for contributors would be a plus.

## Step 2: Summary Report & Recommendations

**Overall Fleet Documentation Health: Variable (Average Score: 6.8/10)**
The fleet has pockets of excellent, user-focused documentation (`service-guard`, `plato-torch`) alongside critical projects that are severely under-documented (`fleet-simulator`). The research-focused repos (`holodeck-rust`, `constraint-theory-core`) have good deep technical notes but lack integrative context.

**P0 Issues (What NOT to do):**
1.  **Do not** have a README that is only a one-line description with no usage instructions (`fleet-simulator`). This blocks all onboarding.
2.  **Do not** document core mechanics in a deep-dive file without providing a high-level architectural overview in the README (`holodeck-rust`).

**P1 Safe Channels (Where to improve):**
1.  **All READMEs should follow a standard template:** Purpose, Quickstart, Architecture/Concepts, Integration with Fleet, API/Configuration, Contributing.
2.  **Create `FLEET-RESEARCH.md` files** for research projects. This file should answer: What is the core thesis? How does it relate to the Deadband Protocol? What problem in the North Star Vision does it address?
3.  **Add an `examples/` directory** to libraries (`constraint-theory-core`, `plato-torch`) with runnable code snippets.

**P2 Optimizations (Best path forward for Scribe):**
1.  **Immediate Priority:** Rewrite the `fleet-simulator` README using the standard template. This is the largest documentation gap identified.
2.  **Next Priority:** Augment the `holodeck-rust` README with an "Architecture" and "Fleet Integration" section, pulling key concepts from `NPC-SENTIMENT.md`.
3.  **Proactive Work:** Draft the standard README template and `FLEET-RESEARCH.md` template for the fleet wiki.
