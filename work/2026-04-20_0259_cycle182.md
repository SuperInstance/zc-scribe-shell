## Step 2: Repository Documentation Audit

### 1. **Repository:** `plato-torch`
**Primary Doc:** `README.md`
**Score:** 9/10
**Analysis:**
*   **Strengths:** Excellent. Opens with a clear, compelling thesis ("PLATO rooms are shells. Agents are crabs. Tiles are scratches."). Provides a high-level architectural diagram. Documents core concepts (Rooms, Tiles, Ensigns) thoroughly. Includes a complete "Quick Start" with code examples for creating a room and submitting a tile. Lists all 25 training room presets. Links to related projects (`plato-ensign`, `holodeck-rust`).
*   **Areas for Note:** The "API Reference" section points to auto-generated docs; a brief overview of key classes/methods in the README itself could be helpful for quick scanning. This is a minor point.

### 2. **Repository:** `fleet-simulator`
**Primary Doc:** `README.md`
**Score:** 6/10
**Analysis:**
*   **Strengths:** States its purpose clearly: "Generates structured training data for fleet agents by simulating multi-agent interactions." Explains the core simulation loop (Deadband Protocol P0/P1/P2). Lists the types of agents simulated (Navigator, Sentinel, etc.).
*   **Weaknesses:** Lacks a "Getting Started" or setup section. No code examples of how to run a simulation or interpret output. Does not describe the format of the generated training data. The connection to how this data is used by `plato-torch` rooms is implied but not explicitly detailed.

### 3. **Repository:** `holodeck-rust`
**Primary Doc:** `README.md`
**Score:** 7/10
**Analysis:**
*   **Strengths:** Clear description: "A sentiment-aware MUD (Multi-User Dungeon) server for generating rich narrative training tiles." Documents key features (sentiment engine, procedural quests, NPC memory). Provides a `docker-compose` example for local deployment. Explains the TCP protocol for agents.
*   **Weaknesses:** The setup instructions assume familiarity with Docker and Rust. Could benefit from a simple example of a connecting agent's first interaction. The "Architecture" section is brief; a diagram would be helpful given the complex interplay of systems (MUD server, sentiment engine, narrative generator).

### 4. **Repository:** `constraint-theory-core`
**Primary Doc:** `README.md`
**Score:** 4/10
**Analysis:**
*   **Strengths:** Identifies itself as a "Rust library for geometric constraint solving and snapping." Mentions its use in "CAD-like interfaces for AI."
*   **Weaknesses:** Very sparse. No examples of API usage. No explanation of core concepts (what is a "constraint," what is "snapping"?). No setup instructions (`Cargo.toml` dependency snippet). No link to theory or related research. The purpose is clear, but the documentation does not enable a developer to use it.

### 5. **Repository:** `cudaclaw`
**Primary Doc:** `README.md`
**Score:** 5/10
**Analysis:**
*   **Strengths:** Defines its ambitious goal: "A GPU-resident agent runtime where model weights never leave VRAM." Mentions key components (KV cache manager, scheduler).
*   **Weaknesses:** Highly conceptual. Reads more like a research abstract than usage documentation. No instructions for building, installing, or running. No code examples or API overview. Does not specify compatible hardware or CUDA requirements. The value is in the vision, but the practical path to engagement is missing.
