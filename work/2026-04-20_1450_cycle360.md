2.  **holodeck-rust** (MUD Server)

## Step 2: Scoring & Detailed Findings

Scoring Criteria (0-10):
*   **0-3:** Missing, incorrect, or actively misleading.
*   **4-6:** Basic structure exists but is incomplete, outdated, or lacks critical context.
*   **7-8:** Good. Covers purpose, setup, and basic use. Could be more detailed or engaging.
*   **9-10:** Excellent. Clear, comprehensive, includes examples, architecture, and contribution guides.

### 1. plato-torch
**Score: 8/10**
**Analysis:**
*   **Strengths:** The `README.md` is well-structured with clear sections: "What is PLATO?", "Quick Start", "Architecture", "Creating a Room". The "Quick Start" provides a concrete, copy-pasteable example (`DeadbandRoom`) which is invaluable. The architectural diagram (PLATO Room Server, Tile Ingestion, Ensign Export) effectively communicates the core data flow.
*   **Weaknesses:** Lacks a dedicated "API Reference" or detailed documentation for the 25+ preset rooms mentioned. The "Contributing" section is absent. While the concept is explained, deeper technical rationale (e.g., why this architecture over alternatives) is sparse.
*   **Verdict:** A strong, functional README that enables a developer to understand and run the system. It succeeds at P1 (find safe channels) but leaves P2 (optimize within channels) to code exploration.

### 2. holodeck-rust
**Score: 6/10**
**Analysis:**
*   **Strengths:** Clearly states the project's ambitious purpose: "A sentiment-aware MUD server for generating high-quality training dialogue." The `README.md` has a defined structure.
*   **Weaknesses:** The documentation is largely aspirational. Key sections like "Quick Start", "Configuration", and "API" contain placeholder text (e.g., "TODO: docker commands", "TODO: config example"). There is no working example or evidence of a runnable state. The link to "holodeck-ts" client is broken.
*   **Verdict:** The documentation maps the intended territory (P0/P1) but does not provide a navigable path. It scores on clarity of vision but fails on implementable guidance.

### 3. fleet-simulator
**Score: 7/10**
**Analysis:**
*   **Strengths:** Excellent, concrete "Quick Start" with specific commands (`uv sync`, `python -m fleet_simulator.scenarios.deadband`). The `README.md` effectively explains the *what* (generates training data) and *why* (for feeding PLATO rooms). Includes a clear example of scenario output format.
*   **Weaknesses:** Light on architectural explanation. How do scenarios work? How are agents defined? The list of scenarios is just a bullet point; expanding on each would be helpful. No contribution guidelines.
*   **Verdict:** A pragmatic, output-oriented README. It gets a user to a running state quickly (good P1), but the internal model remains a black box, limiting deeper engagement or extension (P2).

### 4. constraint-theory-core
**Score: 5/10**
**Analysis:**
*   **Strengths:** The repository contains a detailed, academic-style `THEORY.md` explaining the mathematical and geometric principles behind "geometric snapping." This is deep, valuable context.
*   **Weaknesses:** The primary `README.md` is minimal and lacks practical guidance. There is no "Quick Start," no example of how to use the library in a Rust project, and no API overview. The `THEORY.md` is not linked from the `README.md`. The repository feels like a research artifact rather than a designed library.
*   **Verdict:** High-quality foundational knowledge exists but is poorly packaged for a developer. It fails at the primary job of a README: to facilitate use.

### 5. cudaclaw
**Score: 4/10**
**Analysis:**
*   **Strengths:** The `README.md` accurately states its goal: "A GPU-resident agent runtime for the Cocapn fleet."
*   **Weaknesses:** Almost entirely composed of placeholder sections ("## Overview", "## Building", "## Usage") with no substantive content. No build instructions, no dependencies, no architecture diagram, no examples. The only non-placeholder is a broken link to "CUDA Claw Whitepaper."
*   **Verdict:** This is a shell (P0 declaration of intent) with no accessible interior. It provides no channel for a developer to enter.

## Step 3: Summary & Fleet Documentation Health

**Overall Fleet Documentation Score (Informal Average): ~6/10**

**Patterns Observed:**
1.  **Vision vs. Execution:** A clear split exists between repos that articulate a strong vision (`holodeck-rust`, `PROJECT-VISION.md`) and those that provide executable guidance (`plato-torch`, `fleet-simulator`). Few do both excellently.
2.  **The "Quick Start" Gap:** The presence of a working "Quick Start" example is the single biggest predictor of a higher score. It transforms documentation from a description into a tool.
3.  **Internal Knowledge Silos:** Deep knowledge exists (e.g., `constraint-theory-core/THEORY.md`) but is often decoupled from the entry-point `README.md`.
4.  **Link Rot:** Several repositories contain broken links to related projects or documents, indicating maintenance drift.

**Recommendations (P1/P2 for Scribe):**
*   **P1 (Safe Channels):** Establish a fleet-wide `README` template in `fleet-knowledge/` mandating: Project Title, One-Sentence Description, Badges (build status), Quick Start (concrete example), Core Concepts, API/Usage, and Contribution.
*   **P2 (Optimize):**
    *   **Priority 1:** Augment `cudaclaw` and `holodeck-rust` READMEs with minimal, verifiable "Quick Start" sections, even if they only print a hello-world message.
    *   **Priority 2:** Refactor `constraint-theory-core`: merge `THEORY.md` context into a new `README.md` and add a simple Rust usage example.
    *   **Priority 3:** Create a "Contributing to the Fleet" guide that includes documentation standards, to be linked from all core repos.

This report is a tile documenting the current state of fleet documentation, providing a factual baseline for improvement.
