*   **Clarity (0-3):** Is the purpose and scope immediately understandable?

## Step 2: Analysis & Report

**Overall Fleet Documentation Health: Mixed.**
Scores range from 3 (`fleet-simulator`) to 10 (`plato-torch`). The average score for this sample is **6.8/10**.

**Patterns Observed:**
1.  **Strong Technical, Weak Integration:** Repos like `constraint-theory-core` are well-documented as standalone libraries but lack connective tissue to the fleet's mission.
2.  **Vision vs. Execution:** Repos central to the core vision (`plato-torch`) have excellent docs. Supporting or newer infrastructure (`fleet-simulator`, `cudaclaw`) have gaps that hinder adoption.
3.  **Missing "Why":** Several READMEs explain *what* the code does but not *why* it exists in the fleet or what problem it solves for an agent.

**P0 Issue (Highest Priority):**
*   `fleet-simulator` (Score: 3). This appears to be a critical data generation tool, but its documentation does not allow an agent to use it. This blocks potential work/tile generation.

**P1 Issues (Safe Channels to Improve):**
*   Add "Fleet Context" sections to standalone libs (`constraint-theory-core`).
*   Add basic tutorials/example flows to `holodeck-rust` and `cudaclaw`.

**P2 Optimization:**
*   Standardize README structure across all repos using `plato-torch` as a template.
*   Implement a docs health dashboard in the CI pipeline.
