## Step 2: Documentation Analysis & Scoring

**Scoring Rubric (0-10):**
*   **0-3:** Missing, severely incomplete, or actively misleading.
*   **4-6:** Basic structure exists (e.g., a README), but lacks critical details like setup, purpose, or API. May be outdated.
*   **7-8:** Good. Clearly states purpose, has basic setup/usage instructions, and is reasonably up-to-date. May lack advanced examples or architectural context.
*   **9-10:** Excellent. Comprehensive, includes purpose, setup, API examples, architecture diagrams, contribution guidelines, and links to related projects. Serves as a true entry point.

### 1. plato-torch
*   **Assessment:** The README.md is the primary source of truth. It opens with a strong, clear thesis ("PLATO rooms are shells. Agents are crabs. Tiles are scratches.") which aligns perfectly with the PROJECT-VISION. It provides a high-level architectural overview (Rooms, Ensigns, Tiles). It includes a **concrete, copy-paste example** of creating and using a `DeadbandRoom`, which is invaluable. It lists available room presets. It defines key terms (Tile, Room, Ensign).
*   **Missing/Weak Areas:** It lacks a traditional "Installation" or "Quick Start" section separate from the example. There is no `requirements.txt` or `pyproject.toml` snippet for dependency management. Contribution guidelines and a deeper dive into the `plato-ensign` export process are absent.
*   **Score:** 8/10. It is very good at conveying the philosophy and core use case but assumes some context and lacks step-by-step project bootstrapping details.

### 2. fleet-simulator
*   **Assessment:** The README.md exists. It correctly identifies its purpose: "Generates training data for fleet agents by simulating multi-agent interactions." It has a clear **Features** list (e.g., configurable agent personas, event logging). It includes a basic **Usage** example showing how to create a simulator instance and run a cycle. It defines core concepts (Cycle, Event, Agent State).
*   **Missing/Weak Areas:** The setup instructions are vague ("Ensure you have Python and necessary dependencies"). There is no example of a configuration file (`simulator_config.json`). The output format of the training data is not described or shown. The link between simulation cycles and PLATO tiles is implied but not explicitly documented.
*   **Score:** 6/10. It documents the *what* and has a basic API example, but severely lacks the *how* regarding installation, configuration, and data pipeline integration.

### 3. holodeck-rust
*   **Assessment:** The README.md is structured. It starts with a **Description** that connects it to the fleet's purpose ("sentiment-aware NPCs for generating rich narrative training data"). It has a **Tech Stack** section (Rust, Tokio, PostgreSQL), which is excellent. It includes **Build Instructions** (`cargo build`) and **Run Instructions** (`cargo run -- --config-path`). It mentions a configuration file and provides a link to an example config schema.
*   **Missing/Weak Areas:** It does not show a minimal example configuration file inline. There is no example of connecting an agent to the MUD server or a sample interaction. The API for sending/receiving events or dialogue is not documented. The "sentiment-aware" mechanics are not explained.
*   **Score:** 7/10. Solid for getting the service running, but lacks client-side integration examples and deeper behavioral documentation.

### 4. constraint-theory-core
*   **Assessment:** This is a Rust library/crate. The README.md is very brief. It has a clear, one-sentence **Overview**: "A geometric constraint solving library for CAD-like snapping behaviors in agent environments." It has a **Usage** section with a `Cargo.toml` dependency snippet and a minimal code example showing a `Solver` struct and a `solve()` call.
*   **Missing/Weak Areas:** It lacks any explanation of the theory or algorithm (even at a high level). The data structures (`Constraint`, `Geometry`) are not described. There are no examples of defining actual geometric constraints (e.g., coincidence, parallelism). There is no link to a formal API documentation (e.g., `docs.rs`). The connection to fleet agents or specific use cases (e.g., in `forge-master`) is absent.
*   **Score:** 5/10. It meets the absolute minimum for a crate (how to add it, a one-line call). It fails to explain its own domain, making it inaccessible without reading the source code.

### 5. cudaclaw
*   **Assessment:** The README.md is present but reads as a placeholder or early vision doc. It has a compelling **Vision** section describing a "GPU-resident, persistent agent runtime." It lists aspirational **Features** (zero-copy CUDA memory, agent snapshots).
*   **Missing/Weak Areas:** It contains **no code examples, no build instructions, and no API documentation**. It states "This repository is under active development. API is subject to change," but provides no starting point for contributors or users. It is purely descriptive, not instructional.
*   **Score:** 3/10. It documents intent and future goals but provides zero practical guidance for development or integration. It cannot be used.

## Step 3: Summary Report

**Overall Fleet Documentation Health: Mixed.**
Core active projects (`plato-torch`, `holodeck-rust`) have good foundational documentation. Supporting libraries (`constraint-theory-core`) and experimental/visionary projects (`cudaclaw`) are under-documented. The `fleet-simulator` sits in the middle, needing more operational detail.

**Critical Gap:** A consistent pattern for **Installation & Setup** and **Data Flow/Integration** is missing across most repos. Documentation often explains the concept and shows a snippet but doesn't guide a new contributor from `git clone` to a working, integrated component.

**Recommendations (P1/P2):**
*   **P1 (Safe Channel - Establish Baseline):** Create a fleet-wide documentation template for READMEs. Mandate sections: `## Purpose & Vision`, `## Installation`, `## Quick Start (with a full, working example)`, `## Integration with Fleet`, `## API Reference`, `## Development`.
*   **P2 (Optimize - Improve Existing):**
    1.  **plato-torch:** Add a `## Installation` section with pip/Poetry instructions.
    2.  **fleet-simulator:** Add a sample config JSON and describe the training data output format.
    3.  **holodeck-rust:** Add a `## Client Connection Example` (e.g., using `telnet` or a simple Rust client).
    4.  **constraint-theory-core:** Expand the example to show 2-3 concrete geometric constraints being solved.
    5.  **cudaclaw:** Change the README to reflect current, minimal build status, even if it's a "Hello World" kernel example. Move the visionary text to a separate `VISION.md`.

**Next Task for Scribe:** Begin implementing P1 by drafting the fleet README template and then applying it to the lowest-scoring repo (`cudaclaw`) as a pilot.
