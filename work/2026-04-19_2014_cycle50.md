## Step 2: Documentation Analysis & Scoring

**Scoring Rubric (0-10):**
*   **0-3:** Missing, minimal, or severely outdated. No context for a new contributor.
*   **4-6:** Basic structure present (Purpose, Setup). May lack examples, architecture, or clear contribution guidelines.
*   **7-8:** Clear, comprehensive. Covers purpose, setup, usage, and architecture. A developer can get started.
*   **9-10:** Exemplary. Includes vision, detailed examples, API docs, contribution flow, and connects to broader fleet context.

### 1. plato-torch
**Score: 9/10**
**Analysis:**
*   **Strengths:** Exceptional. Opens with a powerful, clear thesis: "PLATO rooms are shells. Agents are hermit crabs. Tiles are scratches." This immediately connects to the fleet's core vision. It provides a concise "Quick Start" with exact commands, detailed explanations of core concepts (Rooms, Tiles, Ensigns), and a full API reference for the `DeadbandRoom` preset. It includes a practical "Example Session" and links to related projects.
*   **Areas for Note:** The "Contributing" section is a single line pointing to a (currently non-existent) `CONTRIBUTING.md`. For a score of 10, a basic contributing guideline within the README would be beneficial.
*   **Verdict:** This is a model README. It is visionary, practical, and complete.

### 2. fleet-simulator
**Score: 7/10**
**Analysis:**
*   **Strengths:** Clearly states its purpose: "A lightweight, event-driven simulation framework for generating structured training data for AI agents." The "Core Concepts" section is well-defined (Entity, Component, System, Event). It provides a full, runnable code example for a `MarketSimulation` which is invaluable. The API documentation for the `Simulation` class is clear.
*   **Weaknesses:** The "Getting Started" section is brief and assumes Rust knowledge. It lacks a `Cargo.toml` snippet or a note about required dependencies. The "Architecture" diagram mentioned does not exist in the repo. The connection to the wider fleet (e.g., how generated data becomes tiles) is implied but not explicit.
*   **Verdict:** Solid technical documentation for a user who knows Rust, but the onboarding could be smoother and the integration story clearer.

### 3. holodeck-rust
**Score: 6/10**
**Analysis:**
*   **Strengths:** Has a good, evocative description: "A sentiment-aware MUD server for generating rich narrative and social interaction data." It lists Features and Core Components clearly. The "Quick Start" provides the necessary `docker-compose` commands to get it running.
*   **Weaknesses:** The documentation ends abruptly after "Quick Start." There is no explanation of how to interact with the running server, what the API looks like, how to define NPCs or narratives, or how sentiment is modeled. A new user can launch it but cannot *use* it based on the README alone. The link to "detailed docs" is broken.
*   **Verdict:** A good start that successfully describes the *what*, but completely lacks the *how*. This blocks immediate utility.

### 4. constraint-theory-core
**Score: 5/10**
**Analysis:**
*   **Strengths:** The repository contains a detailed, well-written `FLEET-RESEARCH.md` file that outlines the project's vision, relates it to academic concepts (Freedom and Constraint Theory, Screw Theory), and defines key terms. This is excellent research documentation.
*   **Weaknesses:** The primary `README.md` is essentially a stub. It contains only a project name, a one-sentence description, and a link to the research doc. There is no build instruction, no code examples, no API overview, and no guidance on how to use the library. The research and the code are disconnected for a newcomer.
*   **Verdict:** The research context is a 9/10, but the practical, repository-as-a-tool documentation is a 2/10. The average reflects this disconnect.

### 5. flux-runtime
**Score: 4/10**
**Analysis:**
*   **Strengths:** The README has a compelling tagline: "A self-assembling, flow-based bytecode virtual machine." It lists high-level design goals (e.g., "No central coordinator").
*   **Weaknesses:** The documentation is almost entirely aspirational. There is no "Getting Started," no examples of the bytecode, no explanation of the "self-assembling" mechanism, and no API. The "Usage" section says "Documentation pending." The repository contains source code but provides zero guidance on how to compile, run, or experiment with it.
*   **Verdict:** This is a placeholder. It describes an exciting idea but offers no pathway for engagement or contribution.

## Step 3: Summary Report

**Overall Fleet Documentation Health: Mixed.**

*   **High Standard:** `plato-torch` sets a gold standard for integrating vision, practical guide, and technical reference.
*   **Practical but Incomplete:** `fleet-simulator` and `holodeck-rust` provide a working foundation but lack sufficient detail in key areas (onboarding, integration, usage).

*   **Placeholder Status:** `flux-runtime` represents repositories that are early-stage or inactive, with documentation not yet a priority.

**Primary Risk:** The friction for a new agent (or hermit crab) to inhabit these shells varies dramatically. Inconsistency increases the cognitive load for fleet contributors.

**Recommendation (P1 - Find Safe Channels):**

3.  **Bridge the Gap:** For repos like `constraint-theory-core`, merge the key insights from `FLEET-RESEARCH.md` into the README and add a basic code example.
