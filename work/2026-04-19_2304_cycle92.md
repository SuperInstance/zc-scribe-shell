## Step 1: Repository Selection & Scoring

I have selected 5 repositories from the fleet's GitHub organization. The selection is based on a mix of core tools, research projects, and utilities to get a representative sample.

### 1. **plato-torch**
**Score:** 8/10
**Assessment:**
*   **Strengths:** The README.md provides a clear, high-level thesis ("PLATO rooms are shells") and immediately links to the core vision document. It has a well-structured "Quick Start" section with installation commands and a basic usage example. The "Architecture" section explains key concepts (Rooms, Tiles, Ensigns) effectively.
*   **Weaknesses:** The API documentation within the README is minimal. It references a `docs/` folder, but a quick check shows this folder contains primarily vision and protocol docs, not detailed API references. There is no "Contributing" section or clear guidelines for adding new room presets.
*   **Recommendation:** Add a dedicated "API Reference" section or link to auto-generated docs. Include a "Development" subsection explaining how to create and test a new room.

### 2. **holodeck-rust**
**Score:** 6/10
**Assessment:**
*   **Strengths:** The README.md has a compelling tagline ("A sentiment-aware MUD server") and good project branding. It lists primary features clearly. The "Getting Started" section provides the necessary `cargo run` command.
*   **Weaknesses:** It lacks crucial context. What is a "sentiment graph"? How does it integrate with the broader fleet? There is no example of game interaction, protocol specification, or architecture diagram. The setup assumes prior knowledge of MUDs and Rust.
*   **Recommendation:** Add a "Concepts" section explaining sentiment graphs and NPC memory. Include a short example transcript of player/NPC interaction. Add a "Fleet Integration" note.

### 3. **fleet-simulator**
**Score:** 4/10
**Assessment:**
*   **Strengths:** The repository name is descriptive. The README.md correctly identifies its purpose: "Generates training data through multi-agent simulation."
*   **Weaknesses:** The documentation is severely lacking. There is no installation instruction, no usage example, and no explanation of how to configure a simulation (agent types, environments, outputs). The link to "example configs" is broken (points to a non-existent `configs/` directory). The structure is unclear.
*   **Recommendation:** This is a high-priority candidate for a rewrite. Must include a minimal working example, description of the configuration schema, and the format of the generated training data.

### 4. **constraint-theory-core**
**Score:** 7/10
**Assessment:**
*   **Strengths:** README.md provides a good conceptual foundation, defining "constraint theory" and its role in geometric snapping. It links to relevant research (`fleet-knowledge/research/`). The "Build" instructions are clear.
*   **Weaknesses:** Lacks practical, *fleet-specific* examples. How is this library used by `plato-torch` or `holodeck-rust`? The API is not demonstrated. The "Status" section is vague.
*   **Recommendation:** Add a "Usage in Fleet" section with a brief code snippet showing how another project imports and uses it. Consider adding a simple visual example of geometric constraint solving.

### 5. **service-guard.sh**
**Score:** 9/10
**Assessment:**
*   **Strengths:** Excellent, functional documentation. The script itself is heavily commented, explaining each service (`plato-room-server`, `ensign-exporter`, etc.), its health check logic, and failure modes. It serves as its own manual. The purpose is immediately clear.
*   **Weaknesses:** As a standalone script, it lacks a traditional README. Its knowledge is embedded rather than presented. A user must read the script to understand it.
*   **Recommendation:** Create a companion `SERVICE-GUARD.md` file that summarizes the services, their ports, and recovery procedures without needing to parse bash logic.
