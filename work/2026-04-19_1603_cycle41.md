## Step 2: Documentation Analysis & Scoring

**Scoring Rubric (0-10):**
*   **0-3:** Missing or severely deficient. No context, no usage.
*   **4-6:** Basic structure present but incomplete, unclear, or outdated.
*   **7-8:** Good. Clear purpose, setup, and basic API. Could use more examples or deeper context.
*   **9-10:** Excellent. Comprehensive, includes vision, architecture, examples, and contribution guidelines.

### 1. plato-torch
**Score: 9/10**
**Analysis:**
*   **README.md:** Exceptional. Opens with a clear, powerful thesis ("PLATO rooms are shells. Agents are crabs. Tiles are scratches."). Provides a high-level architectural diagram, detailed setup instructions (including Docker), and a thorough explanation of core concepts (Rooms, Tiles, Ensigns). The `DEADBAND-PROTOCOL.md` link is integrated.
*   **Strengths:** Vision-driven, technically complete, well-structured. The "Quick Start" and "Architecture" sections are particularly strong.
*   **Areas for Improvement:** A simple, runnable example script showing a minimal agent submitting a tile would bridge the final gap from concept to code.

### 2. holodeck-rust
**Score: 7/10**
**Analysis:**
*   **README.md:** Solid and functional. Clearly states the purpose ("A sentiment-aware MUD server for generating training data"). Provides build instructions (`cargo build`), a basic example of connecting via telnet, and key feature highlights (sentiment engine, Redis persistence).
*   **Strengths:** Purpose is clear, setup is straightforward. The inclusion of a `docs/` folder with an architecture overview is a good practice.
*   **Areas for Improvement:** The README lacks context on *how* this integrates with the broader Cocapn fleet (e.g., how NPC interactions become training tiles). The API for extending the server or defining new commands/rooms is not documented here.

### 3. fleet-simulator
**Score: 5/10**
**Analysis:**
*   **README.md:** Sparse. Contains only a one-line description ("Fleet simulation for generating training data") and basic Python setup instructions (`pip install -e .`).
*   **Strengths:** The description, while brief, is accurate. Setup is simple.
*   **Areas for Improvement:** Critical information is missing: What does a simulation look like? What is the output format? How is this data consumed by `plato-torch`? There are no examples, no architecture, and no link to related concepts. This documentation is a barrier to contribution or use.

### 4. constraint-theory-core
**Score: 6/10**
**Analysis:**
*   **README.md:** Technical and focused. Explains the core problem ("geometric constraint solving") and the library's approach ("Rust, real-time"). Includes build instructions, a link to formal papers, and a list of core types/systems.
*   **Strengths:** Good for developers who already understand the domain. The link to academic foundations adds credibility.
*   **Areas for Improvement:** Lacks a **concrete, practical example**. A user cannot quickly understand how to *use* this library to solve a simple constraint. The connection to AI/agent training (e.g., how this enables "geometric snapping" for agents) is implied but not explained.

### 5. cudaclaw
**Score: 4/10**
**Analysis:**
*   **README.md:** Very minimal. Title ("GPU-resident agent runtime") is descriptive. Contains only a "Building" section with CMake instructions and a note about CUDA version compatibility.
*   **Strengths:** The title effectively communicates the project's ambitious scope. Build instructions are present.
*   **Areas for Improvement:** There is no explanation of *what* a GPU-resident agent runtime is, why it's needed, or its architecture. No API, no examples, no integration points with `plato-torch` or other fleet components. The documentation does not fulfill its basic purpose of informing a potential user or contributor.

## Step 3: Summary Report

**Overall Fleet Documentation Health: Mixed.**

*   **High Standard (`plato-torch`):** One repository sets an excellent standard, integrating vision, architecture, and practical guidance.
*   **Functional but Isolated (`holodeck-rust`, `constraint-theory-core`):** These repos explain themselves but fail to articulate their role in the larger fleet ecosystem. They are documented as standalone tools.
*   **Deficient (`fleet-simulator`, `cudaclaw`):** These repos have documentation that is insufficient for onboarding, contribution, or even clear understanding of their function. They represent single points of failure for knowledge transfer.

**Primary Gap: Fleet Context.** With the exception of `plato-torch`, documentation rarely connects the component to the overarching Cocapn thesis of inhabited intelligence and the tile/room/ensign pipeline. This makes the fleet feel like a collection of separate projects rather than a coherent system.

1.  **Mandate a README structure** for all repos: Purpose, Fleet Role, Architecture/Design, Setup, Usage Example, API Reference (if applicable).

3.  **Prioritize remediation** for the lowest-scoring repos (`cudaclaw`, `fleet-simulator`). A basic working example is the most urgent need.
