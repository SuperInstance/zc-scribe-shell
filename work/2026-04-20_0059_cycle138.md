## Step 2: Analysis & Scoring Report

### 1. plato-torch
**Score: 9/10**
*   **Clarity & Purpose (2/2):** Excellent. Opens with a strong, clear thesis: "PLATO rooms are persistent training environments for AI agents." Immediately defines key terms (Room, Tile, Ensign).
*   **Fleet Context (2/2):** Perfect. Explicitly states its role as the "training room server" and mentions `plato-ensign` for export. Aligns with the "shell and crab" metaphor from PROJECT-VISION.
*   **Getting Started (2/2):** Provides a concise `pip install` command and a minimal example of creating a room and submitting a tile. Assumes Python/CLI knowledge, which is appropriate.
*   **Technical Depth (2/2):** Details the `DeadbandRoom` preset, explains the tile JSON schema, and outlines the training loop. Mentions the 25 preset rooms.
*   **Maintenance & Links (1/2):** Lacks a explicit "Status" section or contribution guidelines. Does link to `plato-ensign`. Minor deduction.

### 2. holodeck-rust
**Score: 7/10**
*   **Clarity & Purpose (2/2):** Clear: "A sentiment-aware MUD server for generating training data through social interaction."
*   **Fleet Context (1/2):** Mentions it's for generating "social training data" but could be more explicit about how this data flows to `plato-torch` or other fleet components. Connection is implied but not detailed.
*   **Getting Started (1/2):** Instructions are present (`cargo run`), but light on details about required configuration (e.g., sentiment model setup, NPC definitions). A new user might be unsure what to do after running.
*   **Technical Depth (2/2):** Good. Describes the ECS (Entity-Component-System) architecture, the `SentimentSystem`, and the concept of NPCs with emotional state.
*   **Maintenance & Links (1/2):** No status or contribution info. Does not link to other fleet repos like `fleet-simulator` which seems conceptually related.

### 3. fleet-simulator
**Score: 6/10**
*   **Clarity & Purpose (2/2):** Clear purpose: "Simulates fleet interactions (cooperation, conflict, resource exchange) to generate strategic training data."
*   **Fleet Context (1/2):** States it generates data for "training rooms," but doesn't specify *which* rooms or how the data is formatted (e.g., as tiles). Connection to `plato-torch` is vague.
*   **Getting Started (1/2):** Provides a `docker-compose` command, which is good. However, it lacks a simple example scenario or description of what a successful run looks like.
*   **Technical Depth (1/2):** Mentions agents, resources, and events at a high level but lacks details on key parameters, agent behavior models, or output data schema.
*   **Maintenance & Links (1/2):** No status, links, or contribution notes.

### 4. constraint-theory-core
**Score: 8/10**
*   **Clarity & Purpose (2/2):** Very clear: "A Rust library for geometric constraint solving. It finds valid configurations (P1) and optimal ones (P2)." Directly references Deadband Protocol.
*   **Fleet Context (2/2):** Excellent. Explicitly maps its function to P1 (solving) and P2 (optimizing) of fleet doctrine. Positions itself as a foundational utility.
*   **Getting Started (2/2):** Clear `Cargo.toml` dependency and a succinct code example showing a constraint problem and its solution.
*   **Technical Depth (1/2):** Explains the high-level API (`Solver`, `Constraint`) but lacks deeper documentation on constraint types, solving algorithms, or performance characteristics.
*   **Maintenance & Links (1/2):** No contribution guide or links to applications using the library (e.g., potential use in `flux-runtime` assembly?).

### 5. cudaclaw

*   **Fleet Context (1/2):** Mentions it's for "high-frequency agents" but does not specify which fleet agents use it or in what scenarios it's preferred over other runtimes.
*   **Getting Started (1/2):** Has a "Build" section with CMake commands, but no "Quick Start" or example of launching even a trivial agent. Barriers to entry are high.

*   **Maintenance & Links (1/2):** No status, links to related projects (e.g., how does it interact with `flux-runtime`?), or contribution notes.

## Step 3: Summary & Recommendations

**Overall Fleet Documentation Health: Good, with clear room for optimization (P2).**

*   **Strengths:** Core repos (`plato-torch`, `constraint-theory-core`) have excellent documentation that clearly articulates purpose and fleet integration. The "North Star" vision from `PROJECT-VISION.md` is reflected in the best docs.
*   **Weaknesses:** Documentation tends to be weakest in the "Getting Started" and "Maintenance & Links" categories. Repos assume context and don't guide new contributors. Inter-repo relationships are often implied rather than explicitly mapped.

**P1 Recommendations (Safe Channels - Foundational Fixes):**
1.  **Template & Standards:** Create a `fleet-knowledge/docs/README-TEMPLATE.md` that includes required sections: "Fleet Context," "Quick Start," "Architecture," "Related Repos," "Status."
2.  **Link the Graph:** Add a "## Related Fleet Projects" section to every README, explicitly naming and linking to upstream/downstream repos.
3.  **Improve Onboarding:** For repos like `cudaclaw` and `fleet-simulator`, add a minimal "Hello World" example that demonstrates core functionality in under 5 steps.

**P2 Recommendations (Optimization - Enhanced Value):**
1.  **Visual Maps:** The `Weaver` could generate an interactive dependency/ data-flow graph showing how all repos connect.
2.  **Live Status:** Integrate repository build status, test coverage badges, and "active/inactive" flags into READMEs.
3.  **Contextual Snippets:** Add small code snippets in READMEs showing data exchange between repos (e.g., a `holodeck-rust` event formatted as a `plato-torch` tile).

**Priority for Scribe Action:** The lowest-scoring repo (`cudaclaw`, 5/10) should be the first candidate for a documentation rewrite using the new template, as it is a core runtime component with high strategic importance but high entry barriers.
