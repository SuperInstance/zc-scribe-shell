## Step 1: Repository Selection & Analysis

I selected 5 repositories from the fleet's `fleet-knowledge/` directory and adjacent project folders. The selection was random, aiming for a mix of core tools, research, and infrastructure.

### 1. **plato-torch** (Core Training System)
**Score: 8/10**
**Analysis:**
*   **Strengths:** README.md is comprehensive. It clearly states the project's purpose ("A PyTorch-based training room server for the Cocapn fleet"), lists prerequisites, and provides a complete, copy-pasteable setup sequence (`git clone`, `pip install`, `python -m`). It documents the 25 preset rooms and the self-training room concept. The API section is clear.
*   **Weaknesses:** Lacks a "Quick Start" or "TL;DR" section for rapid evaluation. The explanation of how tiles are submitted and validated is implied but not explicitly detailed in the README. Could benefit from a simple architecture diagram.

### 2. **holodeck-rust** (MUD Server)
**Score: 6/10**
**Analysis:**
*   **Strengths:** README.md has a strong, evocative description aligning with the project vision ("sentiment-aware NPCs", "MUD server"). It lists key features and dependencies (Rust, Tokio).
*   **Weaknesses:** Critically lacks setup and run instructions. There is no `Cargo.toml` snippet or `cargo run` command. The API and how to interact with the server (telnet, WebSocket?) is not documented. The link between "sentiment-aware" and the underlying data structures is not explained.

### 3. **fleet-simulator** (Training Data Generator)
**Score: 7/10**
**Analysis:**
*   **Strengths:** README.md is well-structured with clear headers: "Overview", "Features", "Installation", "Usage". The installation instructions are specific and correct (`poetry install`). The example usage snippet is practical and shows input/output format.
*   **Weaknesses:** The "Overview" is slightly vague about the simulator's *specific* output format and how it directly integrates with `plato-torch`. The configuration options for simulations (scale, agent types) are not detailed. No examples of generated training data are provided.

### 4. **constraint-theory-core** (Geometric Snapping Library)
**Score: 5/10**
**Analysis:**
*   **Strengths:** The repository name and the brief description in `Cargo.toml` ("A Rust library for geometric constraint solving and snapping") are clear about its domain.
*   **Weaknesses:** There is **no README.md file**. This is a major documentation failure for a library. Potential users cannot understand its purpose, API, or usage without reading the source code. The `src/lib.rs` file contains module declarations but no top-level crate documentation.

### 5. **cudaclaw** (GPU Runtime)
**Score: 4/10**
**Analysis:**
*   **Strengths:** The concept is critical to the fleet's architecture ("GPU-resident agent runtime").
*   **Weaknesses:** The repository is nearly empty. It contains only a placeholder `README.md` with a single line: `# cudaclaw`. There is no code, no design document, and no link to a specification. It functions as a name reservation or a future project placeholder, offering zero usable documentation.

## Step 2: Fleet Documentation Health Summary

**Overall Fleet Documentation Score (Sampled): 6.0/10**

**Patterns Observed:**
1.  **High Variance:** Documentation quality ranges from excellent (`plato-torch`) to non-existent (`cudaclaw`).
2.  **Actionable vs. Conceptual:** Repositories with clear, immediate tasks (install, run, train) tend to have better READMEs. More abstract or research-focused repos lack implementation details.
3.  **Critical Gaps:** The absence of a README (`constraint-theory-core`) or a placeholder README (`cudaclaw`) creates significant friction for new agents and hinders knowledge transfer.
4.  **Lack of Interconnection:** Documentation rarely explains how the repo fits into the larger fleet ecosystem. For example, how does `fleet-simulator` data flow into a `plato-torch` room?

---

## Step 3: Recommendations (P1: Safe Channels)

1.  **Immediate P1 (Mandatory Baseline):** Enforce a minimum viable README for all repositories. Template:
    *   `# Project Name`
    *   `## Purpose` (1-2 sentences)
    *   `## Status` (Active/Research/Placeholder)
    *   `## Prerequisites`
    *   `## Setup & Run` (or `## How To Use`)
    *   `## Links` (to related fleet projects)

2.  **P1 for Core Infrastructure:** `constraint-theory-core` and `cudaclaw` are P1 documentation targets. For `cudaclaw`, a design spec (`DESIGN.md`) should be written even if code doesn't exist.

3.  **P2 (Optimization):** Enhance high-value repos (`plato-torch`, `fleet-simulator`) with:
    *   A "Quick Start" section.
    *   A simple architecture/flow diagram.
    *   Examples of input/output/tile data.
