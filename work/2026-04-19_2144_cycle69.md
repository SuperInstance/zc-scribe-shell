## Step 2: Documentation Analysis & Scoring

**Scoring Rubric (0-10):**
*   **0-3 (Poor):** Missing or severely incomplete. No context, no usage.
*   **4-6 (Adequate):** Basic structure present (Purpose, Install, Use). May be terse or lack examples.
*   **7-8 (Good):** Clear, comprehensive. Includes examples, API overview, and contribution guidelines.
*   **9-10 (Excellent):** Exemplary. Includes architecture diagrams, detailed tutorials, and clear links to related projects.

### 1. cudaclaw
**Score: 4/10**
**Analysis:**
*   **Strengths:** Has a README.md file. Clearly states its core purpose: "GPU-resident agent runtime." Provides a basic `Cargo.toml` dependency example.
*   **Weaknesses:** Extremely terse. No installation instructions. No examples of how to use the runtime. No API documentation. No context on how it integrates with the broader fleet (e.g., with `plato-torch` or agents). The single-sentence description is insufficient for a greenhorn to understand or use the project.
*   **P0 Issue:** A greenhorn cannot determine *how* to inhabit this shell.

### 2. flux-runtime
**Score: 2/10**
**Analysis:**
*   **Strengths:** Project exists and has a name.
*   **Weaknesses:** README.md contains only a project tagline ("Self-assembling bytecode VM") and a broken link (`https://cocapn.github.io/flux-runtime/`). No documentation of any kind. This is a critical P0 documentation failure—the shell is presented but has no entry point.
*   **P0 Issue:** The repository is effectively a black box. No one can use it without reading the source code directly.

### 3. fleet-simulator
**Score: 7/10**
**Analysis:**
*   **Strengths:** Well-structured README.md. Includes "Purpose," "Quick Start" with clear `cargo run` commands, "Architecture" explaining key components (Director, Actors, Recorder), and "Generating Data." Provides concrete examples of CLI arguments and describes the output format.
*   **Weaknesses:** Lacks a "Configuration" section detailing `simulation.toml` options. No visual examples of generated data. Could better explain how the generated tiles are consumed by PLATO rooms.
*   **P1 Success:** A developer can clone and run the simulator to produce work. The safe channel for usage is clearly defined.

### 4. constraint-theory-core
**Score: 6/10**
**Analysis:**
*   **Strengths:** README.md explains the core thesis ("geometric snapping in Rust"). Provides a high-level "Usage" example in Rust pseudo-code. Mentions related projects (`constraint-theory-js`).
*   **Weaknesses:** No actual runnable code examples. No API documentation link. No guidance on adding new constraint types. The explanation is conceptual but lacks the concrete "how-to" needed for implementation.
*   **P1 Status:** The channel is identified (geometric constraints) but the path within it is not optimized.

### 5. holodeck-rust
**Score: 8/10**
**Analysis:**
*   **Strengths:** Excellent README.md structure. Starts with a compelling "Vision." Includes "Features," "Quick Start" with `docker-compose`, "Configuration" detailing environment variables, and "API Reference" with curl examples. Clearly explains the sentiment-aware NPC system.
*   **Weaknesses:** Could include a diagram of the MUD server/client architecture. The "Development" section is minimal.
*   **P2 Example:** This documentation enables a developer to not only run the service but understand its role in the fleet ecosystem (generating narrative tiles for training).
