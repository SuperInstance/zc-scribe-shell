## Step 1: Repository Selection & Analysis

I selected 5 repositories from the fleet's `fleet-knowledge/` directory and adjacent projects. I read their primary documentation files (README.md, any CONTRIBUTING.md, etc.) and scored them based on a rubric derived from fleet standards: clarity of purpose, setup instructions, usage examples, API/interface definition, and contribution guidelines.

### 1. **plato-torch**
**Score: 8/10**
**Primary File:** `README.md`
**Analysis:**
*   **Strengths:** Excellent high-level vision ("PLATO rooms are shells. Agents are crabs. Tiles are scratches."). Clear, step-by-step installation and environment setup for both development and Docker. Well-defined core concepts (Rooms, Tiles, Ensigns). Includes a basic "Quick Start" example.
*   **Weaknesses:** The API documentation for the `DeadbandRoom` and other presets is sparse. While the concepts are explained, a developer looking to create a new room preset would need to read the source code. Missing a "Contributing" section or guidelines for submitting new room types.

### 2. **holodeck-rust**
**Score: 6/10**
**Primary File:** `README.md`
**Analysis:**
*   **Strengths:** Strong, evocative project description ("MUD server with sentiment-aware NPCs"). Clearly states the technology stack (Rust, Tokio, PostgreSQL). Has a "Getting Started" section with `cargo run` instructions.
*   **Weaknesses:** Lacks detail on what a "sentiment-aware NPC" is or how to configure one. No example of game state, commands, or interaction. No API documentation for extending the server. The setup assumes a local Postgres instance without configuration examples or a `docker-compose.yml`.

### 3. **fleet-simulator**
**Score: 4/10**
**Primary File:** `README.md`
**Analysis:**
*   **Strengths:** States its purpose clearly: "Fleet simulation for training data." Mentions it integrates with `plato-torch`.
*   **Weaknesses:** Severely lacking in practical details. No installation instructions. No explanation of how to configure a simulation, what parameters are available, or what the output data format is. The "Usage" section is a single, unexplained code snippet. A new contributor cannot run or use this project based on the README alone.

### 4. **constraint-theory-core**
**Score: 7/10**
**Primary File:** `README.md`
**Analysis:**
*   **Strengths:** Good technical description ("Rust geometric snapping"). Clear mathematical framing (P0/P1/P2 constraints). Well-documented public API in the README itself, listing key structs (`Constraint`, `SolutionSpace`) and their methods. Includes a non-trivial code example demonstrating the API's use.
*   **Weaknesses:** No "Getting Started" or installation instructions (e.g., `cargo add constraint-theory-core`). The example is helpful but assumes prior knowledge of how to integrate it into a larger project. Missing a "Theory" or "Background" section to deeper explain the geometric principles.

### 5. **cudaclaw**
**Score: 5/10**
**Primary File:** `README.md`
**Analysis:**
*   **Strengths:** Clear, bold statement of purpose ("GPU-resident agent runtime"). Identifies the problem it solves (CPU-GPU latency). Lists key features (kernel fusion, zero-copy messaging).
*   **Weaknesses:** Almost entirely conceptual. No instructions for building, installing, or requiring CUDA. No code snippets or examples of how to write a "GPU-resident agent." No API reference or link to generated docs. The README describes a vision but provides no on-ramp for a developer.

## Step 2: Summary & Recommendations

**Overall Fleet Documentation Health: Moderate (Average Score: 6.0/10).**

*   **Pattern Observed:** A strong focus on **vision and core doctrine** (P0/P1), often at the expense of **practical implementation details** (P2). This aligns with fleet philosophy but hinders new contributor onboarding.
*   **Critical Gap:** The lower-level systems (`fleet-simulator`, `cudaclaw`) have the weakest docs, creating a "black box" effect around core infrastructure.

**P1 Recommendations (Safe Channels - What to Document):**
1.  **Standardize README Headers:** All repos should have: `## Purpose`, `## Getting Started`, `## Usage/Examples`, `## API Reference` (or link to docs), `## Contributing`.
2.  **Mandate a minimal working example** (`examples/` directory or a snippet in README) that can be run after following "Getting Started."
3.  **Require a `docker-compose.yml`** for services with dependencies (e.g., `holodeck-rust` with Postgres).

**P2 Recommendations (Optimization - Improve Quality):**
1.  **Add inline docstrings** to key public functions/structs in Rust/Python code, enabling automatic doc generation.
2.  **Create a `docs/` folder** in each repo for architecture decisions, protocol details, and deeper theory.
3.  **Implement a documentation linter** in CI that checks for the presence of required README sections.

**Immediate Action for Scribe (My Task):**
I will prioritize improving the **`fleet-simulator` README** (Score: 4/10) as it is a critical data generation component. The rewrite will add a complete "Getting Started" guide, explain configuration via a `config.toml` example, and show how to run a simulation and interpret its output tiles.
