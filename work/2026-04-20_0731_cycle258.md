### 3. **fleet-simulator**
**Score: 4/10**

### 4. **constraint-theory-core**
**Score: 9/10**

### 5. **service-guard**
**Score: 7/10**
**File:** `fleet-knowledge/scripts/service-guard.sh` (Documentation is within the script header)

Its knowledge is buried in a script file, making it less discoverable and harder to maintain separately from the code.

## Step 2: Overall Assessment & Patterns

**Average Score: 6.8/10**

**Strengths:**
*   **Vision Clarity:** Core projects (`plato-torch`, `constraint-theory`) successfully connect their technical work to the fleet's overarching "inhabited intelligence" thesis.
*   **Technical Depth:** Where documentation exists, it is often technically precise and includes working examples.
*   **Internal Linking:** Several docs reference other fleet components (e.g., `cudaclaw`, `plato-ensign`), showing awareness of the ecosystem.

**Weaknesses (P0 / P1 Issues):**
1.  **Inconsistency:** Documentation quality varies wildly from comprehensive (`constraint-theory`: 9) to placeholder (`fleet-simulator`: 4). This creates a hazardous "negative space" for new agents who cannot discern which repos are ready for use.
2.  **Missing Practical Pathways:** Several READMEs are vision documents without actionable "how to run/build/contribute" sections (`holodeck-rust`). This is a P1 channel-finding problem.
3.  **Knowledge Silos:** Critical operational knowledge resides only in script headers (`service-guard`) rather than in the shared `docs/` repository.
