## Step 1: Repository Selection & Analysis

I have selected 5 repositories from the fleet's GitHub organization. The selection is based on a mix of core tools, research projects, and utilities to get a representative sample.

### 1. **plato-torch**
**Path:** `cocapn/plato-torch`
**Purpose:** Core training room server and presets for PLATO system.
**Documentation Score:** **8/10**

**Analysis:**
- **README.md:** Comprehensive. Includes quick start, architecture overview (Rooms, Tiles, Ensigns), API reference, and contribution guidelines. Links to related projects (plato-ensign, holodeck-rust).
- **Internal Docs:** Well-structured `docs/` directory with `DEADBAND-PROTOCOL.md`, `ROOM-API.md`, and `TILE-SPEC.md`.
- **Code Comments:** Moderate. Key classes and methods have docstrings.
- **Missing:** A clear, visual diagram of the data flow (Tile submission → Room → Ensign export). The "Getting Started" could use a simpler, copy-paste example for a first-time user.

### 2. **fleet-simulator**
**Path:** `cocapn/fleet-simulator`
**Purpose:** Generates training data by simulating fleet agent interactions.
**Documentation Score:** **6/10**

**Analysis:**
- **README.md:** Explains the concept (generating "tile-like" interaction data) and has a basic usage example. Lacks a detailed explanation of the simulation parameters or the structure of the generated data.
- **Internal Docs:** Minimal. No `docs/` folder. Key configuration (like agent behavior profiles) is not documented.
- **Code Comments:** Sparse. The simulation engine logic is complex but largely uncommented.
- **Missing:** A specification for the output data format. A guide on how to integrate its output with `plato-torch`. Examples of different simulation scenarios.

### 3. **constraint-theory-core**
**Path:** `cocapn/constraint-theory-core`
**Purpose:** Rust library for geometric constraint solving and snapping.
**Documentation Score:** **4/10**

**Analysis:**
- **README.md:** Very brief. Only a one-sentence description and a link to the "Fleet Research" document. No usage examples, API overview, or build instructions.
- **Internal Docs:** None.
- **Code Comments:** Some high-level module comments, but function-level documentation is absent. The crate exports are not explained.
- **Missing:** Everything. This is a critical library but is effectively undocumented for a new contributor. Needs a basic tutorial, examples of solving a simple constraint system, and documentation of the core types (`Constraint`, `SnapEngine`).

### 4. **service-guard**
**Path:** `cocapn/service-guard`
**Purpose:** Bash script to monitor and restart core fleet services.
**Documentation Score:** **7/10**

**Analysis:**
- **README.md:** Practical and clear. Lists the 5 services it guards, provides installation and usage commands, and explains the health check logic. Includes a "Troubleshooting" section.
- **Internal Docs:** Not applicable for a single script.
- **Code Comments:** The bash script is well-commented, explaining each major function and check.
- **Missing:** Could include a diagram of the service dependency tree. Instructions for adding a new service to the guard.

### 5. **holodeck-rust**
**Path:** `cocapn/holodeck-rust`
**Purpose:** MUD server with sentiment-aware NPCs for generating narrative tiles.
**Documentation Score:** **5/10**

**Analysis:**
- **README.md:** Good conceptual introduction explaining the "sentiment-aware NPC" system and its role in the fleet. However, the "Usage" section is lacking.
- **Internal Docs:** A `docs/` folder exists but contains only a legacy design doc. No current API or protocol specification.
- **Code Comments:** Limited. The NPC brain and event system are complex with little explanatory commentary.
- **Missing:** A clear guide on how to run the server, connect a client, and start an interaction. Documentation of the TCP protocol or API for sending/receiving events. Examples of the narrative tiles it produces.

## Step 2: Summary Report & Recommendations

**Overall Fleet Documentation Health: Moderate, with high variance.**
Core infrastructure (`plato-torch`, `service-guard`) is well-documented. Research and library projects (`constraint-theory-core`, `fleet-simulator`, `holodeck-rust`) suffer from significant documentation debt, which creates a barrier to contribution and understanding.

**P0 Issue (Blocking):** `constraint-theory-core` is a foundational library with near-zero documentation. This is a rock. Contributors cannot use or modify it effectively.

**P1 Safe Channels (Immediate Improvements):**
1.  **`constraint-theory-core`:** Write a minimal `README.md` with a code example. Add Rust doc comments to public functions.
2.  **`fleet-simulator`:** Document the JSON output schema. Add a `docs/SCENARIOS.md` file with 2-3 example simulations.
3.  **`holodeck-rust`:** Add a "Quick Start" section to the README with `cargo run` and telnet instructions.

**P2 Optimization (Future):**
*   Create a standard `docs/` template for all repos (Problem, Solution, API, Examples).
*   Implement a documentation freshness check in CI.
*   Use `cargo doc` for Rust projects and host the output.

**Next Action for Scribe:** I will prioritize the **P1** action for `constraint-theory-core` as it represents the most critical gap. I will draft an improved README and submit it as a training tile.
