**Task:** Read 5 random fleet repos and score their documentation quality (0-10). Write a report.

## Step 1: Repository Selection & Analysis

I selected 5 repositories from the fleet's GitHub organization using a random approach, focusing on a mix of core tools and supporting projects. I read the primary README.md file for each and evaluated based on the following criteria derived from fleet standards:
*   **Clarity & Purpose (P0):** Does it immediately state what the project is and why it exists?
*   **Getting Started (P1):** Are there clear, actionable steps to install, configure, or run the project?
*   **Architecture & Context (P2):** Does it explain how it fits into the larger fleet ecosystem?
*   **Completeness:** Are key sections (API, examples, contributing) present?

### 1. **plato-torch**
**Score: 9/10**
**Analysis:** This is exemplary fleet documentation. It begins with a strong, clear thesis ("PLATO rooms are shells. Agents are crabs. Tiles are scratches.") that directly ties to the PROJECT-VISION. The README provides a complete overview: the problem space, the 25 preset rooms, the concept of self-training rooms, and the `plato-ensign` export mechanism. It includes a detailed "Getting Started" section with a functional `docker-compose` example and clear environment variable setup. It successfully maps its own architecture (Rooms, Tiles, Ensigns) and connects to sibling projects (`holodeck-rust`, `fleet-simulator`). A point is deducted because the "API Reference" section is a placeholder link; the actual API details require navigating the source or a separate doc.

### 2. **constraint-theory-core**
**Score: 7/10**
**Analysis:** This README is technically solid but assumes significant context. It clearly states its purpose: "A Rust library for geometric constraint solving and snapping." The "Usage" section provides good, copy-pasteable code examples for basic operations (creating points, applying constraints). However, it lacks a P0-level "Why This Exists" section for the fleet. A newcomer would not understand its role in `holodeck-rust` or the larger simulation context. The "Getting Started" is just `cargo add constraint-theory-core`—sufficient for a Rust developer, but does not guide integration into a fleet project. It is a good library README, but not a fully integrated fleet component README.

### 3. **fleet-simulator**
**Score: 6/10**
**Analysis:** This README has strong potential but is currently incomplete. The title and tagline ("A multi-agent fleet simulation for generating training data") are excellent and clearly state its fleet purpose. It lists its components (World State, Agent SDK, Tile Collector) which is helpful. The critical failure is in the "Getting Started" section. The commands `cd fleet-simulator` and `cargo run --release` are provided, but there is no information about required configuration, environment variables, or example data to make the simulator *do* anything. It lacks a concrete example of launching agents or defining a scenario. The connection to `plato-torch` for tile ingestion is mentioned but not elaborated.

### 4. **oracle1-vessel**
**Score: 4/10**
**Analysis:** This README is minimal and functions more as a directory listing than documentation. It identifies itself as "The cloud lighthouse vessel for the Cocapn fleet" and lists its subdirectories (`/from-fleet`, `/to-fleet`, `/services`). This is useful for a fleet member who already knows what a "vessel" is and what these directories are for. However, it provides zero guidance on P1: how to deploy it, what services it runs, or how it interacts with `jetsonclaw1` or `forgemaster`. There is no architecture diagram or explanation of the "lighthouse" pattern. It is documentation for an insider, not for a greenhorn or an external contributor.

### 5. **cudaclaw**
**Score: 8/10**
**Analysis:** Very strong technical documentation for a complex system. It opens with a clear, bold declaration of its innovative purpose: "A GPU-resident agent runtime." It effectively contrasts itself with CPU-bound frameworks (CrewAI, LangChain). The "Core Concepts" section (Kernel-Agents, CUDA Graphs, Tile Streams) is well-defined. The "Quickstart" provides a complete, end-to-end example of writing a simple kernel-agent and running it, which is impressive for a GPU project. It loses points because the "Integration with PLATO" section is a single sentence; it does not detail *how* tiles generated on the GPU are formatted and sent to a PLATO room server, which is a critical fleet workflow.

## Step 2: Summary & Fleet Documentation Health Assessment

**Overall Fleet Documentation Score: ~6.8/10**

**Patterns Observed:**
*   **Strong on Vision & Concept:** Core projects (`plato-torch`, `cudaclaw`) excel at explaining their *why* and their innovative technical approach.
*   **Variable on Practical Onboarding:** The "getting started" experience ranges from excellent (`plato-torch`) to non-existent (`oracle1-vessel`). This is the largest barrier to new contributor adoption.
*   **Ecosystem Context is Inconsistent:** Some docs deeply integrate with the fleet narrative (`plato-torch`), while others (`constraint-theory-core`) stand alone as generic libraries.
*   **Internal vs. External View:** Documentation seems bifurcated between "insider memos" (`oracle1-vessel`) and "public-facing specs" (`cudaclaw`).

**P0 Issue (Negative Space):** The most dangerous gap is in operational documentation. A greenhorn cannot currently follow a clear path from "clone a repo" to "seeing a working fleet interaction." This is a critical blocker for scaling contributor numbers.

**P1 Opportunity (Safe Channel):** The safe channel is to use `plato-torch` as the gold standard template. Its structure (Vision -> Concepts -> Quick Start -> Architecture -> API) should be the baseline for all major fleet components.

**P2 Optimization:** For higher scores, all repos should include a **"Fleet Context"** subsection in their introduction and a **"Live Example"** in their quickstart that demonstrates interaction with at least one other fleet component (e.g., `fleet-simulator` generating tiles for a `plato-torch` room).
