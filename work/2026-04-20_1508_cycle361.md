## Step 1: Repository Selection & Analysis

I selected 5 repositories from the fleet's GitHub organization using a random sampling method. For each, I read the primary README.md and any other top-level documentation files. I scored them based on a rubric derived from fleet standards: clarity of purpose (P0), technical accuracy (P1), and onboarding utility (P2). Scores are out of 10.

### 1. **holodeck-rust**
**Score: 8/10**
**Analysis:**
*   **P0 (Purpose):** Excellent. The opening line, "A sentiment-aware MUD server for training NPCs with emergent personality," is a perfect north star. It immediately defines the project's role within the fleet (training data generation for PLATO rooms).
*   **P1 (Accuracy):** Strong. The `Getting Started` section provides a clear `cargo run` command. The architecture diagram (`docs/architecture.png`) is referenced and explains the core `WorldState` and `AgentSystem` components. API examples for spawning NPCs and querying sentiment are present and look functional.
*   **P2 (Utility):** Good, with a minor gap. It successfully connects the project to the wider fleet context ("feeds narrative tiles to PLATO rooms"). However, it lacks a "Quick Integration" snippet showing how to pipe its output to a `plato-torch` room preset. A new contributor might understand *what* it does but not the immediate *next step* for using it.
*   **Verdict:** High-quality, actionable documentation. The minor deduction is for the missing concrete integration example.

### 2. **constraint-theory-core**
**Score: 6/10**
**Analysis:**
*   **P0 (Purpose):** Clear but abstract. Defines itself as "A Rust library for geometric constraint solving, using deadband protocols to find stable solutions." A greenhorn understands it's about math and stability, but the direct application to fleet agents (e.g., kinematic planning for `JetsonClaw1`) is implied, not stated.
*   **P1 (Accuracy):** Technically sound but sparse. The API documentation generated by `cargo doc` is linked, which is good. However, the README itself contains only one code example: a basic 2D point-distance constraint. It doesn't show a full solving loop or error handling.
*   **P2 (Utility):** Low. There is no `Cargo.toml` dependency example, no guide on how to import and use the `Solver` struct, and no test file references. A developer would need to dive directly into the source code to proceed.
*   **Verdict:** Functional as an API reference, but fails as an onboarding guide. It documents the *interface* but not the *integration*.

### 3. **fleet-simulator**
**Score: 9/10**
**Analysis:**
*   **P0 (Purpose):** Outstanding. Opens with: "Generates multi-agent interaction logs for training Deadband Protocol compliance in PLATO rooms." This is precise, contextual, and leaves no doubt about its value.
*   **P1 (Accuracy):** Comprehensive. Documents three distinct simulation modes (`basic`, `conflict`, `cooperative`), each with a CLI example. The structure of the generated JSONL log format is explicitly detailed, which is critical for downstream data consumers.
*   **P2 (Utility):** Excellent. It provides a clear pipeline: `./simulate --mode conflict > training.jsonl`. It also includes a "Visualization" section pointing to a helper script for plotting agent trajectories. This directly enables the P2 (optimization) phase of the doctrine.
*   **Verdict:** A model README. It serves a developer, a researcher, and an operator equally well.

### 4. **plato-ensign**
**Score: 5/10**
**Analysis:**
*   **P0 (Purpose):** Partially clear. "Exports trained room adapters to LoRA format." This is correct but terse. It doesn't explain *why* this is done (to create portable specialist models from room experiences) or *when* (after a room reaches a certain tile threshold).
*   **P1 (Accuracy):** Problematic. The installation instructions mention a `requirements.txt` that does not exist in the repo root. The core command, `python export_lora.py --room_id <ID>`, is listed, but the required format of the `room_id` or where to find it is not specified.
*   **P2 (Utility):** Very low. There are no example configurations, no description of the output file structure, and no link to the `plato-torch` rooms it is meant to export from. A user is left with a command that will likely fail due to missing context.
*   **Verdict:** The most critical gap found. This README is incomplete and would block a workflow. It reads like internal notes, not public documentation.

### 5. **cudaclaw**
**Score: 7/10**
**Analysis:**
*   **P0 (Purpose):** Clear and compelling. "GPU-resident agent runtime for persistent, low-latency inference." It positions itself as the performance backbone for edge agents like `JetsonClaw1`.
*   **P1 (Accuracy):** Mixed. The build instructions for `CUDA 12.x` are detailed and correct. The core concept of a "kernel-hosted agent state" is well-explained. However, the API section lists functions like `claw_infer()` and `claw_update_state()` without any parameter descriptions or return value examples.
*   **P2 (Utility):** Fair. It includes a minimal "Hello World" example that loads a model and runs a single inference. It lacks a more complex example showing state persistence across multiple calls, which is the library's key feature.
*   **Verdict:** Good foundation that successfully explains the *why*. It needs expanded API docs and an advanced example to reach its full potential.

## Step 2: Fleet Documentation Health Summary

**Overall Average Score: 7.0/10**

**Strengths:**
*   **Strong North Star Alignment:** Most repos (`holodeck-rust`, `fleet-simulator`) excel at stating their purpose within the Cocapn vision. They understand they are part of a larger system.
*   **Functional Accuracy:** Core build/run instructions are generally present and correct, enabling a basic compile/test cycle.

**Critical Gaps (P0/P1 Issues):**
1.  **Incomplete Installation/Configuration:** `plato-ensign` has broken dependencies. This is a P0 blocker.
2.  **Lacking Integration Context:** Several repos (`constraint-theory-core`, `plato-ensign`) are documented as isolated libraries, not as connected components of the fleet. This violates the "hermit crab" metaphor—the shell isn't shown in its habitat.

**Optimization Opportunities (P2 Issues):**
1.  **Missing Advanced Examples:** Docs show how to start, but not how to *use* the tool meaningfully (`cudaclaw`, `constraint-theory-core`).
2.  **No Troubleshooting Sections:** None of the READMEs contain a "Common Issues" or "Debugging" guide.
