## Step 2: Documentation Analysis & Scoring

**Scoring Rubric (0-10):**
*   **0-3:** Missing or severely deficient. No context, no setup, no purpose.
*   **4-6:** Basic. States purpose and has some setup steps, but is incomplete, unclear, or lacks critical information for a new contributor.
*   **7-8:** Good. Clear purpose, setup instructions work, explains key concepts. May lack advanced usage or architectural context.
*   **9-10:** Excellent. Comprehensive, includes purpose, setup, architecture, usage examples, contribution guidelines, and connects to the wider fleet context.

### 1. plato-torch
**Score: 8/10**
**Analysis:**
*   **Strengths:** The README provides a strong, clear thesis ("Intelligence is not built. It is inhabited.") that aligns with the fleet vision. It explains core concepts like Rooms, Tiles, and Ensigns effectively. The "Quick Start" section is practical and includes a concrete example of creating a `DeadbandRoom`. The structure with "Core Concepts," "Quick Start," and "Architecture" is logical.
*   **Areas for Improvement:** While it mentions the 25 training room presets, it does not list or describe them. The setup instructions assume pre-existing knowledge of Python/poetry environments and do not mention dependency installation or potential environment issues. There is no "Contributing" section or guidelines for adding new room types. The connection to other systems (e.g., how tiles are submitted to a server) is hinted at but not detailed.

### 2. holodeck-rust
**Score: 6/10**
**Analysis:**
*   **Strengths:** The README has a compelling description: "A sentiment-aware MUD server where NPCs remember you." It clearly states the tech stack (Rust, Tokio, PostgreSQL) and has a "Getting Started" section with `cargo run` commands. The "Core Concepts" section defines key terms like Vessels, Channels, and Echoes.
*   **Areas for Improvement:** The "Getting Started" section is minimal and does not cover database setup/configuration, which is a critical component. The API and protocol for how agents or clients interact with the server are not documented. There are no examples of gameplay, NPC interaction, or how sentiment is manifested. The document feels like a high-level overview without the necessary details for someone to run or extend the system.

### 3. fleet-simulator
**Score: 5/10**
**Analysis:**
*   **Strengths:** The repository name and a brief description ("Generates synthetic training data for fleet AI by simulating multi-agent scenarios") immediately communicate its purpose. It lists dependencies (Python, numpy, etc.).
*   **Areas for Improvement:** This is essentially a stub. There is no "Usage" section, no example code, and no description of what a simulation scenario looks like or what format the output data takes. There are no instructions on how to configure agents, environments, or objectives. A contributor would have to read the source code to understand anything beyond the basic premise.

### 4. constraint-theory-core
**Score: 7/10**
**Analysis:**
*   **Strengths:** The README provides a good mathematical and conceptual foundation, explaining the problem domain (geometric constraint solving) and linking to relevant literature. It outlines the key components (Graph, Solver, Snapper) and their relationships. The presence of a "Build" section with the correct `cargo` command is useful.
*   **Areas for Improvement:** It lacks a practical "Quick Start" or "Examples" section. There is no code snippet showing how to define a simple constraint graph and solve it. The API is not described, leaving a user to guess how to interact with the Graph or Solver structs. The documentation is strong on the "why" and "what" but weak on the "how."

### 5. cudaclaw
**Score: 4/10**
**Analysis:**
*   **Strengths:** The tagline "GPU-resident agent runtime" is clear. It lists a bold technical vision ("Agents that never leave VRAM").
*   **Areas for Improvement:** The documentation is extremely sparse. There are no setup instructions, no requirements (CUDA version, GPU architecture), and no examples. Key questions are unanswered: How does an agent get loaded into the runtime? What is the interface between host and GPU agent? How is state managed? The README functions more as a placeholder or vision statement than as documentation for a usable project.
