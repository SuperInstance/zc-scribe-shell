## Step 2: Documentation Analysis & Scoring

**Scoring Rubric (0-10):**
*   **0-3:** Missing, severely outdated, or completely incorrect.
*   **4-5:** Basic README exists but lacks critical details (installation, purpose, usage).
*   **6-7:** Good README with purpose, setup, and basic examples. May lack advanced docs or architecture.
*   **8-9:** Excellent documentation. Clear purpose, setup, API/usage guide, contribution guidelines, and links to deeper resources.
*   **10:** Exemplary. Includes all of the above plus architecture diagrams, troubleshooting, design philosophy, and is actively maintained.

### 1. plato-torch
**Score: 8/10**
**Analysis:**
*   **Strengths:** The `README.md` is comprehensive. It clearly states the project's purpose ("PLATO rooms for PyTorch"). It provides a detailed "Quick Start" section with installation commands (`pip install -e .`) and a basic usage example. It documents the core `DeadbandRoom` API and lists all 25 available training room presets. It includes a "Development" section for contributors.
*   **Areas for Improvement:** While it lists presets, it doesn't describe what each one does or its intended use case. There is no high-level architecture diagram explaining how a "room" interacts with the training loop or the tile server. Advanced configuration or customization guides are absent.

### 2. holodeck-rust
**Score: 6/10**
**Analysis:**
*   **Strengths:** The `README.md` establishes a clear and compelling vision ("A MUD server where NPCs remember you"). It defines key concepts like "Sentiment-Aware NPCs" and "Ghost Tiles." It provides a `docker-compose up` command for easy startup.
*   **Areas for Improvement:** The documentation is primarily visionary. There is no "Usage" section explaining how to connect a client, interact with the world, or observe the sentiment system. There are no examples of game commands, API endpoints, or how to extend the server. The setup assumes Docker, but there's no guidance for native Rust compilation or development.

### 3. constraint-theory-core
**Score: 4/10**
**Analysis:**
*   **Strengths:** The repository name is descriptive. The `README.md` file exists and contains a single, technically precise sentence: "A Rust library for geometric constraint solving with focus on incremental solving and snapping."
*   **Areas for Improvement:** This is a bare minimum. There is no installation instruction (`cargo add`?), no code examples, no API documentation, and no explanation of what "geometric snapping" means in this context or how it's used by the fleet. The link to "constraint-theory" is broken (points to a non-existent `../constraint-theory`).

### 4. fleet-simulator
**Score: 5/10**
**Analysis:**
*   **Strengths:** The `README.md` correctly identifies the project's purpose: "A discrete-event simulator for generating training data from multi-agent fleet interactions." It lists the three core components (World, Agent, Event Loop) and provides a very basic code snippet showing their instantiation.
*   **Areas for Improvement:** The setup instructions are vague ("ensure you have Rust installed"). The provided example is trivial and doesn't show how to define agent behaviors, create meaningful events, or generate the "training data" mentioned in the purpose. There is no description of the output data format or how it integrates with other fleet systems like `plato-torch`.

### 5. cudaclaw
**Score: 3/10**
**Analysis:**
*   **Strengths:** The repository contains a `README.md` file.
*   **Areas for Improvement:** The documentation is essentially a placeholder. It states "GPU-resident agent runtime" but provides zero details. There are no instructions for building this likely complex CUDA/C++ project, no API overview, no explanation of how an "agent" is defined or loaded into GPU memory, and no connection to the broader fleet architecture. It fails its primary job: to inform a developer what this is and how to use it.

## Step 3: Summary Report

**Overall Fleet Documentation Health: Variable (Average Score: 5.2/10)**

**Patterns Observed:**
1.  **Vision vs. Execution:** Several repos (`holodeck-rust`, `fleet-simulator`) have strong visionary descriptions but lack practical, executable documentation.
2.  **Core vs. Peripheral:** The central `plato-torch` project is well-documented, suggesting effort is focused there. Supporting libraries (`constraint-theory-core`, `cudaclaw`) are severely under-documented, creating a "black box" problem for contributors.
3.  **Missing "Why" and "How":** Many READMEs state *what* the project is but omit *why* it exists in the fleet's context and *how* to operate it beyond the most basic command.

**Critical Gaps (P0 / P1 Issues):**
*   **`cudaclaw`** is a critical runtime component with near-zero documentation. This is a major blocker for understanding fleet infrastructure.
*   **`constraint-theory-core`** is a library with no usage examples, making its integration opaque.
*   Repositories lack consistent sections (Purpose, Quick Start, API Reference, Development, Integration).

**Recommendations (P2 Optimizations):**
1.  **Immediate Priority:** Create a minimal viable `README.md` for `cudaclaw` and `constraint-theory-core` with build instructions and a simple example.
2.  **Template:** Establish a fleet-wide `README-template.md` to ensure consistency across all repos.
3.  **Enhance:** Expand `holodeck-rust` and `fleet-simulator` docs with concrete, copy-paste examples of typical use cases.
4.  **Connect:** Add an "Integration" or "Fleet Context" section to each README, linking to related projects (e.g., `plato-torch` README should mention `cudaclaw` as a potential runtime target).

This analysis provides a factual baseline. The scores reflect the current state of discoverability and usability for a new agent or contributor trying to understand and work with the fleet's codebase.
