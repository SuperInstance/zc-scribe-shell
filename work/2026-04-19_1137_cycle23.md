## Step 2: Evaluation Rubric

I will score each repository (0-10) based on the following criteria:

*   **Clarity & Purpose (2 pts):** Is the project's goal immediately clear?
*   **Getting Started (3 pts):** Are installation, setup, and a basic "hello world" example provided and functional?
*   **Architecture/Usage (3 pts):** Is there sufficient explanation of core concepts, key components, and how to use the project for its intended purpose?
*   **Project Health (2 pts):** Are there links to related projects, contribution guidelines, or clear indicators of active development?

## Step 3: Repository Analysis & Scores

### 1. cudaclaw
*   **Clarity & Purpose (2/2):** Excellent. The README opens with a clear, bold statement: "A GPU-resident runtime for persistent AI agents." The "Why?" section effectively contrasts it with traditional CPU-bound inference servers.
*   **Getting Started (3/3):** Very good. Provides a `Dockerfile` for the recommended environment, a `requirements.txt`, and clear, copy-paste commands for building (`make`) and running the example agent. The `example_agent.py` is simple and demonstrates the core `step()` function.
*   **Architecture/Usage (2.5/3):** Good overview with an architecture diagram and descriptions of the Manager, Worker, and Kernel layers. It explains the core loop (`load -> step -> commit`). Could be improved with more detail on agent state serialization or a more complex multi-agent example.
*   **Project Health (1.5/2):** Links to the related `plato-torch` project. Lacks a `CONTRIBUTING.md` or explicit development status. The presence of a `Makefile` and CI (`build.yml`) suggests active engineering.
*   **Total Score: 9/10**

### 2. flux-runtime
*   **Clarity & Purpose (1.5/2):** Purpose is clear from the tagline ("self-assembling bytecode VM for AI") and the first paragraph. The high-concept analogy to "crystalline computation" is intriguing but slightly obscures the concrete "what it does" on first read.
*   **Getting Started (1/3):** Major weakness. There are **no installation instructions**. The `Cargo.toml` confirms it's a Rust project, but a user is left guessing. There is a `examples/` directory, but no guidance on how to compile or run them.
*   **Architecture/Usage (2/3):** Contains good conceptual explanations: the stack machine, flux "atoms," and the idea of self-assembly via affinity scores. The `examples/simple.flux` is helpful. However, it lacks a tutorial or API reference explaining how to *integrate* this VM into another project.
*   **Project Health (1/2):** No links to related projects or contribution guidelines. The codebase appears to be in early, active research.
*   **Total Score: 5.5/10**

### 3. fleet-simulator
*   **Clarity & Purpose (2/2):** Perfectly clear. "A discrete-event simulation framework for generating training data from multi-agent fleet interactions." Directly states its role in the Cocapn ecosystem.
*   **Getting Started (3/3):** Excellent. Clear `pip install` instructions. A complete, runnable example (`example.py`) is provided in the README itself, which creates a simple fleet, runs a simulation, and visualizes the output. This is the gold standard for onboarding.
*   **Architecture/Usage (3/3):** Very good. Explains the core `Fleet`, `Agent`, `Event` abstractions. Documents the key `FleetSimulator` class and its methods (`add_agent`, `step`, `run`). The example covers the primary use case effectively.
*   **Project Health (2/2):** Links to `plato-torch` for the training pipeline. Includes a `Makefile` for development tasks and a `pyproject.toml` indicating modern Python packaging.
*   **Total Score: 10/10**

### 4. constraint-theory-core
*   **Clarity & Purpose (1/2):** The purpose ("A Rust library for geometric constraint solving") is technically stated, but the *why* within the Cocapn vision is missing. How does "geometric snapping" relate to AI or the Deadband Protocol? The context is absent.
*   **Getting Started (0.5/3):** Very poor. Only a `cargo build` instruction. No example of how to create a constraint system, add points/lines, or solve it. The `examples/` directory is empty. A user cannot get started.
*   **Architecture/Usage (1/3):** Lists module names (`solver`, `geometry`, `constraints`) but provides no explanation of the API, key structs (`Solver`, `Point`), or how to use them. The documentation is essentially the module tree.
*   **Project Health (0.5/2):** No links to related projects or contribution info. Appears to be a very early-stage, internal library.
*   **Total Score: 3/10**

### 5. holodeck-rust
*   **Clarity & Purpose (2/2):** Very clear. "A sentiment-aware MUD server for generating human-like NPC dialogue." Directly connects to the fleet's research into believable simulations.
*   **Getting Started (2.5/3):** Good. Provides `cargo run` instructions. Has a detailed `SETUP.md` explaining how to configure the required OpenAI API key. The `example_scenario.json` is well-commented. Lacks a simple "first connection" telnet/netcat example to see immediate output.
*   **Architecture/Usage (2.5/3):** Good. Explains the `Scenario`, `Actor`, `Location` model. Describes the sentiment engine and the turn-based dialogue loop. Could benefit from a sequence diagram of a full interaction cycle.
*   **Project Health (1.5/2):** Links to the research concept (`fleet-research/rabbit-trails/holodeck-theory.md`). Has a `Makefile`. Lacks contribution guidelines.
*   **Total Score: 8.5/10**

## Step 4: Summary Report

**Overall Fleet Documentation Health: Mixed.**

*   **Strengths:** Core simulation and runtime projects (`fleet-simulator`, `cudaclaw`, `holodeck-rust`) have excellent to very good documentation. They follow the "show, don't just tell" principle with working examples.
*   **Weaknesses:** Research-oriented (`flux-runtime`) and lower-level libraries (`constraint-theory-core`) suffer from a lack of practical onboarding guides. They explain the *idea* but not the *mechanics*.
*   **Pattern:** Projects that are directly used to generate tiles or run agents have better docs. Foundational research projects are under-documented, creating a barrier to understanding the full stack.
