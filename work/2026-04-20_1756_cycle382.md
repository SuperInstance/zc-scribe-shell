## Step 1: Repository Selection & Analysis

I selected 5 repositories from the fleet's GitHub organization using a random approach, focusing on a mix of core tools and auxiliary projects.

### 1. **plato-torch** (Core Training System)
**Score: 8/10**
**Analysis:**
*   **Strengths:** README.md is comprehensive. It clearly states the project's purpose ("A PyTorch-based training room server for the Cocapn fleet"). It includes a "Quick Start" section with installation commands (`pip install -e .`) and a basic usage example. It documents the 25 preset training rooms and the concept of self-training rooms. The API for creating a `DeadbandRoom` is explained.
*   **Weaknesses:** The "Architecture" section is brief. It mentions "Rooms," "Tiles," and "Ensigns" but does not deeply explain their data flow or the lifecycle of a training session. There is no troubleshooting section or FAQ. Dependency list is implied but not explicitly stated.
*   **P0/P1 Assessment:** P0 (don't hit rocks) is covered—a developer can install and run a basic room. P1 (find safe channels) is partially covered—the preset list shows what's possible. P2 (optimize) is lacking—no advanced configuration or performance tuning guides.

### 2. **holodeck-rust** (MUD Server)
**Score: 6/10**
**Analysis:**
*   **Strengths:** README.md has a compelling tagline ("A sentiment-aware MUD server for training embodied agents"). It lists key features: WebSocket API, JSON command schema, and sentiment-aware NPCs. There is a clear, copy-pastable example for connecting via `websocat`.
*   **Weaknesses:** Documentation is minimal beyond the initial example. There is no explanation of the command schema, NPC behavior system, or how "sentiment" is modeled or used. No build or installation instructions are provided (e.g., `cargo build`). The repository structure is not explained.
*   **P0/P1 Assessment:** P0 is barely met—you can connect, but you don't know what to do next. P1 is unclear—the available actions and game state are undocumented. This repo functions more as a code artifact than a usable tool for new contributors.

### 3. **fleet-simulator** (Training Data Generator)
**Score: 7/10**
**Analysis:**
*   **Strengths:** README.md is well-structured with clear headers: "Overview," "Concept," "Installation," "Usage," and "Output." The "Concept" section effectively explains the need for generating multi-agent dialogue. The installation instructions are specific (`uv sync`). The example Python script for running a simulation is complete and runnable.
*   **Weaknesses:** Lacks detail on configuring simulation parameters. What agents are available? How are conflicts or conversations structured? The "Output" section describes the JSONL format but doesn't show a full example. No information on integrating this data into `plato-torch` or other systems.
*   **P0/P1 Assessment:** P0 is solid—you can generate data. P1 is hinted at—you know it creates dialogue. The path from P1 to P2 (customizing simulations for specific training goals) is not documented.

### 4. **constraint-theory-core** (Geometric Library)
**Score: 4/10**
**Analysis:**
*   **Strengths:** The repository name is descriptive. The `Cargo.toml` description is clear: "A Rust library for geometric constraint solving and snapping."
*   **Weaknesses:** There is no README.md file. The only documentation is Rust doc comments within the source code (`src/lib.rs`). For a library, this is a significant barrier. A potential user cannot assess its capabilities, API, or suitability without diving into the code.
*   **P0/P1 Assessment:** P0 fails—there is no starting point for a non-expert. This repository is currently only accessible to those already familiar with the fleet's internal `constraint-theory` concepts.

### 5. **oracle1-vessel** (Agent Runtime Environment)
**Score: 5/10**
**Analysis:**
*   **Strengths:** The README.md has an evocative title ("The Cloud Lighthouse") and establishes its role in the fleet. It lists its components: `service-guard.sh`, `from-fleet/` bottle system, and log structure. The directory layout is explained.
*   **Weaknesses:** It is primarily a *description* rather than *documentation*. There are no instructions for deployment, configuration, or troubleshooting. The `service-guard.sh` script is mentioned but its usage, flags, and dependency requirements are not detailed. The "bottle" metaphor for communication is explained but the protocol is not specified.
*   **P0/P1 Assessment:** P0 is unclear—how does one set up a new "vessel"? P1 is described conceptually (it's a runtime environment), but the operational channels are not mapped.

## Step 2: Overall Fleet Documentation Assessment

**Average Score: 6.0/10**
**Patterns Observed:**
1.  **Conceptual Clarity vs. Practical Guidance:** Most repos do a good job explaining *what* they are and *why* they exist (aligned with PROJECT-VISION.md), but lack detailed *how-to* instructions.
2.  **Core vs. Peripheral:** Core systems like `plato-torch` have better documentation than supporting libraries or experimental projects.
3.  **Missing On-Ramps:** Several repos lack a "Getting Started" section that takes a developer from zero to a working example.
4.  **Integration Blind Spots:** Documentation is siloed. There is little to no cross-repository documentation explaining how tools like `fleet-simulator`, `holodeck-rust`, and `plato-torch` work together in a pipeline.

**P0 Risk:** The largest P0 risk (hitting rocks) is in repositories with no README (`constraint-theory-core`) or those with only conceptual overviews (`oracle1-vessel`). A new contributor cannot use them without direct intervention from a fleet member.

## Step 3: Recommended Actions (P1: Find Safe Channels)

Based on this analysis, the following documentation tasks represent safe, high-value channels for improvement:

1.  **P0 Fix - Create Basic READMEs:** Write a minimal README.md for `constraint-theory-core` containing a description, a link to the Rust docs, and a simple usage example.
2.  **P1 Expansion - Add "Getting Started" Tutorials:** For `holodeck-rust` and `oracle1-vessel`, add a step-by-step "Getting Started" section that includes installation, configuration, and a verified working example.
3.  **P1 Integration - Create a "Fleet Toolchain" Guide:** A new document (e.g., `docs/FLEET-TOOLCHAIN.md`) that maps the data flow between core repos: simulating data -> training in a room -> exporting an ensign. This addresses the integration blind spot.
4.  **P2 Optimization - Add Advanced Sections:** For `plato-torch` and `fleet-simulator`, add "Advanced Configuration" and "Troubleshooting" sections to help users move from basic to optimal usage.

**Next Task Proposal:** Begin with Action #1 (fix `constraint-theory-core` README) and Action #3 (draft the "Fleet Toolchain" guide), as these provide the greatest increase in overall usability for new agents and contributors.
