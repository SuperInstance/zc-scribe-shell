## Step 1: Repository Selection & Analysis

I have selected 5 repositories from the fleet's directory structure. I will read their primary documentation files (README.md, any CONTRIBUTING.md, etc.) and score them based on a standard rubric.

**Scoring Rubric (0-10):**
*   **0-2:** Missing or nearly empty.
*   **3-4:** Exists but is incomplete, outdated, or confusing.
*   **5-6:** Basic information present (what it is, how to run). Lacks context, examples, or architecture.
*   **7-8:** Clear, includes purpose, setup, and basic usage. May lack advanced details or integration notes.
*   **9-10:** Excellent. Includes vision, architecture diagrams, detailed setup, examples, API docs, and contribution guidelines.

## Step 3: Summary Report & Recommendations

**Overall Fleet Documentation Health: Variable.** Scores range from 9 (`plato-torch`) to 4 (`cudaclaw`), with an average of **6.2/10**. The flagship project (`plato-torch`) is excellently documented, providing a strong core reference. However, supporting and experimental projects suffer from incomplete or placeholder documentation, which creates friction for new contributors and obscures their purpose and progress.

**P1 (Find Safe Channels) Recommendations:**

1.  **Establish a `docs/` folder pattern:** For repos scoring below 7, mandate a `docs/` directory with at minimum:
    *   `SETUP.md` - Detailed, copy-pastable installation and environment setup.
    *   `ARCHITECTURE.md` - How the project fits into the fleet and its internal design.
    *   `EXAMPLES.md` - At least one complete, runnable example.
2.  **Template Injection:** Apply the successful structure from `plato-torch`'s README (Vision, What It Is/Is Not, Quick Start, Architecture) as a template to other repos.
3.  **Link the Wiki:** For repos like `fleet-simulator` that reference a wiki, a bootstrap script should create the initial wiki pages from template content.

**P2 (Optimize) Recommendations:**

1.  **Priority Fix:** Update `fleet-simulator` README immediately. It is a key data generation tool with no usage guide.
2.  **Contextual Links:** Enhance READMEs like `constraint-theory-core` with a "Fleet Context" section linking to related projects.
3.  **Honest Roadmaps:** For `cudaclaw`, convert the README into a `PROPOSAL.md` and create a new, minimal README that documents any existing proof-of-concept code, even if it's just a build command.

**Next Task for Scribe:** Begin implementing P1 recommendations by creating a `DOCUMENTATION-STANDARD.md` template and rewriting the `fleet-simulator/README.md` as a first example.
