## Step 1: Repository Selection & Analysis

I have selected 5 repositories from the fleet's public GitHub organization (`cocapn`). The selection was random but aimed to cover different project types (core runtime, simulator, library, tool, example). I read the primary `README.md` file for each.

### 1. **cudaclaw**
**Description:** GPU-resident agent runtime for NVIDIA GPUs.
**Score:** 8/10
**Analysis:**
*   **Strengths:** Excellent technical overview. Clearly states the problem (CPU-GPU latency) and solution (CUDA-resident agents). Includes a concise architectural diagram. Provides a "Quick Start" with prerequisites and build commands. Lists key features.
*   **Areas for Improvement:** Lacks a "Contributing" section. The "Getting Started" could be more detailed for users unfamiliar with CUDA toolchains. No explicit link to related fleet projects (e.g., plato-torch).

### 2. **fleet-simulator**
**Description:** A simulation environment for generating training data by modeling fleet agent interactions.
**Score:** 6/10
**Analysis:**
*   **Strengths:** Clear, high-level purpose statement aligned with fleet doctrine ("The work IS the training"). Good conceptual explanation of why simulation is needed.
*   **Areas for Improvement:** Severely lacks implementation details. No installation instructions, dependency list, or API overview. The "Usage" section is a placeholder. Does not explain how it integrates with `plato-torch` for room training. The documentation is a vision statement, not a user guide.

### 3. **constraint-theory-core**
**Description:** A Rust library implementing geometric constraint solving for CAD-like snapping.
**Score:** 9/10
**Analysis:**
*   **Strengths:** Near-exemplary for a library. Starts with a "What is Constraint Theory?" primer. Provides a complete, runnable code example in the `README`. Includes detailed API documentation generated by `cargo doc`. Links to a dedicated `examples/` directory. Clear `Cargo.toml` dependency snippet.
*   **Areas for Improvement:** Could briefly mention potential applications within the fleet (e.g., UI tools, spatial reasoning for agents). A "Related Crates" section would be useful.

### 4. **service-guard**
**Description:** Bash-based process monitor for keeping fleet services (PLATO, MUD, etc.) alive.
**Score:** 4/10
**Analysis:**
*   **Strengths:** The purpose is immediately clear from the title and first sentence.
*   **Areas for Improvement:** Very sparse. Only lists the 5 services it monitors without explaining what they are or their dependencies. No configuration instructions. No example of log output or how to interpret service health. No troubleshooting steps. Relies on the user's pre-existing knowledge of the fleet's service stack.

### 5. **plato-ensign**
**Description:** Tool for exporting trained "room" experiences from `plato-torch` into LoRA adapters.
**Score:** 7/10
**Analysis:**
*   **Strengths:** Good conceptual bridge between two core projects (`plato-torch` rooms and LoRA models). Explains the "why" (creating portable specialist models). Shows the basic command structure.
*   **Areas for Improvement:** Assumes the user has a fully trained room checkpoint, without linking to `plato-torch` documentation on how to create one. Lacks a concrete example with sample input/output. Does not specify supported model architectures or LoRA parameters.

## Step 2: Summary Report & Recommendations

**Overall Fleet Documentation Health: Variable.** Scores ranged from 4 to 9, indicating inconsistent standards across projects. Core libraries (`constraint-theory-core`) are well-documented, while operational tools (`service-guard`) and newer/simulator projects (`fleet-simulator`) lack practical guides.

**Common Patterns:**
*   **Strong:** Most repos successfully communicate their *purpose* and place in the fleet's vision.
*   **Weak:** Many lack actionable "Getting Started" sections, clear dependencies, and integration guides pointing to sibling projects.

**P1 Recommendations (Safe Channels - Foundational Fixes):**
1.  **Template & Checklist:** Create a `README-template.md` for the fleet, mandating sections: Purpose, Prerequisites, Installation, Basic Usage, Integration (Links to related repos), and API/Configuration Reference.
2.  **Immediate Triage:** Prioritize improving the documentation for `service-guard` and `fleet-simulator`. These are operational and data-generation tools; poor docs create friction for fleet bootstrapping and maintenance.
3.  **Cross-Linking:** Add a "## Related Projects" section to all repos, explicitly linking to upstream/downstream dependencies (e.g., `plato-ensign` should link to `plato-torch`).

**P2 Recommendation (Optimization):**
*   Establish a documentation review gate in the contribution workflow, where any PR modifying core functionality must also update the corresponding `README` sections.

**My Next Action (As Scribe):** I will begin implementing P1 by drafting the fleet `README-template.md` and then rewriting the `service-guard` README as a first example.
