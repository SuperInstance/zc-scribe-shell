## Step 2: Analysis & Scoring

### 1. `plato-torch`
**Score: 8/10**
**Documentation Found:** `README.md`
**Analysis:**
*   **Strengths:** Excellent. Opens with a powerful, clear thesis: "PLATO rooms are shells. Agents are hermit crabs. Tiles are scratches." Immediately establishes the project's philosophical core within the fleet. Provides a concise "Quick Start" with installation (`pip install plato-torch`) and a minimal working example. Documents the core `DeadbandRoom` API clearly. Includes a well-structured "Architecture" section explaining Rooms, Tiles, and Ensigns. Links to related projects (`plato-ensign`).
*   **Areas for Improvement:** Lacks a dedicated "Troubleshooting" or "Common Issues" section. The API documentation is present but could be expanded with more complex examples or a link to auto-generated API docs. A diagram of the data flow (Tile -> Room -> Ensign) would elevate it to a 9.

### 2. `holodeck-rust`
**Score: 6/10**
**Documentation Found:** `README.md`
**Analysis:**
*   **Strengths:** Clearly states its purpose: "A sentiment-aware MUD (Multi-User Dungeon) server for generating training data via social interaction." The "Features" list is useful. Provides build instructions (`cargo build --release`) and a basic example of server startup and client connection.
*   **Areas for Improvement:** The setup instructions are minimal and assume familiarity with Rust and MUD clients. There is no explanation of the "sentiment-aware" mechanics or how NPC interactions generate training data. No API documentation for extending the server or creating new scenarios. Lacks context on how it integrates with the broader fleet (e.g., how its data feeds into `plato-torch` rooms).

### 3. `fleet-simulator`
**Score: 5/10**
**Documentation Found:** `README.md`
**Analysis:**
*   **Strengths:** The purpose is clear: "A discrete-event simulation framework for generating multi-agent interaction data." Identifies itself as a key data source for the fleet. Mentions core concepts (Entities, Systems, Events).

A contributor would have to read the source code to understand how to use or extend it. It does not explain what a "simulation scenario" looks like or how to define one.

### 4. `constraint-theory-core`
