## Step 2: Documentation Scoring & Report

I will score each repository's documentation on a scale of 0-10 based on the following criteria derived from fleet doctrine and standard open-source practices:
*   **P0 (0-3):** Does it prevent catastrophic misunderstanding? (Purpose, basic setup, critical warnings)
*   **P1 (4-7):** Does it provide a clear, safe channel for a contributor to get started? (Installation, key concepts, basic examples)
*   **P2 (8-10):** Does it optimize the contributor's path to mastery? (API reference, architecture, contribution guide, advanced use cases)

### **1. plato-torch**
**Score: 9/10**
**Analysis:**
*   **P0:** Excellent. The README immediately states the core thesis ("PLATO rooms are shells. Agents are crabs. Tiles are scratches.") and defines key terms (Room, Tile, Ensign). A clear "Quick Start" section shows the minimal code to create a room.
*   **P1:** Very strong. It provides a detailed "Architecture" breakdown, lists all 25 training room presets, and includes a practical "Exporting an Ensign" example. The `Getting Started` guide is procedural and clear.
*   **P2:** Strong. It links to related projects (`plato-ensign`, `holodeck-rust`), discusses the concept of "Ghost Tiles," and hints at the larger system (`Mirror Play`, `The Shell`). A dedicated "Contributing" section or more detailed API documentation for the `DeadbandRoom` class would push it to a 10.

### **2. holodeck-rust**
**Score: 7/10**
**Analysis:**
*   **P0:** Good. The tagline "A sentiment-aware MUD server for PLATO rooms" clearly defines its purpose. It states it's a work-in-progress research prototype, which is a critical warning.
*   **P1:** Adequate. It provides build instructions (`cargo build`), a basic example of connecting via `telnet`, and explains core entities (World, Zone, Actor, Item). The explanation of "sentiment-aware NPCs" is conceptually clear.
*   **P2:** Limited. The documentation ends abruptly. There is no guide on how to define new zones, items, or NPC behaviors. There is no API documentation for the `Holodeck` struct or how to integrate it with a `plato-torch` room. The path from user to contributor is not mapped.

### **3. fleet-simulator**
**Score: 6/10**
**Analysis:**
*   **P0:** Sufficient. The README states it's for "generating multi-agent interaction data for PLATO room training." This establishes its role in the fleet ecosystem.
*   **P1:** Basic. It lists dependencies (Python, `mesa`) and has a "Running the Simulator" section with a command. A simple example scenario is described.
*   **P2:** Very limited. The documentation lacks depth. There is no explanation of the simulation model, agent definitions, environment rules, or data output format. There is no guidance on how to create new simulation scenarios or extend the framework. A contributor would need to read the source code to understand how to use it effectively.

### **4. constraint-theory-core**
**Score: 8/10**
**Analysis:**
*   **P0:** Very good. Opens with a clear, mathematical definition: "A Rust library for geometric constraint solving via iterative relaxation." The "Note" about being a research prototype is prominent.
*   **P1:** Strong. It provides a well-commented code example in the README that demonstrates creating points, constraints, and solving the system. The `Cargo.toml` dependency is shown.
*   **P2:** Good. It includes a "Theory" section that outlines the algorithm (mass-spring relaxation) and key concepts. However, it lacks a full API reference for the `Solver` and `Constraint` types, and there is no "Contributing" guide or discussion of performance characteristics/limitations.

### **5. cudaclaw**
**Score: 4/10**
**Analysis:**
*   **P0:** Minimal but present. The description "GPU-resident agent runtime" indicates its domain. The "⚠️ Experimental" warning is a critical P0 element.
*   **P1:** Very limited. It lists build requirements (CUDA, C++20) but provides no build instructions or example of how to compile the project. There is no example code snippet showing how to initialize the runtime or load an agent.
*   **P2:** Absent. There is no explanation of the architecture, how agents interact with the GPU, what the API looks like, or how it integrates with other fleet components like `plato-torch`. The documentation consists only of a high-concept description and a requirements list, which is insufficient for a user or contributor.
