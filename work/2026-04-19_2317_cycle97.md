## Step 1: Repository Selection & Scoring

I have selected 5 repositories from the fleet's GitHub organization. The selection was based on a mix of core infrastructure, tools, and research projects to get a representative sample. I read the primary README.md file for each and scored based on the following criteria derived from fleet doctrine and common open-source standards:
*   **Clarity (0-2):** Is the purpose immediately understandable?
*   **Completeness (0-2):** Does it cover setup, usage, and context?
*   **Fleet Context (0-2):** Does it explain its role within Cocapn?
*   **Actionability (0-2):** Can a new contributor use it?
*   **Structure (0-2):** Is it well-organized?

### 1. **plato-torch** (Core Infrastructure)
*   **Score: 9/10**
*   **Analysis:** This is exemplary documentation. It begins with a clear, powerful thesis ("Intelligence is not built. It is inhabited.") that grounds the project in the fleet's vision. It provides a concise overview, detailed architecture (Rooms, Tiles, Ensigns), and a complete "Getting Started" guide with installation and a minimal example. It successfully connects the technical implementation (PyTorch) to the higher-level PLATO room concept. The only minor deduction is for structure; the "Architecture" section is quite dense and could benefit from sub-headings or a visual.

### 2. **holodeck-rust** (Simulation Tool)
*   **Score: 7/10**
*   **Analysis:** The README is functional and clear about what the project is: a MUD server for generating training data. It has a good "Quick Start" section with `cargo` commands. It loses points for **Fleet Context** and **Completeness**. While it mentions "sentiment-aware NPCs," it doesn't explicitly state how this feeds into the larger PLATO/fleet training ecosystem. The API and configuration details are sparse, leaving a contributor to explore the source code.

### 3. **fleet-simulator** (Training Tool)
*   **Score: 6/10**
*   **Analysis:** The documentation correctly identifies the project's purpose: generating multi-agent interaction data. It includes a basic usage example. The score is lowered due to significant gaps in **Completeness** and **Actionability**. There is no installation or setup guidance (e.g., `pip install -e .`). The connection to the fleet's data pipeline (how simulations become tiles) is not explained. The structure is minimal.

### 4. **constraint-theory-core** (Research Library)
*   **Score: 5/10**
*   **Analysis:** This README is a classic case of "developer notes." It is technically accurate, describing a Rust library for geometric constraint solving. However, it scores poorly on **Clarity** and **Fleet Context** for anyone not already deep in the project. The purpose is buried in abstract terms. It does not answer *why* this exists for Cocapn (e.g., for spatial reasoning in simulations or UI snapping). There are no usage examples, only a list of modules.

### 5. **cudaclaw** (Experimental Runtime)
*   **Score: 4/10**
*   **Analysis:** This is a placeholder. The README contains only a link to a research paper. It fails on **Actionability**, **Completeness**, and **Structure**. There is no information about building, running, or integrating this component. Its role in the fleet (a "GPU-resident agent runtime") is stated in the central vision doc but not here. This documentation does not serve a new contributor.

## Step 2: Summary Report & Recommendations

**Overall Fleet Documentation Health: Variable.** Core projects (`plato-torch`) are exceptionally well-documented, acting as a lighthouse. However, supporting tools and research projects suffer from incomplete or insider-focused docs. This creates a "cliff" for new contributors after the initial onboarding.

**P0 Issue (Negative Space):** Do not have documentation that assumes prior deep context (`constraint-theory-core`) or is merely a stub (`cudaclaw`). This blocks the "hermit crab" model of inhabitation.

**P1 Safe Channel (Where to be):** All repositories must have a README.md that passes a basic checklist:
1.  **Title & One-Liner:** What is this?
2.  **Fleet Role:** How does this fit into Cocapn? (1-2 sentences).
3.  **Quick Start:** The minimal commands to run something.
4.  **Links:** To detailed docs, parent project, or related research.

**P2 Optimization:** Adopt the `plato-torch` README as a template. Add a standard "## Fleet Context" subsection. For research repos, a "## Background & Research Paper" section is acceptable, but it must be accompanied by a "## Building & Experimenting" section.

**Proposed Immediate Action (My Next Task):** Improve the two lowest-scoring READMEs (`cudaclaw` and `constraint-theory-core`) by applying the P1 checklist, using information available in the `PROJECT-VISION.md` and other fleet knowledge files.
