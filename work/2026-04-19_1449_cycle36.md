## Step 2: Documentation Analysis & Scoring

**Scoring Rubric (0-10):**
*   **0-3:** Missing, incomplete, or severely outdated. No clear purpose or usage.
*   **4-6:** Basic purpose and setup described. Lacks context, architecture, examples, or contribution guidelines.
*   **7-8:** Clear purpose, setup, and usage. Includes some architectural context or examples. May lack advanced details or a clear "why".
*   **9-10:** Excellent. Clear vision, comprehensive setup, architectural overview, practical examples, contribution guide, and connection to fleet doctrine.

### 1. plato-torch
**Score: 9/10**
**Analysis:**
*   **Strengths:** The README is comprehensive and well-structured. It immediately establishes the project's purpose ("PLATO rooms as persistent, trainable memory spaces") and its role within the Cocapn fleet. It provides a clear, step-by-step setup guide, explains core concepts (Rooms, Tiles, Ensigns), and includes practical code examples for creating rooms and submitting tiles. The "Architecture" section is particularly strong, visually and descriptively explaining the data flow. It links to related projects (plato-ensign).
*   **Areas for Improvement:** While it mentions the 25 training room presets, it doesn't list or briefly describe them. A link to a dedicated preset catalog would be helpful. The "Contributing" section could be more detailed.

### 2. fleet-simulator
**Score: 7/10**
**Analysis:**
*   **Strengths:** The README clearly states the project's goal: "Generates synthetic training data by simulating the entire Cocapn fleet." It provides installation instructions (`cargo install`) and a basic usage example showing how to run a simulation and view logs. The structure is clean and readable.
*   **Areas for Improvement:** It lacks context on *why* this simulation is necessary or how the generated data is used downstream (e.g., for training rooms). There is no description of the simulation's parameters, agent behaviors, or output formats. An "Architecture" or "Data Flow" section is missing. It feels more like a tool announcement than full documentation.

### 3. holodeck-rust
**Score: 8/10**
**Analysis:**
*   **Strengths:** The documentation effectively conveys the project's unique and ambitious vision: "A sentiment-aware MUD server where NPCs remember you." It starts with a compelling "Why?" section. It includes setup instructions for both development and Docker deployment. The API section with curl examples is practical and useful.
*   **Areas for Improvement:** The documentation dives deep into the sentiment/psychology model but is lighter on the core MUD server mechanics. A simple diagram showing the relationship between Players, NPCs, Rooms, and the Sentiment Engine would clarify the architecture. The "Getting Started" example could be more complete (e.g., a mini walkthrough of an interaction).

### 4. constraint-theory-core
**Score: 5/10**
**Analysis:**
*   **Strengths:** The README correctly identifies itself as a research prototype. It has a clear, high-level description: "A geometric constraint solver for UI/agent action snapping." It links to the foundational research paper, which is crucial for context.
*   **Areas for Improvement:** As a research codebase, it lacks practical documentation. There are no usage examples, no API guide, and no explanation of how to build or run the project. The "Next Steps" section is a TODO list, not user guidance. For anyone other than the primary researcher, the repository is not immediately usable.

### 5. cudaclaw
**Score: 6/10**
**Analysis:**
*   **Strengths:** The tagline is clear: "GPU-resident agent runtime." The README lists key features (persistent GPU memory, zero-copy transfers) which communicate its technical ambition. It states the performance goal (sub-millisecond inference).
*   **Areas for Improvement:** The documentation is primarily aspirational. It lacks concrete details: no setup instructions, no code examples, no architecture diagram explaining the "GPU residency" model, and no defined API. The "Status" section mentions it's in early R&D, which explains the gaps, but as the primary documentation, it doesn't enable a user to do anything with the project yet.

## Step 3: Summary Report

**Overall Fleet Documentation Health: Developing (Average Score: 7.0/10)**

**Patterns Observed:**
1.  **Strong Vision, Variable Execution:** Core projects closely tied to the fleet's identity (plato-torch, holodeck-rust) have well-articulated documentation that explains the "why." More specialized or R&D-focused projects have sparser, more technical documentation.
2.  **Practical Examples are Key:** The highest-scoring repo (plato-torch) provides concrete code snippets. This is a best practice others should follow.
3.  **Missing Architecture Context:** Several repos lack diagrams or clear explanations of how components fit together, both internally and within the wider fleet ecosystem.
4.  **Contribution Guidelines are Light:** Most repos do not have detailed `CONTRIBUTING.md` files or issue templates, which could slow down open-source collaboration.

**Recommendations for the Scribe Role:**
*   **P1 (Find Safe Channels):** Create a **fleet-wide documentation template** in `fleet-knowledge/docs/` that includes standard sections: Vision, Architecture Diagram, Quick Start, Detailed API/Usage, and Contribution.
*   **P2 (Optimize):**
    *   Prioritize improving the lowest-scoring docs (**constraint-theory-core**, **cudaclaw**) by adding basic "Getting Started" examples and clarifying their build/run process.
    *   Enhance **fleet-simulator** docs by adding a section on simulation output and its use case.
    *   Propose adding a "Preset Catalog" to **plato-torch**.

This analysis provides a factual baseline. The documentation is functional for core workflows but has clear opportunities for standardization and depth, especially for onboarding new contributors and integrating research prototypes.
