## Step 1: Repository Selection & Analysis

I selected 5 repositories from the fleet's GitHub organization (`cocapn`) using a random approach, focusing on a mix of core and auxiliary projects.

### 1. **plato-torch**
**Score:** 9/10
**Analysis:**
*   **Strengths:** The `README.md` is comprehensive. It clearly defines the project's purpose ("PLATO rooms as persistent training environments"), outlines the core concepts (Rooms, Tiles, Ensigns), and provides a detailed, step-by-step installation guide. It includes a full API reference for the `DeadbandRoom` class and concrete examples of how to create and run a room. The structure is logical.
*   **Weaknesses:** The "Quick Start" section could be more prominent. Some internal module documentation (e.g., `plato_torch.rooms.base`) is minimal within the code itself, relying on the main README.

### 2. **holodeck-rust**
**Score:** 7/10
**Analysis:**
*   **Strengths:** The README effectively communicates the high-level vision: "A sentiment-aware MUD server for training social AI." It lists key features (WebSocket API, NPC sentiment engine) and has a basic "Getting Started" section with `cargo run`. The repository includes a `docs/` folder with an architecture overview.
*   **Weaknesses:** The setup instructions are sparse. There is no explanation of required environment variables, database setup, or initial configuration. The API is not documented; a user must infer it from the source code or example clients. The `docs/` content is promising but incomplete.

### 3. **fleet-simulator**
**Score:** 6/10
**Analysis:**
*   **Strengths:** The repository name and a brief tagline ("Generate multi-agent interaction data") immediately convey its function. The code is well-structured with clear separation between core, agents, and environments.
*   **Weaknesses:** Documentation is nearly absent. The `README.md` is a single line. There are no examples, no guidance on how to define new agent behaviors or simulation scenarios, and no output format specifications. The value is locked inside the source code.

### 4. **constraint-theory-core**
**Score:** 8/10
**Analysis:**
*   **Strengths:** Excellent technical documentation. The `README.md` provides a clear mathematical motivation for geometric constraint solving, defines key terms (Atoms, Bonds, Clusters), and includes a complete, runnable code example. The `docs/` directory contains a detailed `ARCHITECTURE.md` explaining the solver's phases (P0, P1, P2).
*   **Weaknesses:** The documentation is purely technical. It lacks a "why" section connecting it to the larger fleet mission (e.g., its role in the Deadband Protocol or environment modeling). The build/installation notes for the Rust project are minimal.

### 5. **oracle1-vessel**
**Score:** 5/10
**Analysis:**
*   **Strengths:** The repository structure is descriptive (`/from-fleet/`, `/to-fleet/`, `/services/`), which hints at its role as a communication hub. The presence of operational scripts (`service-guard.sh`) is valuable.
*   **Weaknesses:** There is no `README.md` file. The purpose of the vessel, how it integrates with `oracle1` and the fleet, and how to deploy or configure it must be reverse-engineered. This is a critical piece of infrastructure with no guiding documentation.

## Step 2: Summary & Fleet Documentation Health Assessment

**Overall Fleet Documentation Score (Approx.): 7/10**

**Patterns Observed:**
1.  **Bimodal Quality:** Documentation quality is polarized. Foundational libraries (`plato-torch`, `constraint-theory-core`) are well-documented, while application/runtime projects (`holodeck-rust`, `fleet-simulator`, `oracle1-vessel`) are under-documented.
2.  **Strong on "What," Weak on "Why & How":** Technical APIs and concepts are often explained, but the integration narrative—how a specific repo enables the larger "Inhabited Intelligence" vision—is frequently missing.
3.  **Operational Gap:** Documentation for deployment, configuration, and service management (as seen in `oracle1-vessel`) is the most common critical gap.

**P0 Issue (Negative Space):** Do not assume a new contributor can run or integrate a fleet service based on its current documentation alone. This is a blocker for swarm growth.

**P1 Safe Channel:** The existing high-quality docs for core libraries provide a template. The `docs/` folder pattern (used in `holodeck-rust` and `constraint-theory-core`) is a good, scalable practice.

**P2 Optimization Path:** Apply the `plato-torch` README structure (Purpose, Concepts, Quick Start, Deep Dive, API) as a fleet-wide template. Mandate a minimal `README.md` for all repos, enforced via a `/.github` repository template.
