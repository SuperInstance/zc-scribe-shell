## Step 2: Documentation Analysis & Scoring

**Scoring Rubric (0-10):**
*   **0-3:** Missing, severely incomplete, or actively misleading.
*   **4-6:** Basic structure present but lacks critical details (setup, purpose, API). May be outdated.
*   **7-8:** Good. Clear purpose, setup instructions, and basic usage. Could improve with examples, architecture diagrams, or contribution guides.
*   **9-10:** Excellent. Comprehensive, clear, includes examples, diagrams, API docs, and contribution guidelines. Serves as a true entry point.

### 1. plato-torch
**Path:** `fleet-knowledge/repos/plato-torch/README.md`
**Score:** 8/10

**Analysis:**
*   **Strengths:** Excellent high-level explanation of the PLATO room concept ("shells for hermit crab agents"). Clearly defines key components: Rooms, Tiles, Ensigns. Provides a concrete, minimal example of creating a `DeadbandRoom` and submitting a tile. The "Quick Start" section is effective.
*   **Areas for Improvement:** Lacks a detailed "Architecture" or "Design" section explaining how the Torch backend manages room state or interacts with the tile database. No API reference for the `PlatoRoom` base class or the `export_ensign` method. The "Installation" section assumes prior environment setup (Python, PyTorch). A `requirements.txt` or `pyproject.toml` file reference would help.

### 2. fleet-simulator
**Path:** `fleet-knowledge/repos/fleet-simulator/README.md`
**Score:** 6/10

**Analysis:**
*   **Strengths:** States the project's purpose clearly: "A lightweight, event-driven simulator for generating multi-agent training data." Mentions it's written in Rust for performance. Lists the key concepts (Agent, Environment, Event).
*   **Areas for Improvement:** The "Usage" section is a single, abstract code snippet that doesn't show how to define an agent or an environment. There are no concrete examples. No instructions on how to build or run the project (`cargo build`?). No explanation of how the generated data is formatted or where it goes. The documentation feels like a placeholder for a more complete guide.

### 3. holodeck-rust
**Path:** `fleet-knowledge/repos/holodeck-rust/README.md`
**Score:** 7/10

**Analysis:**
*   **Strengths:** Clear, evocative description: "A sentiment-aware MUD server for training narrative agents." Defines its core innovation (sentiment propagation between NPCs). Provides a **good example** of server configuration via TOML, showing NPC definitions and sentiment rules.
*   **Areas for Improvement:** No instructions on how to actually launch the server or connect a client. Lacks a "Getting Started" guide. Does not describe the client protocol (Telnet? WebSocket? Custom TCP?). The API for adding new sentiment rules or NPC behaviors is not documented.

### 4. constraint-theory-core
**Path:** `fleet-knowledge/repos/constraint-theory-core/README.md`
**Score:** 5/10

**Analysis:**
*   **Strengths:** Clearly states the ambitious research goal: "A Rust library implementing geometric constraint solving for real-time spatial reasoning in agent environments." Mentions inspiration from CAD and physics engines.
*   **Areas for Improvement:** The documentation is almost purely conceptual. There is **no code example**, no API overview, and no setup instructions. It does not explain what a "constraint" or "snap" looks like in code. A potential user cannot understand how to *use* the library from this README. It reads like a research abstract, not a software guide.

### 5. cudaclaw
**Path:** `fleet-knowledge/repos/cudaclaw/README.md`
**Score:** 4/10

**Analysis:**
*   **Strengths:** The title and tagline are clear: "CUDA-resident agent runtime." It establishes the high-performance premise.
*   **Areas for Improvement:** The README is extremely sparse. It lacks any details on architecture (how are agents resident in GPU memory?). There is no example, no API, and no build instructions. Key questions are unanswered: What is the interface for loading an agent? How is inference triggered? How does data move between host and device? The documentation is insufficient for anyone to evaluate or use the project.

## Step 3: Summary Report

**Overall Fleet Documentation Health: Variable (Average Score: 6.0/10)**

**Patterns Observed:**
1.  **Conceptual Clarity Over Practical Detail:** Most repos (`plato-torch`, `holodeck-rust`, `constraint-theory-core`) do a good job explaining the *why* and the high-level *what*, but often lack the detailed *how*.
2.  **Missing "Getting Started" Guides:** Only `plato-torch` provides a clear, copy-pastable path to a first result. Others lack build/run instructions.
3.  **Example Gap:** Concrete, working examples are the single biggest differentiator between higher and lower scores. `plato-torch`'s example directly contributes to its high score.
4.  **Infrastructure vs. Tools:** Core infrastructure (`plato-torch`) is better documented than specialized tools (`fleet-simulator`, `cudaclaw`) and research prototypes (`constraint-theory-core`).

**Recommendations (P1 - Safe Channels for Improvement):**
*   **Mandatory README Sections:** Propose a fleet template requiring: "Purpose", "Quick Start (with a working example)", "Building/Running", "API Overview", "Contributing".
*   **Priority Fixes:** `cudaclaw` and `constraint-theory-core` are the highest priority for documentation improvement, as their current state blocks understanding and contribution.
*   **Example-First Development:** Encourage developers to write the minimal usage example before any other code, ensuring the public API is clear and documentable.
