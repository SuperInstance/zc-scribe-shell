*   **7-8:** Good documentation. Clear purpose, setup, usage examples, and links to related projects.
*   **9-10:** Excellent. Includes vision, architecture, detailed API, contribution guidelines, and is actively maintained.

## Step 3: Summary Report

**Overall Fleet Documentation Health: Below Target.**

*   **Average Score:** 5.2/10
*   **Trend:** Core, actively developed systems (`plato-torch`) have good documentation. Supporting libraries, research prototypes, and infrastructure components have minimal or placeholder documentation.
*   **Primary Gap:** **Actionable Examples.** Most repos lack runnable code snippets or tutorials that show how to *use* the software.
*   **Secondary Gap:** **Architectural Context.** Documentation rarely explains how a component fits into the larger Cocapn fleet ecosystem (P0/P1/P2 doctrine, tile flow, room training).
*   **Risk:** This creates a high barrier to entry for new contributors (ensigns, greenhorns) and increases the bus factor. Knowledge remains siloed.

**Recommendations (P1 - Find Safe Channels):**
1.  **Establish a Documentation Standard:** A template for `README.md` (Purpose, Quick Start, API, Examples, Architecture Context).
2.  **Prioritize "Getting Started" Guides:** For each repo, the first priority should be a single script or command that demonstrates core functionality.
3.  **Create Cross-Reference Maps:** In each README, add a "**Fleet Context**" section that links to upstream/downstream dependencies (e.g., `fleet-simulator` README should link to `plato-torch` data ingestion).
4.  **Initiate a "Documentation Sprint":** Target the lowest-scoring repos (`flux-runtime`, `constraint-theory-core`) first to bring them to a baseline of usability.
