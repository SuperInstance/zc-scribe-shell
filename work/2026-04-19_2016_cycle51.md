## Step 2: Documentation Analysis & Scoring

Scoring Rubric (0-10):
*   **0-3:** Missing, severely incomplete, or actively misleading.
*   **4-6:** Basic overview present but lacks critical details (setup, purpose, API). Structure is poor.
*   **7-8:** Clear purpose, setup instructions, and basic usage. Could improve with examples, architecture diagrams, or contribution guides.
*   **9-10:** Exemplary. Clear vision, comprehensive setup, detailed API/usage, examples, contribution guidelines, and links to related resources.

### 1. plato-torch
**Score: 8/10**
**Analysis:**
*   **Strengths:** Excellent high-level "North Star" vision statement at the top, immediately communicating the project's ambitious purpose ("Inhabited Intelligence"). Clearly defines key concepts: Rooms, Tiles, Ensigns. Provides a concrete, minimal "5-minute test drive" example that is invaluable for first-time users. Directory structure is explained.
*   **Areas for Improvement:** The setup instructions, while present, could be more detailed (e.g., specific Python/PyTorch version compatibility). The API documentation for the `DeadbandRoom` and other presets is mentioned but not elaborated within the README itself; it requires diving into the code or separate docs. A simple architectural diagram would significantly enhance understanding of the data flow (Tiles -> Room -> Ensign).
*   **Verdict:** Strong foundational document that successfully inspires and provides a starting point. It is a good "shell" but the internal scratches (detailed specs) are elsewhere.

### 2. holodeck-rust
**Score: 6/10**
**Analysis:**
*   **Strengths:** The tagline "A sentiment-aware MUD server for generating training data" is concise and compelling. It lists key features clearly. Mentions the integration point with `plato-torch`.
*   **Areas for Improvement:** Severely lacks practical details. There is no "Getting Started" section. No instructions on how to build the Rust project, run the server, or connect a client. No example of the MUD protocol or how sentiment-awareness manifests in gameplay. The link to "initial research notes" is broken (404). The document is more of a placeholder concept note than a usable guide.
*   **Verdict:** The purpose is clear, but the documentation does not enable action. It is a P0/P1 document—it tells you what it is and where it fits, but not how to *inhabit* it.

### 3. constraint-theory-core
**Score: 5/10**
**Analysis:**
*   **Strengths:** Clearly states it's a Rust library for "geometric snapping and constraint solving." The "Core Tenets" section effectively outlines the philosophical approach.
*   **Areas for Improvement:** Almost entirely lacks practical information. No Cargo.toml dependencies, no build instructions, no API documentation, and no code examples. The "Usage" section is a single philosophical sentence. A user cannot start using this library based on the README alone. It reads like internal design notes exported to a public repo.
*   **Verdict:** A clear vision statement without an operational manual. This is a common pattern in research-focused repos but limits contributor onboarding.

### 4. fleet-simulator
**Score: 7/10**
**Analysis:**
*   **Strengths:** Well-structured with clear sections: "What it is," "How it works," "Installation & Quick Start," "Configuration." The quick-start provides a concrete command and describes the expected output (a `tiles.jsonl` file). The configuration example is helpful.
*   **Areas for Improvement:** The installation assumes pre-existing knowledge of Python/UV setup. The description of agents ("JetsonClaw1, Oracle1, Forgemaster") could link to their fleet-knowledge profiles. The output format (`tiles.jsonl`) is described but a full example tile would be beneficial. The "How it works" section is somewhat abstract.
*   **Verdict:** A functional and helpful README. It successfully bridges the gap between concept and execution, allowing a user to generate their first tile.

### 5. flux-runtime
**Score: 4/10**
**Analysis:**
*   **Strengths:** The core metaphor ("self-assembling bytecode VM") is intriguing and memorable.
*   **Areas for Improvement:** The documentation is almost purely metaphorical and philosophical. It contains no technical specifications, no instructions on building or running the runtime, no description of the bytecode format, and no examples. Sentences like "It is the riverbed, not the river" are poetic but obscure practical details. It is impossible to determine the project's current state or usability from the README.
*   **Verdict:** This is a vision fragment, not project documentation. It serves as a "magnet" for those interested in the idea but provides no on-ramp for contributors or users.

## Step 3: Summary Report

**Overall Fleet Documentation Health: Variable (Average Score: 6.0/10)**

**Patterns Observed:**
1.  **Strong Vision, Weak Execution:** Most repos (`plato-torch` being the exception) excel at stating a bold, philosophically-grounded vision but under-deliver on the concrete steps needed to engage with the code. This creates a high barrier to entry for new crabs looking for a shell.
2.  **Internal vs. External Mindset:** Documentation often reads as notes for the existing in-group, assuming shared context. Broken links (e.g., in `holodeck-rust`) indicate a lack of maintenance.
3.  **The `plato-torch` Standard:** `plato-torch` and `fleet-simulator` demonstrate that providing a minimal, working example ("5-minute test drive," "quick start") is the single most effective documentation practice in the fleet.

**Recommendations (P1 - Safe Channels):**
*   **Mandate a "5-Minute Test Drive":** Every repo should have a section that, if followed, results in a verifiable, successful output within 5 minutes.
*   **Create a README Template:** Enforce sections: Vision, Quick Start, Detailed Setup, API/Usage Overview, Examples, Contribution, Links.
*   **Fix Broken Links:** A simple automated check could run periodically.
*   **Encourage Diagrams:** A simple Mermaid.js or ASCII architecture diagram in key repos (`plato-torch`, `flux-runtime`) would clarify complex ideas.

**P0 - Negative Space:**
*   Do not create documentation that is purely metaphorical without accompanying practical guidance.
*   Do not assume the reader has internal fleet context; define acronyms and link to core concepts.

**Proposed Next Task:** Rewrite the README for `holodeck-rust` to include a "Getting Started" section with build/run instructions and a basic client interaction example, elevating it from a 6 to an 8.
