## Step 1: Repository Selection & Analysis

I have selected 5 repositories from the fleet's public GitHub organization (`cocapn`). The selection was random, focusing on a mix of core infrastructure, tools, and research projects to get a representative sample of documentation quality.

### 1. **Repository:** `cocapn/plato-torch`
**Score:** 8/10
**Analysis:**
*   **Strengths:** The `README.md` is comprehensive. It clearly states the project's purpose ("PLATO room server for training AI agents"), lists prerequisites (Python 3.10+, CUDA 12.1), and provides a complete, step-by-step installation guide (`pip install -e .`). It documents the core concepts (Rooms, Tiles, Ensigns) and includes a "Quick Start" section with example code to run a room. The API for `DeadbandRoom` is partially documented.
*   **Weaknesses:** The API documentation within the README is not exhaustive. It references `presets/` but does not list or describe all 25 training room presets. There is no dedicated `CONTRIBUTING.md` file. Advanced configuration and the `plato-ensign` export process are mentioned but not detailed.

### 2. **Repository:** `cocapn/holodeck-rust`
**Score:** 6/10
**Analysis:**
*   **Strengths:** The `README.md` has a strong, clear tagline: "A sentiment-aware MUD server for AI agent socialization and narrative generation." It explains the core idea (NPCs with emotional state) and lists key features. A basic "Getting Started" section includes commands to clone, build, and run the server.
*   **Weaknesses:** The installation instructions are minimal ("`cargo build --release`"). There is no guidance on dependencies or system requirements. There is no example of how to connect an agent to the MUD, no API specification for interacting with NPCs, and no explanation of the "sentiment" model or how to configure worlds. The documentation assumes prior knowledge of MUDs.

### 3. **Repository:** `cocapn/fleet-simulator`
**Score:** 4/10
**Analysis:**
*   **Strengths:** The repository name is descriptive. The `README.md` correctly identifies its purpose: "Generates training data through multi-agent fleet simulations."
*   **Weaknesses:** The documentation is severely lacking. The README is essentially a placeholder. It contains no installation instructions, no explanation of how to configure a simulation, no example of the output data format, and no API for integrating with other fleet components (like `plato-torch`). A new contributor would not know how to use this tool.

### 4. **Repository:** `cocapn/constraint-theory-core`
**Score:** 7/10
**Analysis:**
*   **Strengths:** Good conceptual documentation. The `README.md` explains the problem domain ("geometric snapping in CAD for AI") and the core theory (P0/P1/P2 constraints). It links to foundational research papers. The code structure is clean, and Rust doc comments are present in key source files (e.g., `src/lib.rs`).
*   **Weaknesses:** Practical usage documentation is sparse. There is no "Quick Start" example showing how to use the library in a Rust project. The build instructions are just "`cargo build`." There are no examples of defining or solving a constraint network. The leap from theory to implementation is not bridged for a user.

### 5. **Repository:** `cocapn/service-guard`
**Score:** 9/10
**Analysis:**
*   **Strengths:** Exemplary tool documentation. The `README.md` is precise and action-oriented. It defines the problem (keeping fleet services alive), lists the 5 protected services, and provides clear, copy-paste commands for installation (`make install`), configuration (editing `config.toml`), and operation (`sudo systemctl start service-guard`). It includes troubleshooting steps and a log location.
*   **Weaknesses:** Very minor. Could include an example of a custom service configuration or a diagram of the monitoring lifecycle. The score is nearly top-tier because it successfully enables a user to deploy and use the tool without external help.

## Step 2: Summary Report & Recommendations

**Overall Fleet Documentation Health: Variable (Average Score: 6.8/10)**

The fleet exhibits a clear split between **well-documented tools** (`service-guard`, `plato-torch`) and **under-documented research/experimental projects** (`fleet-simulator`, `holodeck-rust`). Core libraries like `constraint-theory-core` fall in the middle, with strong theory but weak practical guides.

**Patterns Observed:**
1.  **Operational tools** have the best documentation. They are designed for immediate use.
2.  **Research/Simulation projects** often have placeholder READMEs, treating the code as its own documentation.
3.  **Conceptual explanation** is generally stronger than **practical API/usage examples**.

**P1 (Safe Channel) Recommendations for Scribe:**
1.  **Immediate Action (Backlog Task):** Target the lowest-scoring repo, `fleet-simulator`. Create a basic but functional `README.md` with: a "Quick Start" example, a description of input/output data, and a link to a more detailed `FLEET-RESEARCH.md` file to be created later.
2.  **Template Creation:** Develop a fleet `README-TEMPLATE.md` that enforces sections: Purpose, Prerequisites, Installation, Quick Start, API/Usage, and Links to deeper docs (`FLEET-RESEARCH.md`, `ARCHITECTURE.md`).
3.  **Phase 2 Alignment:** The "Rewrite 3 fleet READMEs" task should focus on `holodeck-rust`, `fleet-simulator`, and `constraint-theory-core` to bring them up to the standard of `service-guard`.

**P2 (Optimization) Future Consideration:**
*   Implement a documentation linter in CI that checks for required README sections.
*   Encourage the use of Rust `cargo doc` / Python `pdoc` to auto-generate API references and link them from the README.
