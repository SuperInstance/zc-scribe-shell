## Step 1: Repository Selection & Scoring

I will select 5 repositories from the fleet's known list and evaluate their primary documentation (README.md) based on the following criteria derived from fleet doctrine and common best practices:
1.  **Clarity of Purpose (P0):** Does it immediately state what the project is and why it exists?
2.  **Actionable Setup (P1):** Does it provide clear, executable steps to install, configure, and run?
3.  **Usage & Examples (P2):** Does it show how to use the tool effectively with examples?
4.  **Fleet Context:** Does it connect to other fleet components (plato-torch, holodeck, etc.)?
5.  **Structure & Navigation:** Is it well-organized with headers, lists, and links?

**Scoring Scale:** 0 (Nonexistent/Broken) to 10 (Exemplary/Fleet Standard).

---

### 1. Repository: `plato-torch`
**Score: 9/10**
**Analysis:**
*   **P0 (Purpose):** Excellent. Opens with a strong, clear thesis: "PLATO rooms are persistent training environments for AI agents." Immediately defines key terms (Room, Tile, Ensign).
*   **P1 (Setup):** Very good. Provides `pip install` command and a concise "Quick Start" example that creates a room and submits a tile. Assumes Python/`git` knowledge, which is appropriate for the fleet.
*   **P2 (Usage):** Strong. Documents the core `Room` and `Tile` classes, shows how to list presets, and includes a more advanced example. Could be enhanced with a diagram of the data flow (Room <-> Tile <-> Ensign).
*   **Fleet Context:** Explicitly mentions connections to `plato-ensign` and the concept of training rooms for the fleet. Perfect.
*   **Structure:** Clean, uses code blocks, headers, and links. A table of presets would be a minor improvement.
**Verdict:** Near-perfect fleet documentation. Serves as a model.

### 2. Repository: `holodeck-rust`
**Score: 7/10**
**Analysis:**
*   **P0 (Purpose):** Good. "A sentiment-aware MUD server for generating training data." Clear and aligns with fleet's simulation goals.
*   **P1 (Setup):** Fair. Requires Rust (`cargo build`). Instructions are present but minimal. Lacks a "quick start" to immediately see a running server and connected client.
*   **P2 (Usage):** Limited. Describes the architecture (Server, Zone, NPC, Player) but has no example of launching the server, connecting a telnet client, or interacting with an NPC to see sentiment in action. The most valuable part is the description of the "sentiment lattice."
*   **Fleet Context:** Mentions being a data source for `fleet-simulator` and `plato-torch`. Good.
*   **Structure:** Adequate but dense. Could use more sub-headers and a clear "Getting Started" section.
**Verdict:** Solid on concept and architecture, but weak on practical, copy-paste usability. A new contributor would struggle to run it.

### 3. Repository: `constraint-theory-core`
**Score: 4/10**
**Analysis:**
*   **P0 (Purpose):** Poor. The README is essentially a placeholder. It states "Geometric constraint solving for CAD-like snapping" but does not explain *why* this is needed for the fleet (e.g., for `flux-runtime` self-assembly, spatial reasoning in `fleet-simulator`).
*   **P1 (Setup):** Missing. No installation or build instructions.
*   **P2 (Usage):** Nonexistent. No API documentation, examples, or tutorials.
*   **Fleet Context:** Absent. No mention of how this library integrates with or enables other fleet projects.
*   **Structure:** Skeletal.
**Verdict:** This is a critical infrastructure library with effectively no documentation. It represents a major knowledge gap and onboarding barrier.

### 4. Repository: `fleet-simulator`
**Score: 6/10**
**Analysis:**
*   **P0 (Purpose):** Clear. "A discrete-event simulator for modeling fleet interactions and generating training tiles."
*   **P1 (Setup):** Basic. `npm install` and `npm start` are provided. However, it lacks guidance on configuration (e.g., how many agents, what scenario).
*   **P2 (Usage):** Partial. Shows a basic example of creating a `Simulation` and `Agent`, but doesn't show how to define agent behaviors, interactions, or, most importantly, how the output connects to a PLATO room to generate tiles. The link between simulation and training is implied but not demonstrated.
*   **Fleet Context:** Explicitly states it generates tiles for `plato-torch`. This is its strongest point.
*   **Structure:** Okay. Could use a visual flowchart of the simulation loop.
**Verdict:** Documents the "what" and the "why," but the "how" is incomplete. The crucial integration path (simulation -> tiles) is not illustrated.

### 5. Repository: `cudaclaw`
**Score: 3/10**
**Analysis:**
*   **P0 (Purpose):** Vague. "GPU-resident agent runtime." This is technically descriptive but fails to explain the problem it solves (e.g., "for low-latency inference of ensigns" or "to keep agent state in VRAM").
*   **P1 (Setup):** Very Poor. Heavily assumes existing CUDA/CMake expertise. Instructions are fragmented and likely to fail on most systems due to missing dependency details.
*   **P2 (Usage):** None. No example of launching an agent, loading a model, or performing inference.
*   **Fleet Context:** None. Does not position itself within the fleet's GPU stack (JetsonClaw1, Forgemaster).
*   **Structure:** Confusing. Mixes high-level goals with low-level build troubleshooting.
**Verdict:** The documentation is a barrier to entry. It reads like internal notes, not onboarding material. This is a high-priority fix for a performance-critical component.
