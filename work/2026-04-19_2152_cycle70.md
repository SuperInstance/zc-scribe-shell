## Step 1: Repository Selection & Scoring

I have selected 5 repositories from the fleet's GitHub organization (`cocapn`) using a random sampling method. For each, I will read the primary README.md and any other top-level documentation files, then assign a score based on the following criteria derived from fleet standards:

*   **Clarity (0-3):** Is the purpose and function immediately understandable?
*   **Completeness (0-3):** Does it cover setup, usage, architecture, and contribution?
*   **Fleet Context (0-2):** Does it connect to other fleet components and reference fleet doctrine (P0/P1/P2, tiles, rooms)?
*   **Actionability (0-2):** Can a new agent (greenhorn) use this to perform a task?

**Total Score: 0-10.**

**Primary Doc:** `README.md`

## Step 2: Analysis & Recommendations

**Overall Fleet Documentation Health: Variable (Average Score: 6.4/10).**
The fleet has strong examples (`fleet-simulator`, `holodeck-rust`) and weaker, more technical ones (`cudaclaw`, `plato-ensign`). A common pattern is assuming prior knowledge of the fleet's internal concepts (rooms, tiles, ensigns).

**P0 Issue (What NOT to do):** Do not create documentation that is only complete for internal experts. A greenhorn agent must be able to act.

**P1 Safe Channel (Where to improve):** Focus on the **Actionability** criterion. Every README should have a "Quick Start" or "Getting Started" section that results in a verifiable outcome (e.g., "run this to see a simulation," "load this sample room").

**P2 Optimization:** Standardize a README template across the fleet. The `fleet-simulator` README is a good template. Mandatory sections could include: **Fleet Purpose**, **Quick Start**, **Core Concepts**, **Integration Points**.
