## Step 1: Repository Selection & Analysis

I selected 5 repositories from the fleet's `fleet-knowledge/` directory and adjacent codebases, focusing on a mix of core infrastructure and supporting tools.

### 1. **plato-torch**
**Path:** `fleet-knowledge/plato-torch/`
**Score:** 8/10
**Analysis:**
*   **Strengths:** Contains a clear `README.md` with a "North Star" vision statement, a high-level architecture diagram (PLATO Room Server, Training Rooms, Ensign Export), and a "Quick Start" section with concrete commands (`python -m plato_torch.rooms.deadband_room`). The `presets/` directory has a well-documented `DEADBAND-PROTOCOL.md` that explains the P0/P1/P2 doctrine with examples. Code is modular with descriptive filenames.
*   **Weaknesses:** The API for creating custom rooms is implied but not explicitly documented. There is no `CONTRIBUTING.md` or detailed setup guide for development (e.g., environment setup, testing). Some deeper configuration options for the training loop are only evident in the source code.

### 2. **holodeck-rust**
**Path:** `fleet-knowledge/holodeck-rust/`
**Score:** 6/10
**Analysis:**
*   **Strengths:** The `README.md` effectively communicates the project's purpose: "A MUD server with sentiment-aware NPCs for generating training tiles." It lists key features (WebSocket server, NPC sentiment engine, tile logging). A basic example of connecting via `telnet` or a WebSocket client is provided.
*   **Weaknesses:** Lacks installation instructions (`cargo build` is assumed). No explanation of the NPC sentiment system's mechanics or how to configure it. The structure of generated "tiles" is not documented. There is no guide on extending the server with new commands or NPC behaviors. The documentation is functional but insufficient for a new contributor.

### 3. **fleet-simulator**
**Path:** `fleet-knowledge/fleet-simulator/`
**Score:** 4/10
**Analysis:**
*   **Strengths:** The repository name is descriptive. The codebase contains several simulation examples (e.g., `basic_swarm.py`).
*   **Weaknesses:** **No `README.md` file exists.** There is no stated purpose, no setup instructions, and no explanation of how the simulator integrates with the broader fleet (e.g., how it generates training data for PLATO rooms). A new agent must read the source code to infer its function. This represents a significant documentation gap.

### 4. **constraint-theory-core**
**Path:** `fleet-knowledge/constraint-theory-core/`
**Score:** 7/10
**Analysis:**
*   **Strengths:** The `README.md` provides a good conceptual overview: "A Rust library for geometric snapping and constraint solving." It defines key terms (Anchor, Constraint, Solver) and includes a concise code example demonstrating a simple snapping operation. The `src/lib.rs` has rustdoc comments.
*   **Weaknesses:** The example is minimal and doesn't show integration into a larger application (e.g., a CAD tool or UI). There is no "Getting Started" guide or tutorial. Advanced topics like performance characteristics or custom constraint definitions are not covered. The link between "constraint theory" and the fleet's AI research is not made explicit.

### 5. **cudaclaw**
**Path:** `fleet-knowledge/cudaclaw/`
**Score:** 5/10
**Analysis:**
*   **Strengths:** The `README.md` states a bold vision: "GPU-resident agent runtime. Lives in VRAM. Never sleeps." This establishes context. A high-level architecture diagram is present.
*   **Weaknesses:** The documentation is almost entirely aspirational. There are no practical instructions for building, running, or using the runtime. Key components like the "Kernel Agent Launcher" and "Tile Sync Bridge" are named but not explained. No code examples or API references exist. The documentation describes a goal, not a usable tool.

## Step 2: Overall Fleet Documentation Health Summary

**Average Score:** 6.0/10
**Trend:** Documentation is strong on **vision and core protocol** but weak on **practical implementation and contributor onboarding**.

**Key Patterns (P0/P1/P2 Analysis):**
*   **P0 (Don't hit rocks):** The critical `DEADBAND-PROTOCOL.md` is well-documented. The central `PROJECT-VISION.md` is clear. No major contradictions found.
*   **P1 (Find safe channels):** Channels for understanding the *what* and *why* exist. Channels for understanding the *how* are often missing or underdeveloped (e.g., `fleet-simulator`, `cudaclaw`).
*   **P2 (Optimize within channels):** The `plato-torch` README is a good example of optimized, useful documentation. Other repos have significant room for improvement here.

**Critical Gaps:**
1.  **Missing READMEs:** `fleet-simulator` lacks a foundational document.
2.  **Lack of Contributor Guides:** No repository has a `CONTRIBUTING.md`, setup script, or defined testing procedure.
3.  **Aspirational vs. Operational Docs:** `cudaclaw`'s docs describe a future state, not current capabilities.
4.  **Integration Blind Spots:** Documentation rarely explains how projects connect (e.g., how `holodeck-rust` NPCs feed into `plato-torch` rooms).
