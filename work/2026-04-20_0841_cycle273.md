## Step 1: Repository Selection & Analysis

I selected 5 repositories from the fleet's `fleet-knowledge/` directory and adjacent project folders. The selection was random, aiming for a mix of core infrastructure, tools, and research.

### 1. **holodeck-rust**
**Path:** `fleet-knowledge/projects/holodeck-rust/`
**Score:** 8/10
**Analysis:**
*   **Strengths:** Excellent `README.md` with clear purpose ("A sentiment-aware MUD server for training embodied agents"), quick start instructions, and a detailed architecture diagram. The `docs/` folder contains comprehensive guides on NPC scripting, sentiment systems, and protocol definitions. Code is well-commented.
*   **Weaknesses:** The API reference in `docs/api.md` is partially stubbed. Some advanced configuration options are only documented in example `.toml` files without a central guide.
*   **Recommendation:** Complete the API reference stubs. Add a "Configuration" section to the main README linking to the example files.

### 2. **constraint-theory-core**
**Path:** `fleet-knowledge/research/constraint-theory-core/`
**Score:** 6/10
**Analysis:**
*   **Strengths:** The `THEORY.md` file is a strong piece of research documentation, clearly explaining the P0/P1/P2 deadband protocol and its geometric analogy. The core thesis is well-articulated.
*   **Weaknesses:** There is no `README.md` in the project root. The `src/` directory lacks code-level documentation. There are no usage examples or instructions on how to integrate the library. The project feels like a research paper without an engineering interface.
*   **Recommendation:** Create a basic `README.md` with a one-line description, a link to `THEORY.md`, and a "Getting Started" example showing how to use the snapping functions from `src/snapping.rs`.

### 3. **plato-ensign**
**Path:** `fleet-knowledge/projects/plato-ensign/`
**Score:** 7/10
**Analysis:**
*   **Strengths:** README clearly defines its role ("Exports room experience into LoRA adapters"). It has a good, concise example of the export workflow and lists the key Python dependencies. The link to the parent `plato-torch` project is present.
*   **Weaknesses:** Lacks detail on the adapter format, configuration options for the export (e.g., rank, alpha), and how to *use* the generated LoRA after export. No information on how it interacts with the tile server or room state.
*   **Recommendation:** Add a "Output Format" section describing the `.safetensors` file structure. Add a "Usage" section showing how to load the adapter with PyTorch or `peft`. Document any required environment variables for server integration.

### 4. **fleet-simulator**
**Path:** `fleet-knowledge/projects/fleet-simulator/`
**Score:** 5/10
**Analysis:**
*   **Strengths:** The `README.md` has a compelling vision statement about generating multi-agent training data. The repository structure is clean (`/agents`, `/environments`, `/scripts`).
*   **Weaknesses:** The README's "Quick Start" section is a placeholder (`# TODO: docker command`). There are no installation instructions. The `agents/` and `environments/` directories contain code but no documentation on how to define a new agent or environment. Key concepts like "scene" and "orchestrator" are undefined.
*   **Recommendation:** Fulfill the TODO with a minimal Docker or `uv` run command. Add a `docs/` folder with a basic tutorial on creating a simple agent and running a simulation.

### 5. **service-guard**
**Path:** `fleet-knowledge/tools/service-guard.sh`
**Score:** 9/10
**Analysis:**
*   **Strengths:** This is a model of excellent tool documentation. The script itself is heavily commented at the top with a clear "PURPOSE," "USAGE," and "SERVICES" section. Each service has a defined health check command and restart logic. It's practical and immediately usable.
*   **Weaknesses:** The script is a single file; there's no associated `README.md` or markdown doc in a parent `tools/` directory. This could make it harder to discover for someone browsing the repo structure.
*   **Recommendation:** Create a `tools/README.md` that lists `service-guard.sh` with a brief description and a link. Consider adding a "Configuration" section to the script's header comments for advanced users.

## Step 2: Overall Fleet Documentation Assessment

**Average Score:** 7.0/10
**Trend:** Documentation quality is inconsistent but has high peaks. Core tools (`service-guard`) and flagship projects (`holodeck-rust`) are well-documented. Research projects and newer/simulator codebases suffer from incomplete "getting started" guides and lack of practical examples.

**Common Pattern:** Many repos articulate the **"why"** (vision/theory) well but are weak on the **"how"** (concrete steps to run/use/extend). This creates a barrier for new contributors (greenhorns) during the "Read & Understand" phase of boot camp.
