## Step 1: Repository Selection & Scoring

I have selected 5 repositories from the fleet's GitHub organization (`Cocapn`). The selection is based on a mix of core tools, research projects, and utilities to get a representative sample.

### 1. **plato-torch**
**Score: 8/10**
**Analysis:**
*   **Strengths:** The README.md is comprehensive. It clearly defines the project's purpose ("PLATO rooms for AI agents"), lists key features (25 training room presets, self-training), and provides a quick start guide with installation commands (`pip install plato-torch`). It includes a well-structured "Architecture" section explaining Rooms, Tiles, and Ensigns, which aligns with the core fleet doctrine.
*   **Weaknesses:** The API documentation within the README is high-level. A user would need to explore the source code or dedicated docstrings to understand class methods and parameters in detail. A link to more detailed API docs or examples would improve the score.
*   **File Check:** `README.md` exists and is substantial (~120 lines).

### 2. **holodeck-rust**
**Score: 6/10**
**Analysis:**
*   **Strengths:** The README.md has a strong, clear tagline: "A sentiment-aware MUD server for AI agents." It explains the core concept (NPCs with emotional state) and provides a `docker-compose up` command for easy local deployment. The inclusion of a "Development" section is good.
*   **Weaknesses:** It lacks a "Quick Start" or "Usage" section explaining how an agent *interacts* with the MUD. What is the connection endpoint? What protocol does it use (Telnet, WebSocket)? There are no example interactions or API calls. The documentation assumes prior MUD knowledge.
*   **File Check:** `README.md` exists. No `docs/` folder observed in the root.

### 3. **fleet-simulator**
**Score: 4/10**
**Analysis:**
*   **Strengths:** The repository name is descriptive. The README.md states its purpose: "A fleet simulation environment for generating training data."
*   **Weaknesses:** The documentation is minimal. It does not explain how to configure a simulation, what the output data format is, or how it integrates with other fleet components (like `plato-torch`). There are no usage examples or setup instructions beyond listing Python as a prerequisite. This creates a high barrier to entry.
*   **File Check:** `README.md` is very short (~15 lines). No other documentation files are present.

### 4. **constraint-theory-core**
**Score: 7/10**
**Analysis:**
*   **Strengths:** This is a Rust library, and its `README.md` follows good Rust crate conventions. It has a clear "Usage" section with code snippets showing how to add it as a dependency and import modules. The "What is Constraint Theory?" section provides necessary conceptual context.
*   **Weaknesses:** The API documentation is primarily left to `cargo doc`. The README does not illustrate a complete, practical example of solving a constraint problem. The link to "the main theory write-up" is a relative path (`/docs/theory.pdf`) which may not resolve correctly on GitHub.
*   **File Check:** `README.md` exists. `docs/` directory exists but appears to contain theoretical PDFs, not API docs.

### 5. **service-guard**
**Score: 9/10**
**Analysis:**
*   **Strengths:** Excellent, operational documentation. The `README.md` is task-oriented: it immediately states what it does ("orchestrates core fleet services") and lists the specific services it manages. It provides clear, copy-paste commands for installation (`make setup`), starting (`make start`), and checking status (`make status`). The "Failure Modes" section is particularly valuable, outlining specific symptoms and recovery steps.
*   **Weaknesses:** Very minor. Could include a diagram of service dependencies.
*   **File Check:** `README.md` is thorough. Also contains a `TOOLS.md` with extended tool descriptions.

## Step 2: Summary Report & Recommendations

**Overall Fleet Documentation Health: Variable.** Scores range from 4 to 9, indicating inconsistent documentation practices across projects.

**Common Pattern:** READMEs are present and usually describe the *what* and sometimes the *why*, but often lack clear, actionable *how* for new users or integrating agents.

**P0 Recommendation (Blocking Issue):** Address the `fleet-simulator` (Score 4) documentation gap first. It is a core data generation component, and poor documentation blocks its use and contribution. A minimal fix should include a setup script, a configuration example (e.g., `config.example.yaml`), and a description of the simulation output.

**P1 Recommendation (Safe Channel):** Establish a fleet-wide documentation template in `fleet-knowledge/docs/`. This template should mandate sections: **Purpose**, **Quick Start (with a working command)**, **Architecture/Key Concepts**, **API/Usage Examples**, and **Integration with Fleet**.

**P2 Recommendation (Optimization):** For higher-scoring repos like `plato-torch` and `constraint-theory-core`, enhance documentation by adding a `docs/` directory with detailed API examples and tutorial notebooks that demonstrate integration pathways (e.g., "Generating tiles from a simulator run and loading them into a PLATO room").
