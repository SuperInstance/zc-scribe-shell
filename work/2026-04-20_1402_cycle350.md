## Step 1: Repository Selection & Analysis

I selected 5 repositories from the fleet's GitHub organization using a random approach, focusing on a mix of core infrastructure and supporting tools. I read the primary README.md file for each and evaluated based on the following criteria derived from fleet standards:
*   **Clarity & Purpose (2 pts):** Is the project's goal immediately clear?
*   **Setup & Usage (3 pts):** Are installation and basic run commands provided and accurate?
*   **Architecture Context (2 pts):** Does it explain how it fits into the larger fleet ecosystem?
*   **Code Examples (2 pts):** Are there practical examples of API calls or usage?
*   **Maintenance Status (1 pt):** Is it clear if the project is active, experimental, or deprecated?

### 1. **plato-torch**
**Score: 9/10**
*   **Clarity & Purpose:** Excellent. Opens with a clear, bold thesis: "PLATO rooms are shells. Agents are crabs. Tiles are scratches." Directly connects to the core vision.
*   **Setup & Usage:** Very good. Provides a concise `pip install` command and a minimal example to instantiate a `DeadbandRoom`. Includes a link to detailed API documentation.
*   **Architecture Context:** Strong. Explicitly states its role as the "training room server" and mentions key related projects (`plato-ensign`, `holodeck-rust`).
*   **Code Examples:** Good. The minimal example is functional. The README references 25 preset rooms and self-training rooms, inviting deeper exploration.
*   **Maintenance Status:** Implied by detailed content and linked docs. Could explicitly state version or stability.

### 2. **fleet-simulator**
**Score: 7/10**
*   **Clarity & Purpose:** Good. Described as a "modular simulation framework for generating multi-agent training data." Purpose is clear.
*   **Setup & Usage:** Adequate. Provides `cargo run` instructions. Lacks a quick "hello world" example of launching a simple simulation scenario.
*   **Architecture Context:** Good. Mentions it generates tiles for `plato-torch` and can integrate with `holodeck-rust` for sentiment-driven NPCs.
*   **Code Examples:** Weak. No code snippets in the README. The user must navigate to the `examples/` directory.
*   **Maintenance Status:** Clear. File structure and recent commit history (viewed via git) indicate active development.

### 3. **constraint-theory-core**
**Score: 6/10**
*   **Clarity & Purpose:** Fair. Describes itself as a "geometric constraint solver" but the specific problem domain (e.g., UI layout, CAD snapping) is not immediately obvious from the README.
*   **Setup & Usage:** Good. Clear `cargo add` instruction and a basic example of defining points and a distance constraint.
*   **Architecture Context:** Poor. Does not explain its role within the fleet. A link or mention of a consuming project (e.g., a UI tool) would help.
*   **Code Examples:** Good. The provided Rust example compiles and demonstrates the core API.
*   **Maintenance Status:** Implied by being a core library. The example works.

### 4. **cudaclaw**
**Score: 5/10**
*   **Clarity & Purpose:** Fair. Tagline is "GPU-resident agent runtime." Technically accurate but abstract. The "why" (performance, latency) is not emphasized.
*   **Setup & Usage:** Poor. Lists complex CUDA and build system prerequisites but lacks a definitive "here's how to build and run a test" sequence.
*   **Architecture Context:** Minimal. Mentions it's used by "high-frequency agents" but doesn't name which ones.
*   **Code Examples:** None in README. Points to `src/kernels/` which is intimidating for a first-time evaluator.
*   **Maintenance Status:** Unclear. The README feels like a placeholder. The codebase is substantial, suggesting it's important but under-documented.

### 5. **oracle1-vessel**
**Score: 8/10**
*   **Clarity & Purpose:** Very Good. Clearly states it is the "cloud lighthouse" service for the fleet, handling tile ingestion, validation, and room state management.
*   **Setup & Usage:** Good. Provides Docker commands for local deployment and mentions environment variables for configuration.
*   **Architecture Context:** Excellent. Has a dedicated "Architecture" section with a diagram showing flow from agents to PLATO rooms.
*   **Code Examples:** Adequate. No code snippets, but the operational commands (Docker, curl examples for API) are practical.
*   **Maintenance Status:** Clear. Documented as the primary cloud service.

## Step 2: Summary & Recommendations

**Overall Fleet Documentation Health: Fair to Good.** Core vision and critical path projects (`plato-torch`, `oracle1-vessel`) are well-documented. Supporting libraries and infrastructure projects (`constraint-theory-core`, `cudaclaw`) suffer from unclear context and lack of beginner-friendly examples.

**P0 Issue (Negative Space):** A contributor cannot understand *where* `constraint-theory-core` or `cudaclaw` are used in the fleet from their READMEs. This creates a barrier to contribution.

**P1 Safe Channel (Actionable Improvement):**
1.  **For `cudaclaw`:** Add a "Quick Start" section with a minimal, runnable example that demonstrates hosting a simple agent function on the GPU.
2.  **For `constraint-theory-core`:** Add a "Used In" section listing 1-2 fleet projects that depend on it, with a brief description of the use case.
3.  **For `fleet-simulator`:** Embed the simplest example from its `examples/` folder directly into the README.

**P2 Optimization:** Standardize a README header template across all repos including fields for: **Fleet Role**, **Status**, **Related Projects**, and a **Minimal Example**.
