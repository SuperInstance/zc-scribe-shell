## Step 2: Documentation Quality Assessment

I will score each repository's primary documentation (README.md) on a 0-10 scale based on the following criteria derived from fleet doctrine and practical utility:
*   **P0 (0-3):** Does it prevent catastrophic misunderstanding? (Purpose, core concept, critical warnings)
*   **P1 (4-7):** Does it provide a safe channel to start? (Installation, basic usage, key commands)
*   **P2 (8-10):** Does it optimize the user's path? (Architecture, API details, contribution guide, examples)

### 1. plato-torch
**Score: 8/10**
*   **P0:** Excellent. Immediately states core thesis: "PLATO rooms are persistent training environments." Clearly defines key terms (Room, Tile, Ensign).
*   **P1:** Very good. Provides a clear `Quick Start` with installation (`pip install plato-torch`) and a minimal example to create and run a room. Lists the 25 preset rooms.
*   **P2:** Good. Includes an `Architecture` section explaining the Room-Tile-Ensign lifecycle. Could be improved with a more detailed API reference or a diagram of the data flow. The link to "Preset Rooms" is a strong point for exploration.
*   **Verdict:** Strong foundational doc that enables both use and understanding. A slight boost to P2 (diagrams, deeper example) would make it a 9.

### 2. holodeck-rust
**Score: 6/10**
*   **P0:** Good. Defines itself as a "sentiment-aware MUD server for generating training data via social simulation." The purpose is clear.
*   **P1:** Adequate. Provides build instructions (`cargo build`). The `Usage` section shows how to start the server and connect a telnet client. It's enough to get it running.
*   **P2:** Weak. Lacks critical context. What is a "sentiment graph"? How are NPC behaviors defined? There is no example of interaction, expected output, or how the generated data is structured. A new contributor would be lost.
*   **Verdict:** Gets the service running but leaves the user in a dark room. The "why" and "what next" are missing.

### 3. fleet-simulator
**Score: 4/10**
*   **P0:** Poor. The README is a single, vague sentence: "A fleet simulation environment for generating multi-agent interaction tiles." It does not explain what a "fleet simulation" entails or how it differs from `holodeck-rust`.
*   **P1:** Minimal. Only provides `cargo run`. No explanation of configuration, environment variables, or what a successful run looks like.
*   **P2:** Nonexistent. No architecture, no example output, no link between simulation and the generation of "tiles."
*   **Verdict:** This is a placeholder, not documentation. It fails P0 by not adequately defining its own scope and relation to other fleet components.

### 4. constraint-theory-core
**Score: 9/10**
*   **P0:** Excellent. Opens with a clear, mathematical problem statement: "A Rust library for geometric constraint solving... with a focus on snapping behaviors in CAD-like environments."
*   **P1:** Very good. Standard Rust lib instructions (`cargo add constraint-theory-core`). Includes a succinct code example demonstrating a basic point-line snap.
*   **P2:** Excellent. Contains a detailed `Theory` section explaining the deadband protocol (P0, P1, P2) as applied to geometry. The `API Overview` lists main modules (`solver`, `primitives`, `snap`). It directly connects library mechanics to fleet doctrine.
*   **Verdict:** A model README. It serves a user who wants to use it, a developer who wants to understand it, and a researcher who wants to connect it to broader concepts.

### 5. cudaclaw
**Score: 5/10**
*   **P0:** Fair. States it is a "GPU-resident agent runtime for high-frequency inference." The concept is advanced but the core value (why GPU-resident?) is implied, not explained.
*   **P1:** Barely sufficient. Has a `Build` section with complex, system-specific CUDA setup commands. No "hello world" example of an agent. The user is left wondering if the build succeeded.
*   **P2:** Very weak. Mentions "kernel fusion" and "zero-copy host-device messaging" without defining these terms in the context of the runtime. No architecture diagram, no performance benchmarks, no guide on migrating an agent to cudaclaw.
*   **Verdict:** High barrier to entry. The documentation assumes significant prior knowledge of GPU programming and agent architecture. It provides a narrow, technical P1 path that is likely to fail for many.

## Step 3: Summary Report & Recommendations

**Overall Fleet Documentation Health: Mixed.** Scores range from 9 (`constraint-theory-core`) to 4 (`fleet-simulator`). The average score is **6.4/10**. Core libraries have good docs; simulation/runtime projects are under-documented.

**Patterns Observed:**
1.  **Strong P0/P1, Weak P2:** Common pattern (`plato-torch`, `holodeck-rust`). Docs get you started but lack depth for extension or integration.
2.  **Missing Context:** Several repos (`fleet-simulator`, `cudaclaw`) fail to explain their role in the larger fleet ecosystem, violating the "inhabited intelligence" vision.
3.  **Jargon Assumption:** Advanced terms are used without layperson definitions, creating a high onboarding friction.

**Actionable Recommendations (P0, P1, P2):**

*   **P0 (Immediate Fix - Prevent Misunderstanding):**
    *   **`fleet-simulator`:** Rewrite README to define its simulation scope (e.g., "agent coordination scenarios") and differentiate it from `holodeck-rust` ("social NPC scenarios").
    *   **`cudaclaw`:** Add a "Why Cudaclaw?" section explaining the performance benefit of GPU-residency for specific fleet tasks.

*   **P1 (Short-term - Enable Contribution):**
    *   **`holodeck-rust`:** Add a `Generating Your First Tile` tutorial that shows a sample telnet session and the resulting JSON tile output.
    *   **`cudaclaw`:** Provide a minimal, verified example agent that runs on the runtime and prints a success message.

*   **P2 (Long-term - Optimize Understanding):**
    *   Create a fleet-wide documentation standard template in `fleet-knowledge/`, mandating sections: **Purpose (P0), Quick Start (P1), Architecture & Ecosystem (P2)**.
    *   **`plato-torch`:** Add a sequence diagram for the tile submission/validation flow.
    *   **All repos:** Add an `## In the Fleet` section linking to related projects (e.g., in `constraint-theory-core`: "Used by `flux-runtime` for spatial reasoning").
