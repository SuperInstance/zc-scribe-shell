## Step 1: Repository Selection & Analysis

I have selected 5 repositories from the fleet's known list. I will read their primary documentation files (README.md, any CONTRIBUTING.md, etc.) and score them based on the following criteria derived from fleet doctrine and common open-source standards:

*   **Clarity (0-2):** Is the purpose and scope immediately clear?
*   **Completeness (0-2):** Does it cover setup, usage, and contribution?
*   **Fleet Context (0-2):** Does it link to related fleet projects and explain its role?
*   **Actionability (0-2):** Can a new agent (greenhorn) use it to do work?
*   **Structure (0-2):** Is it well-organized with headers, code blocks, and links?

**Total Score:** Sum of the above (0-10).

## Step 2: Repository Documentation Audit & Scores

### 1. **Repository:** `plato-torch`
*   **Primary Doc:** `README.md`
*   **Analysis:**
    *   **Clarity (2/2):** Opens with a strong, clear thesis: "PLATO rooms are persistent training environments." The metaphor is consistent with fleet vision.
    *   **Completeness (2/2):** Covers core concepts (Rooms, Tiles, Ensigns), has a "Quick Start" with install commands, details the API (`DeadbandRoom`), and includes a "Contributing" section.
    *   **Fleet Context (2/2):** Explicitly names `plato-ensign` and `fleet-simulator` as related projects. Positions itself as the core training server.
    *   **Actionability (2/2):** Provides a complete, copy-pastable example to create a room, submit a tile, and train. A greenhorn can run it.
    *   **Structure (2/2):** Logical flow from vision to example to API. Uses code fences, lists, and clear section headers.
*   **Total Score:** **10/10**
*   **Notes:** Exemplary. Serves as a model for other fleet repos.

### 2. **Repository:** `holodeck-rust`
*   **Primary Doc:** `README.md`
*   **Analysis:**
    *   **Clarity (1/2):** Title "holodeck-rust" is clear, but the description ("A MUD server for sentiment-aware NPCs") assumes knowledge of "MUD" and "sentiment-aware." Could use a one-sentence plain English summary.
    *   **Completeness (1/2):** Has a "Getting Started" with `cargo run`, but lacks detail on configuring NPCs, defining sentiment rules, or interacting with the server. No contribution guidelines.
    *   **Fleet Context (2/2):** Clearly states it's part of Cocapn fleet and is a "companion simulation" to `plato-torch`. Good.
    *   **Actionability (1/2):** An agent can start the server, but the next steps (how to connect a client, send commands, see results) are missing. Action path is incomplete.
    *   **Structure (1/2):** Very brief. Could use sections for Architecture, API/Protocol, Example Session, and Configuration.
*   **Total Score:** **6/10**
*   **Notes:** Functional but sparse. The core idea is compelling, but documentation is a barrier to contribution.

### 3. **Repository:** `constraint-theory-core`
*   **Primary Doc:** `README.md`
*   **Analysis:**
    *   **Clarity (0/2):** The opening is technical and abstract ("geometric snapping library for CAD-like constraints"). It does not state *why* this exists for the fleet.
    *   **Completeness (0/2):** No installation instructions, no usage examples, no API overview. Only lists module names (`src/solver.rs`).
    *   **Fleet Context (0/2):** No mention of Cocapn, other fleet projects, or a use case within the intelligence-inhabitation thesis.
    *   **Actionability (0/2):** A greenhorn cannot use this repo based on the README.
    *   **Structure (1/2):** Has a "Modules" section, but that's it. It is structured, but the content is absent.
*   **Total Score:** **1/10**
*   **Notes:** This is a "code dump." The library may be powerful, but the documentation fails at P0 (don't create confusion). It actively blocks contribution.

### 4. **Repository:** `fleet-simulator`
*   **Primary Doc:** `README.md`
*   **Analysis:**
    *   **Clarity (2/2):** Clear title and description: "Generates training data for PLATO rooms by simulating fleet agent interactions."
    *   **Completeness (1/2):** Has a "Usage" section with a command, but the example output is truncated (`...`). Lacks explanation of simulation parameters, output format, or how to plug data into `plato-torch`.
    *   **Fleet Context (2/2):** Directly links to `plato-torch` as its consumer. Perfect.
    *   **Actionability (1/2):** An agent can run the command, but won't understand the output or how to modify the simulation. The path from run-to-result is unclear.
    *   **Structure (2/2):** Well-organized with Purpose, Usage, and Links sections.
*   **Total Score:** **8/10**
*   **Notes:** Strong foundation. Points deducted for incomplete examples and lack of configuration details.

### 5. **Repository:** `cudaclaw`
*   **Primary Doc:** `README.md`
*   **Analysis:**
    *   **Clarity (1/2):** "GPU-resident agent runtime" is clear technically, but the *problem it solves* (why keep an agent in GPU memory?) is not stated upfront.
    *   **Completeness (0/2):** No installation, no build instructions, no example. Only a list of source files and a "Roadmap" with future ideas.
    *   **Fleet Context (1/2):** Mentions it's for the "Cocapn fleet," but does not specify which agents or workloads use it.
    *   **Actionability (0/2):** No actionable instructions. A greenhorn cannot run or test anything.
    *   **Structure (1/2):** Has sections, but they are mostly placeholders or lists.
*   **Total Score:** **3/10**
*   **Notes:** Appears to be an early-stage or experimental project. Documentation reflects that. It states *what* but not *how* or *why for us*.

Scores range from 1 to 10, with an average of **5.6/10**. The flagship project (`plato-torch`) sets a high standard, but several key dependencies and components are under-documented, creating friction for new agents and hindering the "inhabited intelligence" flow.

**P0 Issue (Blocking):** `constraint-theory-core` (Score: 1). This is a critical path blocker. An agent cannot use or contribute to it. This violates the "find safe channels" doctrine because there is no channel.
