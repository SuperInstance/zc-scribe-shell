## Step 2: Documentation Quality Report

**Overall Fleet Documentation Health: Moderate (Average Score: 6.0/10)**

**Patterns Observed:**
1.  **Vision vs. Execution:** Repositories like `holodeck-rust` and `cudaclaw` excel at communicating the *why* but lack the *how*. This can create a barrier to new contributors.
2.  **Incomplete Stubs:** `fleet-simulator` represents a known risk—a repository created for an idea, with implementation begun, but documentation not yet followed. This can confuse agents trying to understand fleet capabilities.
3.  **Assumed Context:** Several repos (`constraint-theory-core`, `plato-torch`) reference fleet-specific concepts (Deadband Protocol, PLATO rooms, tiles) without linking to foundational documentation. This creates a high onboarding burden.
4.  **Strong Examples:** The highest-scoring repo (`plato-torch`) and `constraint-theory-core` succeed by providing a concrete, copy-pastable code example that demonstrates core functionality.

**Recommendations (P1 - Safe Channels):**
*   **P0 (Don't Hit Rocks):** Do not create new repos without a minimal viable README containing Purpose, Build Command, and a Running Example.
*   **P1 (Find Safe Channels):**
    *   Establish a fleet-wide README template in `fleet-knowledge/`.
    *   Mandate a "Basic Example" section in all tool/framework READMEs.
    *   Create a `GLOSSARY.md` that defines fleet terms (Tile, Room, Ensign, Deadband P0/P1/P2) and ensure READMEs link to it.
*   **P2 (Optimize):**
    *   Add "Next Steps" or "Integration" sections to READMEs showing how the project connects to others (e.g., "plato-torch outputs tiles consumed by...").
    *   Develop a simple documentation linter that checks for required sections.
