## Step 1: Repository Selection & Analysis

I selected 5 repositories from the fleet's GitHub organization using a random method (alphabetical, skipping the first 10). I read the primary README.md file for each and evaluated based on the following criteria derived from fleet standards:
1.  **Clarity of Purpose (P0):** Does it state *what* the project is and *why* it exists?
2.  **Getting Started (P1):** Does it provide clear, actionable steps to run or use the project?
3.  **Architecture & Context (P2):** Does it explain how it fits into the broader fleet ecosystem?
4.  **Completeness:** Are key sections (installation, usage, contribution) present?

### 1. **constraint-theory-core**
**Score: 8/10**
*   **P0 (Purpose):** Excellent. Opens with a clear, compelling thesis: "A geometric constraint solver for inhabited intelligence." Immediately defines its role.
*   **P1 (Getting Started):** Good. Provides `cargo run` command and a link to a detailed example. Could be stronger with a one-line "quick try" snippet.
*   **P2 (Context):** Strong. Explicitly links to `plato-torch` and `holodeck-rust`, explaining its place in the stack. Includes a "Related Projects" section.
*   **Notes:** Well-structured with a TOC. Includes a visual diagram (asset link). Lacks a dedicated "API" or "Reference" section, which is the main deduction.

### 2. **fleet-simulator**
**Score: 6/10**
*   **P0 (Purpose):** Clear. "A multi-agent simulation environment for generating training data." Good.
*   **P1 (Getting Started):** Weak. Instructions are: `npm install`, `npm start`. No explanation of what `start` does, what to expect, or required configuration (e.g., API keys for agents). This is a P1 hazard.
*   **P2 (Context):** Moderate. Mentions it feeds `plato-torch` but doesn't detail the data format or integration mechanism. Lacks a clear architecture diagram.
*   **Notes:** Very brief. Missing sections on agent definition, scenario configuration, and data output format. Functions more as a placeholder.

### 3. **holodeck-rust**
**Score: 9/10**
*   **P0 (Purpose):** Excellent. "A sentiment-aware MUD server for inhabited intelligence." Nails the "what" and "why."
*   **P1 (Getting Started):** Very Good. Step-by-step guide: clone, `cargo build`, configure `config.toml`, run server, connect via telnet. Includes example commands.
*   **P2 (Context):** Excellent. Has a dedicated "Architecture" section explaining the Event Loop, NPC System, and how it connects to `plato-torch` for tile ingestion.
*   **Notes:** Includes a "Contributing" section, code structure overview, and link to a design doc. A model README for the fleet.

### 4. **plato-ensign**
**Score: 5/10**
*   **P0 (Purpose):** Confusing. Title is "PLATO Ensign," but the description is technical and abstract: "Exports room experience into LoRA adapters." A greenhorn would not understand its role.
*   **P1 (Getting Started):** Poor. Instructions: `python export.py --room_id 5`. No information on dependencies (`pip install -r requirements.txt`?), where to get a `room_id`, or what the output is.
*   **P2 (Context):** Missing. Does not explain what a "room experience" is, what a "LoRA adapter" is in the fleet context, or who the consumer of this output is.
*   **Notes:** This README is written for an expert already deep in the fleet's concepts. It fails as an onboarding document.

### 5. **zeroclaw-shell**
**Score: 7/10**
*   **P0 (Purpose):** Good. "A git repository as an agent shell." Clear and intriguing.
*   **P1 (Getting Started):** Good. Provides the git clone command and explains the `cycle.sh` workflow. Could show example output.
*   **P2 (Context):** Fair. Mentions it submits tiles to a PLATO room, which is crucial. Could better diagram the data flow (agent -> shell -> room server).
*   **Notes:** Strong "Workflow" and "Concepts" section. Lacks a troubleshooting or "Common Errors" section, which is important for a bootstrapping tool.

## Step 2: Summary & Recommendations

**Overall Fleet Documentation Health: Fair (Average Score: 7.0/10)**
The quality is bimodal. Foundational tools (`holodeck-rust`, `constraint-theory-core`) are well-documented. Higher-level/data-flow tools (`fleet-simulator`, `plato-ensign`) are under-documented, creating friction in the integration pipeline.

**P0 (Negative Space - What to Avoid):**
*   **Avoid** READMEs that assume deep prior knowledge of the fleet's ontology (e.g., "room experience").
*   **Avoid** listing commands without explaining their expected outcome or prerequisites.

**P1 (Safe Channels - Where to Improve):**
1.  **Mandate a README template** for all repos. Minimum sections: Purpose, Quick Start, Architecture/Context, Links.
2.  **Create a "Fleet Glossary"** linked from all READMEs to define terms like *tile*, *room*, *ensign*, *deadband*.
3.  **Implement a documentation audit script** as part of CI that scores READMEs against the template.

**P2 (Optimization - Best Path):**
*   **Priority Fix:** `plato-ensign` and `fleet-simulator` READMEs are the highest leverage points for improvement, as they are critical integration nodes.
*   **Enhancement:** Add a "See Also" or "Next Steps" section to each README, directing users to the next tool in the workflow.
