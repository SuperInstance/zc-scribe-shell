## Step 2: Documentation Analysis & Scoring

**Scoring Rubric (0-10):**
*   **0-3:** Missing, incomprehensible, or severely misleading.
*   **4-6:** Exists but is incomplete, vague, or lacks critical context (e.g., how to run, purpose).
*   **7-8:** Clear, functional, and covers the basics (purpose, setup, core use).
*   **9-10:** Exemplary. Includes purpose, setup, architecture, examples, contribution guidelines, and connects to fleet doctrine.

### 1. plato-torch
**Primary Doc:** `README.md`
**Findings:** The README is comprehensive. It clearly states the project's purpose ("PLATO rooms for AI self-training"), provides a high-level architecture diagram, lists all 25 training room presets, and includes detailed setup instructions for both development and Docker. It explains key concepts like "tiles," "ensigns," and "mirror play" in the context of the fleet. It links to related projects (plato-ensign).
**Score:** **9/10**
**Rationale:** An excellent core document. A point is deducted because while it describes *what* the rooms are, a new agent might need a clearer, step-by-step example of the *interaction loop* between an agent and a room to produce a tile.

### 2. holodeck-rust
**Primary Doc:** `README.md`
**Findings:** The README is direct and functional. It starts with a clear, bold thesis statement ("A MUD server where NPCs have sentiment and memory"). It provides a `docker-compose` command for immediate setup, which is excellent for onboarding. It lists core features and dependencies. However, it lacks deeper context. How do NPC sentiments work? What is the data model? How does it integrate with or feed into the broader PLATO/fleet ecosystem? The setup works, but the "why" and architecture are light.
**Score:** **7/10**
**Rationale:** It successfully enables a user to get the service running and understand its high-level promise. It misses the opportunity to fully explain its role in the fleet's research (generating rich interaction tiles for training).

### 3. constraint-theory-core
**Primary Doc:** `README.md`
**Findings:** This README is minimal. It states the purpose ("Geometric constraint solving in Rust") and has a basic example of the `SnapPoint` API. It lacks any setup instructions (`Cargo.toml` snippet), a list of features, testing guidelines, or a link to more detailed documentation (if any exists). For a library, this is insufficient. A developer cannot easily integrate it without exploring the source code.
**Score:** **5/10**
**Rationale:** The core idea is present, but almost all practical implementation details are absent. It serves as a placeholder rather than functional documentation.

### 4. fleet-simulator
**Primary Doc:** `README.md`
**Findings:** The documentation is good. It opens with a clear purpose statement for the project and its role in the fleet ("Generate synthetic training data"). It provides a well-structured "Quick Start" section with prerequisites and run commands. It includes a helpful "Architecture" subsection that diagrams the major components (Director, Actors, Environment). It also has a "Development" section. The main gap is the lack of concrete examples of the *output* (the training data format) and how that data is consumed by other fleet components like `plato-torch`.
**Score:** **8/10**
**Rationale:** Strong on setup and internal architecture. Could be improved by showing the downstream use of its output, strengthening the connective tissue within the fleet.

### 5. cudaclaw
**Primary Doc:** `README.md`
**Findings:** This README is an archetype of experimental project documentation. It is highly technical and assumes significant prior knowledge. It begins with an abstract vision ("GPU-resident agent runtime") and dives straight into build instructions for a specific, complex setup (CUDA, specific LLM server). There is no "Quick Start" or simple example. While it contains important technical details and references to related concepts (VMCI, flux-runtime), it does not guide a new contributor. The "Why this exists" is buried in technical jargon.
**Score:** **4/10**
**Rationale:** The information is likely valuable for a specialist already deep in the fleet's GPU research. For almost anyone else, including a new fleet agent, it is opaque and inaccessible. It fails the basic task of onboarding.

## Step 3: Summary Report

**Overall Fleet Documentation Health: Mixed.**

*   **Strengths:** Core, active projects (`plato-torch`, `fleet-simulator`) have robust, useful documentation that aligns with fleet goals and enables contribution.
*   **Weaknesses:** Libraries (`constraint-theory-core`) and advanced experimental work (`cudaclaw`) are severely under-documented. This creates a "cliff" where agents can easily engage with high-level tools but hit a wall when needing to understand or modify foundational components.
*   **Common Gap:** A lack of explicit **inter-component linkage**. Documentation often silos a project without clearly mapping its inputs from and outputs to other fleet repos. This hinders the understanding of the fleet as a cohesive system.

**Recommendations (P1 - Safe Channel):**
1.  **Establish a minimal README template** for all repos (Purpose, Quick Start, API/Usage Example, Links to Related Fleet Projects).
2.  **Priority uplift targets:** `constraint-theory-core` (highest impact for a library) and `cudaclaw` (requires a "High-Level Concept" section before the technical deep dive).
3.  **Add a "Fleet Context" section** to key READMEs, explicitly stating: "This project provides [X] which is used by [Y] for [Z]."
