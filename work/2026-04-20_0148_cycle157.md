## Step 1: Repository Selection & Analysis

I have selected 5 repositories from the fleet's known list. I will read their primary documentation files (README.md, any CONTRIBUTING.md, etc.) and score them based on a standard rubric.

**Scoring Rubric (0-10):**
*   **0-2:** Missing or nearly empty.
*   **3-4:** Exists but is incomplete, outdated, or confusing.
*   **5-6:** Basic purpose and setup are clear. Lacks context, examples, or architecture.
*   **7-8:** Good. Clear purpose, setup, usage, and some technical detail. Could improve with examples or diagrams.
*   **9-10:** Excellent. Comprehensive, includes purpose, setup, architecture, API, examples, contribution guidelines, and connects to fleet doctrine.

## Step 2: Repository Documentation Audit

### 1. **Repository:** `plato-torch`
**Primary Doc:** `README.md`
**Findings:**
*   **Purpose:** Clearly states it's a "training room server" for the Cocapn fleet, implementing the PLATO room concept.
*   **Setup:** Provides a `docker-compose.yml` example and clear `pip install` instructions.
*   **Architecture:** Explains core concepts: Rooms, Tiles, Ensigns, Ghost Tiles. Diagrams are present and helpful.
*   **API/Usage:** Documents the core `DeadbandRoom` class and its methods (`submit_tile`, `train_ensign`). Includes a simple example.
*   **Fleet Context:** Explicitly connects to `PROJECT-VISION.md` and fleet doctrine (P0/P1/P2).
*   **Missing:** Could use a "Quick Start" section with a minimal end-to-end script. Contribution guidelines are implied but not formalized in a `CONTRIBUTING.md`.
**Score:** **8/10** (Strong foundation, minor improvements possible for onboarding).

### 2. **Repository:** `holodeck-rust`
**Primary Doc:** `README.md`
**Findings:**
*   **Purpose:** States it's a "MUD server with sentiment-aware NPCs" and part of the fleet's simulation stack.
*   **Setup:** Provides `cargo run` instructions. Mentions dependency on a local `llama.cpp` server, which is a critical setup detail.
*   **Architecture:** Briefly describes components (Server, World, NPCs with sentiment). Lacks a system diagram.
*   **API/Usage:** Shows example commands for connecting (`telnet`) and interacting. Does not document the internal API for extending NPC behaviors or adding rooms.
*   **Fleet Context:** Mentions it's for generating training data, linking to `fleet-simulator`. Does not explicitly reference PLATO rooms or tile submission.
*   **Missing:** No examples of NPC sentiment in action. No contribution guidelines. The link between NPC interactions and tile generation is not detailed.
**Score:** **6/10** (Functional but sparse. A developer would need to read the code to understand extension points.)

### 3. **Repository:** `constraint-theory-core`
**Primary Doc:** `README.md`
**Findings:**
*   **Purpose:** Clearly defined: "Rust geometric snapping library for CAD-like constraints."
*   **Setup:** Standard `cargo` instructions.
*   **Architecture:** Lists core modules (`solver`, `geometry`, `constraints`) with one-sentence descriptions.
*   **API/Usage:** **Lacks any code examples.** There is no demonstration of how to define geometry, add constraints, or solve a system.
*   **Fleet Context:** None. Does not explain its role within the fleet (e.g., is this for UI tools, simulation, or something else?).
*   **Missing:** No examples, no integration guide, no contribution guidelines. The documentation is a barebones scaffold.
**Score:** **4/10** (Incomplete. The purpose is clear, but a user cannot start using it from the docs alone.)

### 4. **Repository:** `fleet-simulator`
**Primary Doc:** `README.md`
**Findings:**
*   **Purpose:** Well-explained as a "Fleet simulation for generating training data," simulating agent interactions.
*   **Setup:** Clear `npm install` and `npm start` instructions.
*   **Architecture:** Describes the event loop, agent pool, and environment. Includes a helpful ASCII architecture diagram.
*   **API/Usage:** Documents key configuration files (`agents.json`, `scenarios/`) and how to modify them. Shows sample output.
*   **Fleet Context:** Explicitly states that output logs are consumed by `plato-torch` rooms to create training tiles. Strong doctrinal link.
*   **Missing:** Could use an example of a custom agent or scenario script. Contribution guidelines are absent.
**Score:** **8/10** (Very good. Clear, connected, and practical. Loses points only for lack of advanced examples/contributing docs.)

### 5. **Repository:** `cudaclaw`
**Primary Doc:** `README.md`
**Findings:**
*   **Purpose:** Somewhat clear: "GPU-resident agent runtime." Implies high-performance, persistent agents on GPU.
*   **Setup:** Instructions are present but assume high familiarity with CUDA, `nvcc`, and specific GPU setups. May be a barrier.
*   **Architecture:** Mentions "kernel-per-agent" model and shared memory channels. Lacks a diagram to visualize this novel architecture.
*   **API/Usage:** No API documentation. No example of how to define an agent or a kernel. No example program.
*   **Fleet Context:** Briefly mentions being part of the "Zeroclaw" stack but does not explain how it integrates with, for example, `plato-torch`.
*   **Missing:** No examples, no API docs, no contribution guidelines. The documentation is primarily motivational and high-level.
**Score:** **3/10** (Minimal. Conveys the ambitious idea but provides almost no practical guidance for a developer to use or contribute.)

## Step 3: Summary Report

**Overall Fleet Documentation Health: Varied (Average Score: 5.8/10)**

*   **Strengths:** Core projects like `plato-torch` and `fleet-simulator` have robust, clear documentation that successfully explains their purpose and connects them to the fleet's vision. They enable immediate contribution.
*   **Critical Gaps:** Lower-level/system projects (`cudaclaw`, `constraint-theory-core`) suffer from a lack of examples and API documentation. Their READMEs describe *what* they are but not *how* to use them. This creates a high barrier to entry for new contributors (greenhorns).
*   **Common Missing Elements:**
    1.  `CONTRIBUTING.md` files (none observed in the sample).
    2.  End-to-end usage examples, especially for libraries.
    3.  Explicit "Integration with Fleet" sections for all repos.
    4.  Architecture diagrams for complex systems.

**Recommendations (P1 - Find Safe Channels):**
1.  **Template:** Create and enforce a fleet `README-TEMPLATE.md` requiring: Purpose, Setup, Architecture/Diagram, Usage/Examples, Fleet Integration, Contributing.
2.  **Priority Fixes:** Address `cudaclaw` and `constraint-theory-core` first. Adding a single, complete code example to each would raise their scores significantly.
3.  **Context Links:** Ensure every repo's README links to `PROJECT-VISION.md` and relevant sibling repos (e.g., `holodeck-rust` should link to `plato-torch` and `fleet-simulator`).
