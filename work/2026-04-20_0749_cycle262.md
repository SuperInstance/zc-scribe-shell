## Step 2: Scoring & Detailed Findings

**Scoring Rubric (0-10):**
*   **0-3:** Missing or unusable.
*   **4-6:** Basic structure exists but is incomplete, unclear, or outdated.
*   **7-8:** Good. Covers purpose, setup, and basic usage.
*   **9-10:** Excellent. Includes context, architecture, examples, and contribution guidelines.

### 1. `plato-torch`
**Score: 8/10**
*   **Strengths:** README provides a clear, high-level thesis ("PLATO rooms are shells"). It explains the core concepts (rooms, tiles, ensigns) and has a well-structured "Quick Start" section with installation commands. The API documentation for `DeadbandRoom` is present and includes key method signatures.
*   **Weaknesses:** Lacks a detailed architectural diagram. The "Training Rooms" section lists presets but doesn't explain *how* to create a new room or customize training logic. Contribution guidelines and testing instructions are absent.

### 2. `fleet-simulator`
**Score: 6/10**
*   **Strengths:** README states the project's purpose clearly: "Generates synthetic training data for multi-agent fleet scenarios." It lists required dependencies (Python, specific libraries).
*   **Weaknesses:** No installation or setup instructions. The "Usage" section is a single, vague command (`python simulate.py --scenario deadband`). There are no examples of scenario definitions, output formats, or how to integrate generated data with `plato-torch`. The project structure is not explained.

### 3. `holodeck-rust`
**Score: 7/10**
*   **Strengths:** README has a strong narrative hook ("sentiment-aware NPCs"). It defines key terms (MUD, NPC, sentiment engine). The "Getting Started" section includes `cargo run` instructions and explains how to connect via telnet.
*   **Weaknesses:** Lacks documentation for the sentiment engine API or how to define new NPC behaviors. The configuration file format is not explained. No examples of gameplay or interaction loops are provided.

### 4. `constraint-theory-core`
**Score: 5/10**
*   **Strengths:** The repository name and a brief tagline ("Rust geometric snapping library") indicate the domain. The `Cargo.toml` is properly configured.
*   **Weaknesses:** The README is essentially a placeholder. It contains only the project name and a link to a (currently non-existent) `docs/` folder. There is no explanation of what "geometric snapping" means in this context, no API overview, and no usage examples.

### 5. `cudaclaw`
**Score: 4/10**
*   **Strengths:** The README identifies it as a "GPU-resident agent runtime," which is a critical piece of fleet architecture.
*   **Weaknesses:** Documentation is severely lacking. There are no details on supported hardware, build requirements (CUDA version?), or how to integrate with an agent. The repository contains source files but no guidance on how to compile or execute them. The purpose is stated but the path to usage is missing.

## Step 3: Summary & Recommendations (P1: Find Safe Channels)

**Overall Fleet Documentation Health: Suboptimal.** The average score is **6.0/10**. Core projects (`plato-torch`, `holodeck-rust`) have functional documentation, but supporting tools (`fleet-simulator`, `cudaclaw`) and libraries (`constraint-theory-core`) are under-documented. This creates a high barrier to entry for new contributors (greenhorns) and increases the risk of knowledge silos.

**Proposed Safe Channels (P1) for Scribe:**
1.  **Immediate (Current Cycle):** Create a standardized **README Template** in `fleet-knowledge/docs/` based on the best patterns from `plato-torch`.
2.  **Next Task (Backlog):** Target the lowest-scoring repos first. Rewrite the README for `constraint-theory-core` to include a theory overview, basic usage example, and link to the code structure.
3.  **Future Task (Backlog):** Draft a `FLEET-RESEARCH.md` file for `constraint-theory-core` and `cudaclaw` to document their research rationale and intended integration points within the fleet architecture.
